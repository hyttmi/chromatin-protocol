# chromatindb-cli Design Spec

**Date:** 2026-04-15
**Status:** Draft

## Overview

Standalone C++ CLI client that speaks the node's native PQ-encrypted wire protocol directly. SCP/SSH-like UX for encrypted file storage and retrieval. Links against `chromatindb_lib` — same PQ crypto, same Connection class, same codecs. No new dependencies.

All data is envelope-encrypted client-side. The node is a zero-knowledge store. Always.

## Commands

### Identity

```
chromatindb-cli keygen                     # Generate ML-DSA-87 + ML-KEM-1024 keypairs
chromatindb-cli whoami                     # Print namespace (SHA3-256 of signing pubkey)
chromatindb-cli export-key                 # Export KEM + signing pubkeys (shareable file)
```

### Data

```
chromatindb-cli put <file> [host[:port]]                  # Encrypt + upload, print hash
chromatindb-cli put --stdin [host[:port]]                  # Encrypt + upload from stdin
chromatindb-cli put <file> --share <pubkey>... [host[:port]]  # Encrypt for self + recipients
chromatindb-cli get <hash> [host[:port]]                   # Fetch + decrypt, restore filename
chromatindb-cli get <hash> --stdout [host[:port]]          # Fetch + decrypt to stdout
chromatindb-cli rm <hash> [host[:port]]                    # Tombstone
chromatindb-cli reshare <hash> --share <pubkey>... [host[:port]]  # Re-encrypt for new recipients, tombstone old
```

### Query

```
chromatindb-cli ls [host[:port]]                    # List own namespace
chromatindb-cli ls --namespace <ns_hex> [host[:port]]  # List another namespace
chromatindb-cli exists <hash> [host[:port]]         # Check blob existence
chromatindb-cli info [host[:port]]                  # Node info (version, peers, uptime, storage)
chromatindb-cli stats [host[:port]]                 # Own namespace stats (count, bytes, quota)
```

### Global Flags

```
--identity <path>    # Default: ~/.chromatindb/identity.key
--uds <path>         # Override UDS socket path
-p, --port <port>    # Override port (default: 4200)
-q, --quiet          # Suppress everything except output data/hash
```

## Connection Strategy

1. Try UDS at default path (`/run/chromatindb/node.sock`), overridable with `--uds <path>`
2. If no UDS socket found, TCP with full PQ handshake (ML-KEM-1024 + HKDF + ChaCha20-Poly1305)
3. After handshake, drain `SyncNamespaceAnnounce(62)` silently
4. Send request, receive response, disconnect

UDS uses TrustedHello (nonce + pubkey exchange, no KEM). TCP uses full PQ handshake. Both result in AEAD-encrypted sessions. The CLI is always a client (initiator), never a peer.

## Identity

`~/.chromatindb/` contains:

| File | Purpose |
|------|---------|
| `identity.key` | ML-DSA-87 signing secret key |
| `identity.pub` | ML-DSA-87 signing public key |
| `identity.kem` | ML-KEM-1024 decapsulation (private) key |
| `identity.kpub` | ML-KEM-1024 encapsulation (public) key |

Generated by `chromatindb-cli keygen`. The signing keypair authenticates the connection and signs blobs. The KEM keypair enables envelope encryption/decryption.

`export-key` outputs both public keys (signing + KEM) in a single file that recipients use for `--share`.

Namespace is `SHA3-256(signing_pubkey)` — same derivation as the node uses.

## Encryption Model

Every blob uploaded by the CLI is envelope-encrypted. No exceptions.

### Envelope Format

Uses the existing envelope format from PROTOCOL.md:

```
[magic:4][version:1][suite:1][recipient_count:2 BE]
[data_nonce:12]
[N x (kem_pk_hash:32 + kem_ct:1568 + wrapped_dek:48)]
[ciphertext + tag]
```

Suite `0x01` (ML-KEM-ChaCha) by default.

### Put Flow

1. Read file data
2. Build metadata JSON: `{"name":"filename.txt","size":12345}`
3. Construct payload: `[metadata_len:4BE][metadata_json][file_data]`
4. Generate random DEK (32 bytes)
5. For each recipient (always includes self):
   - ML-KEM-1024 encapsulate with recipient's KEM pubkey -> shared secret
   - HKDF-SHA256(shared_secret, "chromatindb-envelope-kek-v1") -> KEK
   - Wrap DEK with KEK (ChaCha20-Poly1305)
6. AEAD-encrypt payload with DEK
7. Build envelope blob data (magic + stanzas + ciphertext)
8. Sign: SHA3-256(namespace || envelope_data || ttl_be32 || timestamp_be64)
9. Build FlatBuffer Blob, send as Data(8) message
10. Receive WriteAck(30), print blob hash

### Get Flow

1. Send ReadRequest(31) with namespace + hash
2. Receive ReadResponse(32)
3. Check envelope magic bytes at start of blob data
4. Find our KEM stanza by matching `kem_pk_hash` against our KEM pubkey hash
5. ML-KEM-1024 decapsulate -> shared secret -> KEK -> unwrap DEK
6. AEAD-decrypt ciphertext with DEK
7. Parse payload: read metadata_len, extract metadata JSON, extract file data
8. Save to original filename (from metadata) or hash-based name if stdin upload

### Reshare Flow

1. `get` the blob (fetch + decrypt as above)
2. Re-encrypt for the new recipient list (always includes self)
3. `put` the new blob
4. `rm` (tombstone) the old blob
5. Print new hash

## Blob Payload Format

Inside the envelope ciphertext, the plaintext payload is:

```
[metadata_len:4 bytes big-endian uint32]
[metadata_json: metadata_len bytes, UTF-8]
[file_data: remaining bytes]
```

Metadata is a JSON object. Required field: `name` (original filename, empty string for stdin). Optional: `size` (original file size for verification).

## Wire Protocol Usage

| Command | Wire Messages |
|---------|---------------|
| `put` | Data(8) -> WriteAck(30) |
| `get` | ReadRequest(31) -> ReadResponse(32) |
| `rm` | Delete(17) -> DeleteAck(18) |
| `ls` | ListRequest(33) -> ListResponse(34), paginated |
| `exists` | ExistsRequest(37) -> ExistsResponse(38) |
| `info` | NodeInfoRequest(39) -> NodeInfoResponse(40) |
| `stats` | StatsRequest(35) -> StatsResponse(36) |

All through AEAD-encrypted frames. request_id is always 1 (single request per connection).

## Project Structure

```
cli/
  cli_main.cpp           # Arg parsing, command dispatch
  commands/
    keygen.cpp           # Identity generation (signing + KEM)
    put.cpp              # Envelope encrypt + upload
    get.cpp              # Fetch + envelope decrypt
    rm.cpp               # Tombstone
    reshare.cpp          # Fetch + decrypt + re-encrypt + upload + tombstone
    ls.cpp               # List blobs
    info.cpp             # Node info + namespace stats + exists
  client_connection.cpp  # UDS-first -> TCP fallback, handshake, drain announce
  client_connection.h
  envelope.cpp           # Envelope encrypt/decrypt (ML-KEM + ChaCha20-Poly1305)
  envelope.h
  metadata.cpp           # Payload metadata encode/decode
  metadata.h
CMakeLists.txt           # Builds chromatindb-cli, links chromatindb_lib
```

Top-level `CMakeLists.txt` gets:
```cmake
add_subdirectory(cli)
add_executable(chromatindb-cli cli/cli_main.cpp)
target_link_libraries(chromatindb-cli PRIVATE chromatindb_lib)
```

## What It Doesn't Do

- No subscriptions or push notifications (future client)
- No interactive/shell mode
- No chunked file manifests (single blobs up to 500 MiB)
- No key rotation or key ring management
- No group encryption
- No directory/key discovery service
