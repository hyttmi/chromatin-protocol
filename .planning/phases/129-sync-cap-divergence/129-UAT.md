# Phase 129 UAT: 2-node cap divergence (VERI-05)

**Goal:** Prove that a node with a smaller `blob_max_bytes` cap silently skips
oversized blobs from a larger-cap peer, the per-peer skip counter increments,
and sub-cap blobs still replicate normally.

## Topology

- **Node A** (smaller cap) — `blob_max_bytes = 1 MiB`. This is the `home` node
  for this test.
- **Node B** (larger cap) — `blob_max_bytes = 8 MiB`. This is the `local` node.
- Both nodes are in each other's `trusted_peers` mesh and have a full
  post-handshake NodeInfoRequest/Response exchange (Phase 129 Wave 1).

## Setup

1. On node A (home): ensure `config.json` has `"blob_max_bytes": 1048576`
   (1 MiB). Send `SIGHUP` to the node to apply the new cap live.
2. On node B (local): ensure `config.json` has `"blob_max_bytes": 8388608`
   (8 MiB). Send `SIGHUP` to apply.
3. Confirm the peering is live:

        cdb --node home peers
        cdb --node local peers

   Both nodes should list the other as a connected trusted peer with a non-zero
   handshake timestamp.

4. Record baseline counter values on A for delta comparison:

        curl -s http://<node-A-metrics-bind>/metrics | grep sync_skipped_oversized

   Expect either no sample lines for node B's address (first run) or an
   existing counter value to subtract from later.

## Test 1: Sub-cap blob replicates normally

Write a 256 KiB blob to node B:

    cdb --node local put my-ns some-key --size 256K

Wait approximately one sync round (10 s is safe with default config).

Verify on node A:

    cdb --node home get my-ns some-key

**Expected:** The blob is retrieved on A with a matching SHA3-256 hash. No
increment on `chromatindb_sync_skipped_oversized_total{peer="<B>"}`.

## Test 2: Over-A's-cap blob does NOT replicate to A

Write a 6 MiB blob to node B (well above A's 1 MiB cap, well below B's 8 MiB
cap):

    cdb --node local put my-ns big-blob --size 6M

**Expected on B:** accepted (B's cap is 8 MiB, 6 MiB passes).

Wait approximately one sync round.

On A, attempt retrieval:

    cdb --node home get my-ns big-blob

**Expected:** not-found. The blob is absent on A — B never announced it on any
of the three sync-out paths (BlobNotify fan-out, BlobFetch response, PULL
reconciliation announce) because A's `advertised_blob_cap = 1 MiB` filters it
out before the wire send.

## Test 3: Skip counter incremented on the skipping side

Scrape B's `/metrics` (B is the side that skips, because A advertised a cap
that is smaller than the blob B is trying to send):

    curl -s http://<node-B-metrics-bind>/metrics | grep sync_skipped_oversized

**Expected:**

    chromatindb_sync_skipped_oversized_total{peer="<A-address>"} N

where `N >= 1` (at least one skip recorded for Test 2's 6 MiB blob). If Test 2
was repeated across a cursor-miss window, `N` may be higher.

## Test 4: Direct BlobFetch from A for the 6 MiB blob

If the current `cdb` surface exposes a direct blob-fetch-by-hash form (check
`cdb --help` for a `fetch` / `pull` subcommand), force A to request B's 6 MiB
blob by its content hash. If no such subcommand exists, skip this test and
note it below — Test 2 + Test 3 already cover the PUSH and PULL announce
paths end-to-end.

**Expected:** B refuses to send (BlobFetch response site 2 filter fires) and
the counter on B increments with `{peer="<A-address>"}`.

## Test 5 (optional): Cap-unknown replication safety (D-01)

Shut down node A. Restart it with `advertised_blob_cap = 0` on B's side
before B receives a fresh `NodeInfoResponse` from A (short window — achievable
by killing A immediately before its first sync round).

**Expected:** during the "unknown" window, B does NOT filter any blob. The
D-01 conservative default means replication continues; B's skip counter for
A does not advance. This test is optional because the window is brief and
hard to hit reliably outside an instrumented run.
