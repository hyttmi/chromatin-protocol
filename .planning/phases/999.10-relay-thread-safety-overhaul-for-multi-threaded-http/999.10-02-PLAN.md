<interfaces>
After Plan 01, the strand type is:
```cpp
using Strand = asio::strand<asio::io_context::executor_type>;
// Created in relay_main.cpp:
auto strand = asio::make_strand(ioc);
```

The pattern for posting to strand from an HTTP handler coroutine:
```cpp
// Off-strand: heavy work (parse, validate, encode)
auto payload = build_payload(...);

// Enter strand for shared state access
co_await asio::post(strand_, asio::use_awaitable);

// On-strand: touch shared state (register_request, send, create_promise, etc.)
uint32_t relay_rid = router_.register_request(...);
auto promise = promises_.create_promise(relay_rid, strand_);
bool sent = uds_mux_.send(std::move(transport_msg));

// After co_await promise->wait(), we resume on the executor the timer uses,
// which is the strand (since the promise was created with strand_ executor).
// No extra post needed -- we're still on strand for the response handling.
```

Key: The post(strand_) call suspends the coroutine and resumes it on the strand.
After the strand-confined work is done, the coroutine naturally continues on whatever
executor it was scheduled on. Since we co_await promise->wait() (which uses a timer
on the strand executor), we remain on the strand through the response path too.

For sync handlers that become async:
- subscribe/unsubscribe currently return HttpResponse (sync)
- They need to co_await post(strand_) before SubscriptionTracker access
- Convert to async handlers (return asio::awaitable<HttpResponse>)
- Update route registration from add_route to add_async_route

For auth handlers:
- Currently sync lambdas in register_auth_routes
- Need strand access for ChallengeStore and TokenStore
- Convert to async with co_await post(strand_)

For check_auth middleware:
- Currently a sync function called from dispatch_async
- Needs strand for TokenStore::lookup
- Move auth check into strand-posting pattern in dispatch_async
