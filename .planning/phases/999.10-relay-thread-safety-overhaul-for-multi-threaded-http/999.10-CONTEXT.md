# Phase 999.10: Relay Thread-Safety Overhaul — ABANDONED

**Status:** Abandoned — strand confinement approach proved insufficient.

## What Happened

The relay was designed for single-threaded WS access. Phase 999.9 replaced WS with HTTP+SSE but kept the multi-threaded `io_context` with `hardware_concurrency()` threads. This exposed systemic concurrency bugs:

1. **Partial mutex fixes** — Added mutexes to 5 shared data structures (RequestRouter, WriteTracker, SubscriptionTracker, TokenStore, ResponsePromiseMap). Fixed individual races but created whack-a-mole.

2. **Strand confinement attempt** — Removed mutexes, created global strand, bound UDS coroutines to strand, made HTTP handlers `co_await post(strand)` before shared state access. But:
   - `co_await post(strand)` only puts you on the strand for the NEXT step
   - After `co_await promise->wait()`, resume executor depends on timer executor, not the posting strand
   - `send()` still accessed from multiple threads even with caller-side strand posting
   - Making `send()` self-strand via `asio::post` caused SEGV in drain coroutine

3. **Root cause:** The entire UdsMultiplexer, with its send queue, drain coroutine, read loop, and connection state, was designed as a single-owner object. Multi-threaded access requires either a full redesign or single-threaded execution.

## Decision

**New milestone: v4.0.0 — Relay Architecture v3** with proper concurrency model from the ground up. Options to evaluate:
- Single-threaded io_context (simplest, eliminates all races)
- Thread-per-connection with message passing to UDS strand
- Actor model with typed message channels

The HTTP+SSE transport layer (Phase 999.9) is correct. The concurrency model is what needs replacement.
