# Phase 999.10: Relay Thread-Safety Overhaul for Multi-Threaded HTTP - Context

**Gathered:** 2026-04-14
**Status:** Ready for planning

<domain>
## Phase Boundary

The relay was designed for single-threaded WS access. With HTTP transport and hardware_concurrency() threads on the io_context, all shared data structures have concurrent access from multiple threads — confirmed by ASAN heap-use-after-free crashes under benchmark load. Fix with a single global strand that serializes all shared state access. Remove the partial mutex fixes and replace with strand-based dispatch.

</domain>

<decisions>
## Implementation Decisions

### Concurrency Model
- **D-01:** Single global `asio::strand<asio::io_context::executor_type>` wrapping the io_context. ALL shared state access goes through this strand.
- **D-02:** HTTP handler coroutines run on any thread. When they need to access shared state (send UDS message, register request, lookup session), they `co_await asio::post(strand_, use_awaitable)` to enter the strand, do the work, then return to the general executor.
- **D-03:** UDS read_loop already runs as a coroutine — bind it to the strand so it naturally accesses shared state without posting.
- **D-04:** The strand serializes access to: UdsMultiplexer::send_queue_, RequestRouter, SubscriptionTracker, WriteTracker, TokenStore, ResponsePromiseMap, and SessionDispatch callbacks.

### Cleanup — Remove Partial Mutex Fixes
- **D-05:** Remove std::mutex from RequestRouter (request_router.h/cpp). Strand replaces it.
- **D-06:** Remove std::mutex from WriteTracker (write_tracker.h). Strand replaces it.
- **D-07:** Remove std::mutex from SubscriptionTracker (subscription_tracker.h/cpp). Strand replaces it.
- **D-08:** Remove std::mutex from TokenStore (token_store.h/cpp). Strand replaces it.
- **D-09:** Remove std::mutex from ResponsePromiseMap (response_promise.h). Strand replaces it. Keep shared_ptr ownership (that fixes a real lifetime issue, not a threading issue).

### Strand Wiring
- **D-10:** Create the strand in relay_main.cpp: `auto strand = asio::make_strand(ioc);`
- **D-11:** UdsMultiplexer's connect_loop and read_loop coroutines bind to the strand via `asio::bind_executor(strand, ...)` or `co_spawn(strand, ...)`.
- **D-12:** HTTP handler's `send_and_await()` (in handlers_data.cpp) and `forward_query()` (in handlers_query.cpp) must `co_await asio::post(strand, use_awaitable)` before touching RequestRouter::register_request, UdsMultiplexer::send, ResponsePromiseMap::create_promise.
- **D-13:** Subscribe/unsubscribe handlers must post to strand before touching SubscriptionTracker.
- **D-14:** Auth handlers and token operations must post to strand before touching TokenStore.
- **D-15:** The cleanup_loop (timeout purge) already runs in the UDS coroutine context — it's on the strand by D-11.

### Performance
- **D-16:** The strand only serializes short operations (map lookups, queue pushes, counter increments). Heavy work (TLS handshake, HTTP parsing, body reading, crypto verification) runs off-strand on the thread pool. This should not be a bottleneck.

### ASAN Verification
- **D-17:** The benchmark tool (tools/relay_benchmark.py) must run ASAN-clean at 100 concurrent clients. This is the acceptance test.

### Claude's Discretion
- Whether to pass strand reference down through constructors or use a global accessor
- Whether HttpConnection needs to know about the strand or just the handlers
- Whether to co_await post(strand) before AND after the critical section, or use a scoped pattern

</decisions>

<code_context>
## Existing Code Insights

### Current Threading Model
- relay_main.cpp creates `hardware_concurrency()` threads, each running `ioc.run()`
- HTTP accept loop spawns HttpConnection coroutines on `ioc` (any thread)
- UDS connect_loop/read_loop spawned on `ioc` (any thread)
- All shared state accessed from any thread — the root cause of all ASAN crashes

### Strand Pattern
```cpp
auto strand = asio::make_strand(ioc);
// UDS coroutines run on strand:
asio::co_spawn(strand, uds_mux.connect_loop(), asio::detached);
// HTTP handlers post to strand for shared state:
co_await asio::post(strand, asio::use_awaitable);
// ... access shared state safely ...
```

### What Stays Off-Strand
- TCP accept loop (HttpServer)
- TLS handshake
- HTTP header/body parsing
- FlatBuffer encoding/decoding
- Crypto verification (ML-DSA-87)
- Response writing to client socket

</code_context>

<specifics>
## Specific Ideas

- The strand is essentially a single-threaded executor — same model as the old WS relay, but opt-in per operation instead of global
- ResponsePromiseMap should keep shared_ptr (D-09) because that fixes a real lifetime issue independent of threading
- The benchmark at 100 clients is the litmus test — if it's ASAN-clean, we're done

</specifics>
