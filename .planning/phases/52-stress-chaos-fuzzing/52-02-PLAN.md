Create `tests/integration/test_stress02_peer_churn.sh` following established patterns.

**Topology:** 5-node cluster on dedicated network.
- Network: `chromatindb-stress02-test-net` subnet `172.48.0.0/16`
- Nodes: `chromatindb-stress02-node{1-5}` at `172.48.0.{2-6}`
- Named volumes: `chromatindb-stress02-node{1-5}-data` (persistent identity across restarts)
- Temp config files per node (mktemp pattern from DOS-06)
- Config: bind_address, bootstrap_peers (chain: node2->node1, node3->node1+node2, node4->node2+node3, node5->node3+node4), log_level info (not debug -- 30 minutes of debug logs would be enormous), sync_interval 5, inactivity_timeout 0

**Test flow:**

Phase 1 -- Setup and baseline (2 min):
1. Create network, volumes, config files. Start all 5 nodes via `docker run -d` (not compose, since we need per-node restart control with persistent volumes).
2. Wait all healthy.
3. Ingest 200 baseline blobs to node1 (rate 50, size 256, ttl 3600).
4. Wait for sync to node5 (120s timeout). This proves the cluster is working.

Phase 2 -- Continuous ingest during churn:
5. Start a background loadgen process that continuously writes to a surviving node. Use a bash background job:
   - Loop: pick a random surviving node IP, run loadgen with --count 20 --size 256 --rate 10 --ttl 3600, repeat.
   - Track total blobs ingested in a file (/tmp/stress02_ingest_count).
   - Use `docker run --rm` per batch, targeting the node with lowest IP that is currently running.
   - The loadgen loop runs for the duration of the churn phase.

Phase 3 -- Churn loop (30 minutes):
6. Main loop for 30 minutes (60 iterations, 30s each):
   a. Randomly pick kill_count (1 or 2) via `$((RANDOM % 2 + 1))`
   b. Collect list of currently running node containers
   c. Randomly select kill_count nodes from running list (use shuf)
   d. `docker kill` each selected node (sends SIGKILL)
   e. Log: "Killed node(s): X, Y"
   f. Sleep 15s (let surviving nodes detect disconnection)
   g. Restart killed nodes: `docker start` (container still exists, volume preserved)
   h. Wait healthy for restarted nodes (60s timeout each, non-fatal if slow)
   i. Log iteration number, time elapsed, which nodes killed/restarted

   Key: never kill ALL 5 nodes simultaneously. If kill_count=2 and only 2 are running, kill only 1. Always keep at least 1 node alive.

   The ingest background job will hit connection errors when targeting killed nodes -- this is expected. It should catch errors and retry on a different node.

Phase 4 -- Convergence (120s):
7. Stop the background ingest loop.
8. Send SIGHUP to all 5 nodes (clear reconnect backoff).
9. Wait 120 seconds for convergence.
10. Get blob counts from all 5 nodes via SIGUSR1 metrics.
11. Get XOR fingerprints from all nodes. Method: for each node, `docker exec chromatindb-stress02-nodeN chromatindb_verify hash-fields /data 2>&1 | tail -5`. Actually, fingerprints are available via SIGUSR1 metrics output. Check for the `fingerprints=` or namespace fingerprint fields in metrics output.

   Alternative convergence check (simpler): compare blob counts across all 5 nodes. They should be within a small delta (allow 5% tolerance since some blobs may have been in-flight during the final kills). All nodes should have at least the baseline 200 blobs.

Phase 5 -- Verification:
12. Check 1: All 5 nodes are running.
13. Check 2: All 5 nodes have blob counts >= 200 (baseline).
14. Check 3: The maximum blob count difference between any two nodes is <= 5% of the max. (If max is 500, min must be >= 475.)
15. Check 4: No SIGSEGV/SIGABRT/corruption in any node's logs.

**Cleanup:** trap function removes all 5 containers (docker rm -f), volumes, network, temp configs, temp ingest count file.

**Implementation note on background ingest:**
Use a simpler approach than a persistent background loop. Instead, during Phase 3, after each restart step (g), run a small loadgen batch (--count 10) to a surviving node. This avoids the complexity of managing a background Docker container that might target dead nodes. Track cumulative ingest count in a variable.

Actually, even simpler per the CONTEXT.md decision: "Continuous ingest to surviving nodes during churn." Use a detached Docker container running loadgen with high count and long duration, targeting node1. When node1 is killed, the loadgen will get connection errors and exit. After each restart cycle, launch a new short loadgen batch. This is more realistic chaos.

Simplest correct approach: in each churn iteration, AFTER restarting killed nodes, run a quick loadgen batch (--count 20 --rate 10 --size 256 --ttl 3600) targeting the node with the lowest number that is currently running. This guarantees continuous writes during churn without managing background containers.

Create `tests/integration/test_stress03_namespace_scaling.sh` following established patterns.

**Topology:** 3-node cluster on dedicated network.
- Network: `chromatindb-stress03-test-net` subnet `172.49.0.0/16`
- Nodes: `chromatindb-stress03-node{1-3}` at `172.49.0.{2-4}`
- Named volumes: `chromatindb-stress03-node{1-3}-data`
- Temp config: bind_address, bootstrap_peers (node2->node1, node3->node1+node2), log_level info, sync_interval 5, inactivity_timeout 0
- Use `docker run -d` pattern (not compose)

**Test flow:**

Phase 1 -- Setup:
1. Create network, volumes, configs. Start 3 nodes, wait healthy.

Phase 2 -- Generate 1000 identities and ingest:
2. Create 1000 identities in batches. Each identity = a unique namespace (loadgen generates fresh keypair by default).

   Use loadgen's `--identity-save` and `--identity-file` flags for persistence. Actually, since each loadgen invocation without --identity-file generates a fresh keypair, simply run 1000 loadgen invocations with --count 10 each. This naturally creates 1000 unique namespaces.

   Batch approach for efficiency: run loadgen in batches of 10 concurrent Docker containers (10 at a time), each writing 10 blobs. Total: 100 batches of 10 = 1000 namespaces.

   Implementation:
   ```
   TOTAL_NS=1000
   BATCH_SIZE=10
   BATCHES=$((TOTAL_NS / BATCH_SIZE))
   for batch in $(seq 1 $BATCHES); do
       # Launch BATCH_SIZE loadgen containers in parallel
       PIDS=()
       for i in $(seq 1 $BATCH_SIZE); do
           # Round-robin across the 3 nodes
           NODE_IP="172.49.0.$((2 + (i - 1) % 3))"
           docker run --rm --network chromatindb-stress03-test-net \
               --entrypoint chromatindb_loadgen \
               chromatindb:test \
               --target "${NODE_IP}:4200" --count 10 --size 128 --rate 100 --ttl 3600 &
           PIDS+=($!)
       done
       # Wait for all in this batch
       for pid in "${PIDS[@]}"; do
           wait "$pid" 2>/dev/null || true
       done
       # Log progress every 10 batches
       if (( batch % 10 == 0 )); then
           log "Ingested $((batch * BATCH_SIZE)) / $TOTAL_NS namespaces"
       fi
   done
   ```

   Note: `docker run` in background with `&` and `wait` -- the docker run commands themselves run in parallel, each creating a fresh identity.

Phase 3 -- Wait for sync convergence:
3. Wait 180s for all 10,000 blobs to sync across the 3-node cluster.
4. Get blob counts from all 3 nodes via SIGUSR1.
5. Expected: each node has 10,000 blobs (1000 ns x 10 blobs). Allow some tolerance: >= 9900 blobs (99%).

Phase 4 -- Verification:
6. Check 1: All 3 nodes have >= 9900 blobs (10,000 expected, allow 1% loss from transient errors during batch ingest).
7. Check 2: Blob counts match across nodes (max - min <= 100, i.e., <= 1% divergence).
8. Check 3: Cursor storage bounded -- restart node3, check startup integrity scan log for blob count. The restart triggers integrity_scan which reports total blobs. If integrity scan reports ~10,000 blobs, cursor state is sound. Also check node3 data directory size is reasonable (not exploding with cursor entries for 1000 namespaces).
9. Check 4: No crashes/corruption in logs.

**Cleanup:** trap cleanup removes containers, volumes, network, temp configs.

**Sizing rationale:** 1000 namespaces x 10 blobs = 10,000 blobs at 128 bytes each = ~1.28 MB of data. Cursor storage: 1000 namespaces x 3 peers x ~40 bytes per cursor = ~120 KB. Well within bounds.
