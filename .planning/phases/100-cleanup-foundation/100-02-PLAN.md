    **relay/core/session.h** (per D-08, D-09, D-10 — independent from db/ but same pattern):
    ```cpp
    #pragma once
    #include <asio.hpp>
    #include <deque>
    #include <string>
    #include <cstddef>

    namespace chromatindb::relay::core {

    class Session {
    public:
        explicit Session(asio::any_io_executor executor, size_t max_queue = 256);

        // Enqueue a message for sending. Returns false if queue full (overflow).
        // On overflow, session is closed immediately (D-10).
        asio::awaitable<bool> enqueue(std::string message);

        // Drain coroutine — co_spawn this alongside session lifecycle.
        // Pops messages from queue and "sends" them (in Phase 100, write is a no-op;
        // Phase 101 replaces with actual WebSocket write).
        asio::awaitable<void> drain_send_queue();

        // Close the session. Signals all pending messages as failed.
        void close();

        bool is_closed() const;

        // Access the delivered messages (for testing in Phase 100; removed in Phase 101).
        const std::deque<std::string>& delivered() const;

    private:
        struct PendingMessage {
            std::string data;
            asio::steady_timer* completion;  // Owned by enqueue caller's stack
            bool* result_ptr;                // Points to local in enqueue
        };

        // The actual write operation. Virtual-like dispatch point for Phase 101.
        // In Phase 100, appends to delivered_ deque for testing.
        asio::awaitable<bool> do_send(const std::string& data);

        std::deque<PendingMessage> send_queue_;
        std::deque<std::string> delivered_;  // For Phase 100 testing
        asio::steady_timer send_signal_;
        size_t max_queue_;
        bool closed_ = false;
        bool drain_running_ = false;
    };

    } // namespace chromatindb::relay::core
    ```

    **relay/core/session.cpp:**
    Implement the proven send queue pattern from db/net/connection.cpp (lines 834-898), adapted for the relay:

    1. `Session::Session(executor, max_queue)`: initialize send_signal_ with executor, set max_queue_.

    2. `Session::enqueue(std::string message)`:
       - If closed_, co_return false.
       - If send_queue_.size() >= max_queue_: log warning via spdlog, call close(), co_return false. (D-10: disconnect immediately on overflow)
       - Create local `bool result = false;` and `asio::steady_timer completion(send_signal_.get_executor());` on the coroutine stack.
       - Set completion.expires_after(std::chrono::hours(24)).
       - Push {std::move(message), &completion, &result} to send_queue_.
       - Cancel send_signal_ to wake drain coroutine.
       - co_await completion.async_wait(use_nothrow).
       - co_return result.

    3. `Session::drain_send_queue()`:
       - Set drain_running_ = true.
       - Outer loop: while (!closed_)
         - Inner loop: while (!send_queue_.empty() && !closed_)
           - Move front element out of deque, pop_front.
           - bool ok = co_await do_send(msg.data).
           - Set *msg.result_ptr = ok, cancel msg.completion.
           - If !ok: signal all remaining queue entries as failed, break.
         - If closed_: break.
         - Wait on send_signal_ with 24-hour expiry.
       - Signal all remaining queue entries as failed.
       - Set drain_running_ = false.

    4. `Session::do_send(const std::string& data)`:
       - In Phase 100: just push data to delivered_ deque, co_return true.
       - In Phase 101: this becomes the WebSocket write.

    5. `Session::close()`:
       - Set closed_ = true.
       - Cancel send_signal_ to wake drain coroutine.

    6. `Session::is_closed()`: return closed_.
    7. `Session::delivered()`: return delivered_.

    CRITICAL PATTERN NOTES (from Pitfall 5 and Pitfall 6 in research):
    - The `completion` timer and `result` bool are on the enqueue coroutine's stack — NOT in the deque. The deque stores raw pointers to these stack-local variables.
    - The drain loop moves the PendingMessage out of the deque BEFORE any co_await. After co_await, never access anything that was in the deque before suspension.
    - send_signal_ uses expires_after(hours(24)), NOT time_point::max() (overflow risk on some platforms).
    - Use `asio::as_tuple(asio::use_awaitable)` for the completion.async_wait to get [ec] without exceptions.
