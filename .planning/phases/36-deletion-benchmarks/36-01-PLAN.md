Extend the loadgen with identity persistence and blob hash collection for the write-then-delete pipeline.

   - `std::string identity_file_path` (empty = generate fresh)
   - `bool delete_mode = false`
   - `std::string hashes_from` (empty = not delete mode; "stdin" = read from stdin)

2. **parse_args()**: Add CLI flags:
   - `--identity-save PATH` -> `cfg.identity_save_path = argv[++i]`
   - `--identity-file PATH` -> `cfg.identity_file_path = argv[++i]`
   - `--delete` -> `cfg.delete_mode = true`
   - `--hashes-from stdin` -> `cfg.hashes_from = argv[++i]`
   Update `print_usage()` with new flags.

3. **main()**: Identity lifecycle change:
   - If `--identity-file` is set: call `NodeIdentity::load_from(identity_file_path)` (note: `load_from` expects a directory containing `node.key` + `node.pub` files, not a single file)
   - Else: call `NodeIdentity::generate()` as before
   - If `--identity-save` is set: call `identity.save_to(identity_save_path)` after generation. Create the directory with `std::filesystem::create_directories()` first.

4. **LoadGenerator**: Add `std::vector<std::string> blob_hashes_` member. In `subscribe_and_send()`, after computing `hash_hex` for each blob, push it: `blob_hashes_.push_back(hash_hex)`. Add `const std::vector<std::string>& blob_hashes() const { return blob_hashes_; }` accessor.

6. **stats_to_json()**: When `delete_mode` is true, set `j["scenario"] = "delete"` instead of "mixed"/"fixed". Add a `bool delete_mode` parameter.

<task type="auto">
  <name>Task 2: Add delete mode with tombstone construction and DeleteAck latency</name>
  <files>loadgen/loadgen_main.cpp</files>
  <action>
Implement the delete mode that reads blob hashes from stdin, constructs tombstones, sends Delete messages, and measures DeleteAck-based latency.

1. **Hex decoding utility**: Add `from_hex()` function that converts a 64-char hex string to `std::array<uint8_t, 32>`. Used to parse blob hashes from stdin.

2. **Read hashes from stdin**: In `main()`, if `cfg.delete_mode && cfg.hashes_from == "stdin"`, read lines from `std::cin` into a `std::vector<std::array<uint8_t, 32>> target_hashes`. Each line is a 64-char hex string (one blob hash per line). Skip empty lines. Override `cfg.count` to `target_hashes.size()` so the send loop matches.

3. **LoadGenerator changes for delete mode**:
   - Add `std::vector<std::array<uint8_t, 32>> target_hashes_` member, set from main before `prepare()`.
   - Add `bool delete_mode_` member, set from Config.
   - In `prepare()`: skip random data pool generation when in delete mode (tombstones are always 36 bytes). Still compute size_classes_ (all Small, since tombstone data is 36 bytes).

4. **Tombstone construction**: Add `make_tombstone_request()` method to LoadGenerator (or as free function), following the pattern from RESEARCH.md:
   ```cpp
   chromatindb::wire::BlobData make_tombstone_request(
       const chromatindb::identity::NodeIdentity& id,
       std::span<const uint8_t, 32> target_hash,
       uint32_t ttl, uint64_t timestamp)
   ```
   Uses `wire::make_tombstone_data(target_hash)` for the data field. Signs with `wire::build_signing_input()`.

5. **Send loop branching**: In `subscribe_and_send()`, branch based on `delete_mode_`:
   - **Delete path**: Instead of constructing a random blob, call `make_tombstone_request(identity_, target_hashes_[i], cfg_.ttl, now_us)`. Encode with `wire::encode_blob()`. Hash with `wire::blob_hash()`. Send with `TransportMsgType_Delete` (type 18) instead of `TransportMsgType_Data`.
   - Track `total_bytes_sent_ += TOMBSTONE_DATA_SIZE` (always 36 bytes for tombstone data).
   - Collect blob_hashes_ the same way as write mode (the tombstone blob's own hash, not the target hash).

6. **DeleteAck handling**: In `handle_message()`, add a case for `TransportMsgType_DeleteAck` (type 19):
   - Payload format: `[blob_hash:32][seq_num_be:8][status:1]` = 41 bytes
   - Extract blob_hash from first 32 bytes, convert to hex
   - Look up in `pending_sends_`, compute latency, push to `latencies_`, erase from pending
   - If all latencies received (`latencies_.size() == blobs_sent_`), cancel `drain_timer_`
   - This is the PRIMARY ACK for delete mode. Notifications also arrive but DeleteAck is used for latency measurement per the CONTEXT decision.

7. **Do NOT subscribe in delete mode**: In `subscribe_and_send()`, skip the Subscribe message send when in delete mode. DeleteAck comes directly from the connected node, not via pub/sub. (The subscription was for Notification-based ACK matching in write mode.)
