<objective>
Implement dedup-before-crypto (PERF-03), store path re-encode elimination (PERF-01), and sync receive copy reduction (PERF-05).

Purpose: These three optimizations are tightly coupled in the engine.ingest() pipeline. PERF-03 skips the expensive ML-DSA-87 verify (~67ms for 1 MiB) for duplicate blobs. PERF-01/05 eliminate the redundant encode_blob() + blob_hash() inside store_blob() by passing pre-computed values from the ingest pipeline. Together with Plan 01's hash-then-sign change, this completes the crypto throughput optimization.

Output: engine.ingest() computes encode+hash once, checks dedup before crypto, passes pre-computed values to storage. All tests pass.

   - Uses `precomputed_hash` in place of `hash` everywhere (make_blob_key, dedup check, seq_map, expiry_map, delegation_map)
   - Uses `precomputed_encoded` in the `encrypt_value()` call instead of `encoded`
   - Everything else (dedup check, encryption, seq_num, expiry, delegation index, tombstone index) stays identical

3. **Refactor the original store_blob**: After adding the new overload, refactor the original `store_blob(const wire::BlobData& blob)` to compute encoded + hash and then delegate to the new overload:
```cpp
StoreResult Storage::store_blob(const wire::BlobData& blob) {
    auto encoded = wire::encode_blob(blob);
    auto hash = wire::blob_hash(encoded);
    return store_blob(blob, hash, encoded);
}
```
This ensures both paths use the same implementation. The delete_blob() path in engine.cpp continues to use the original single-argument overload without change.

**Critical:** The new overload trusts that `precomputed_hash` and `precomputed_encoded` are correct for the given `blob`. This is safe because the only caller is engine.ingest(), which computes them from the same blob.

Rewrite the `BlobEngine::ingest()` method. The new pipeline order is:

**Step 0: Size check** (unchanged)
**Step 0b: Capacity check** (unchanged)
**Step 1: Structural checks** (unchanged)
**Step 2: Namespace ownership OR delegation check** (unchanged)

**Step 2.5 (NEW): Compute content hash + dedup check**
```cpp
// Compute encoded blob and content hash ONCE (used for dedup, tombstone check, and storage)
auto encoded = wire::encode_blob(blob);
auto content_hash = wire::blob_hash(encoded);

// Dedup check: skip expensive signature verification for already-stored blobs
if (storage_.has_blob(blob.namespace_id, content_hash)) {
    WriteAck ack;
    ack.blob_hash = content_hash;
    ack.seq_num = 0;  // Not looked up for dedup short-circuit (see RESEARCH.md Pitfall 5)
    ack.status = IngestStatus::duplicate;
    ack.replication_count = 1;
    return IngestResult::success(std::move(ack));
}
```

**Step 3: Signature verification** (unchanged logic, just moved after dedup)
```cpp
auto signing_input = wire::build_signing_input(
    blob.namespace_id, blob.data, blob.ttl, blob.timestamp);
if (!crypto::Signer::verify(signing_input, blob.signature, blob.pubkey)) {
    // ... existing rejection ...
}
```

**Step 3.5: Tombstone handling** -- reuse the already-computed `content_hash` and `encoded`:
- For tombstone blobs: unchanged (extract target, delete target)
- For regular blobs: use the existing `content_hash` for `has_tombstone_for()` check. Do NOT call `encode_blob()` or `blob_hash()` again.

**Step 4: Store** -- pass pre-computed values:
```cpp
auto store_result = storage_.store_blob(blob, content_hash, encoded);
```
Replace the existing `storage_.store_blob(blob)` call with the new overload.

**Important details:**
- The `has_blob()` check uses a read transaction internally. The subsequent `store_blob()` uses a write transaction with its own dedup check. This is safe: the read-then-write pattern may occasionally let a concurrent write through, but store_blob's internal dedup catches it. In practice, chromatindb is single-threaded so this never happens.
- `delete_blob()` method is NOT modified. It continues to use the original `store_blob(blob)` overload. Tombstones are tiny (36 bytes), so the redundant encode is negligible.
- The dedup short-circuit returns `seq_num = 0`. This is safe because: (a) PeerManager only logs the WriteAck for duplicates, (b) SyncProtocol notification callback only fires for `IngestStatus::stored`, not `duplicate`. See RESEARCH.md Pitfall 5.
