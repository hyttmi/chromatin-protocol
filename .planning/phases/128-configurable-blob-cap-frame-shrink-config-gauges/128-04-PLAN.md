**WARNING to executor:** If the project's Config mutation on SIGHUP is via REPLACING the Config object (not mutating it), then a live-reference approach breaks METRICS-02. Verify by reading reload_config() at db/peer/peer_manager.cpp:502+. The current pattern loads a NEW Config as `new_cfg`, copies selected fields into running objects via setters, then `new_cfg` goes out of scope. The persistent `config_` member on PeerManager is assigned from `parse_args`-derived config at startup — and is NOT re-assigned on SIGHUP per the existing reload logic. This means **the live `config_` reference on MetricsCollector would be STALE** after SIGHUP because PeerManager's own config_ is not mutated.

**Mitigation:** On SIGHUP, PeerManager must now also mutate its own `config_` (or MetricsCollector must hold a different, always-live reference). The simplest fix: inside `PeerManager::reload_config()`, after loading `new_cfg` and validating it, ASSIGN the relevant numeric fields onto the local `config_` member so the gauge read stays live. Since `config_` is declared `const config::Config& config_` (not a copy), this requires making it a mutable copy.

**Executor decision rule:** Read peer_manager.h:150 — currently:
```cpp
const config::Config& config_;
```

This is a reference to the Config owned by main.cpp's stack. reload_config loads new_cfg but does NOT mutate `config_` — so a live `std::to_string(config_.blob_max_bytes)` read WOULD stay at the startup value even after SIGHUP. THIS IS A BUG for METRICS-02.

**Fix:** Change `config_` from a const-ref to a mutable value (owned copy) in peer_manager.h:150. Update the constructor init-list in peer_manager.cpp (the `config_(config)` initialization stays the same since copy-construction works). In reload_config(), after loading and validating new_cfg, overwrite `config_ = new_cfg;` at the end of the function — this makes all subsequent `config_.X` reads (including from MetricsCollector) see the post-SIGHUP values.

**Verification that this refactor is safe:** `config_` in PeerManager is already read-only in practice (used for passing values into setters, never mutated inline). Converting from const-ref to owned copy is source-compatible. Sites that read `config_.X` remain unchanged.
