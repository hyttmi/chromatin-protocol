# Phase 999.7: Standardize All Endianness to Big-Endian - Research

**Researched:** 2026-04-12
**Domain:** Wire protocol byte-order standardization (C++ refactor)
**Confidence:** HIGH

## Summary

This phase eliminates the two remaining little-endian encoding sites in the codebase, standardizing all wire encoding to big-endian. The change touches three layers: (1) the `build_signing_input()` function in `db/wire/codec.cpp` where ttl and timestamp are LE-encoded into the SHA3-256 hash input, (2) the auth payload encode/decode in `db/net/auth_helpers.h` where pubkey_size is LE-encoded, and (3) the relay UDS multiplexer and tools which duplicate the auth payload LE encoding inline.

The scope is larger than CONTEXT.md estimated. CONTEXT.md decision D-06 states "no relay code changes needed" but the relay's `uds_multiplexer.cpp` contains inline LE auth payload encoding and decoding that communicates directly with the node's LE auth payload format. When the node switches to BE, the relay and tools MUST switch too or handshakes will fail. Additionally, `tools/relay_test_helpers.h` has a standalone `build_signing_input()` copy with LE encoding, and `tools/relay_uds_tap.cpp` has inline LE auth payload code.

**Primary recommendation:** Change all 6 LE encoding/decoding sites to BE, update PROTOCOL.md, update test expectations, and run full test suites for both db/ and relay/ to verify.

### Complete LE Site Inventory

Every LE encoding/decoding site in the codebase, with exact locations:

#### Site 1: build_signing_input() -- db/wire/codec.cpp:83-103
```cpp

    static_cast<uint8_t>(ttl),
```

**Fix:** Replace with `store_u32_be()` and `store_u64_be()` from `db/util/endian.h`. Rename variables from `ttl_le`/`ts_le` to `ttl_be`/`ts_be`.

#### Site 2: encode_auth_payload() -- db/net/auth_helpers.h:26-31

#### Site 3: decode_auth_payload() -- db/net/auth_helpers.h:47-50

```cpp
                   (static_cast<uint32_t>(data[2]) << 16) |
```

**Fix:** Reverse shift pattern to BE: `data[0] << 24 | data[1] << 16 | data[2] << 8 | data[3]`. Or use `read_u32_be()` from `db/util/endian.h` (requires adding `#include "db/util/endian.h"`).

### CONTEXT.md Scope Corrections

The CONTEXT.md underestimates scope in two ways:

1. **D-06 is incorrect:** "No relay code changes needed" -- FALSE. `relay/core/uds_multiplexer.cpp` has inline LE auth payload encode (line 310-313) and decode (line 342-346) that MUST match the node. Both node and relay participate in the same PQ handshake auth exchange; if one side writes BE and the other reads LE, handshakes break.

2. **D-10 is partially incorrect:** "Relay smoke test has no byte-order assumptions" -- TRUE for the relay's JSON translation layer, but FALSE for `tools/relay_test_helpers.h` which has a standalone `build_signing_input()` copy with LE encoding. The smoke test's `make_data_message()` calls this to produce valid signed blobs. After the node switches to BE, blobs signed with LE input will fail verification.

### BE Utility Functions Available

Both `db/util/endian.h` and `relay/util/endian.h` provide the needed BE functions:

| Function | Signature | Use For |
|----------|-----------|---------|
| `store_u32_be(uint8_t*, uint32_t)` | Fixed buffer write | codec.cpp ttl |
| `store_u64_be(uint8_t*, uint64_t)` | Fixed buffer write | codec.cpp timestamp |
| `write_u32_be(vector<uint8_t>&, uint32_t)` | Vector append | auth_helpers.h encode |
| `read_u32_be(span<const uint8_t>)` | Bounds-checked read | auth_helpers.h decode |
| `read_u32_be(const uint8_t*)` | Unchecked read | uds_multiplexer.cpp |

### Anti-Patterns to Avoid
- **Partial update:** Changing db/ but forgetting relay/ or tools/ -- handshakes will fail silently in integration tests
- **Comment-only fix:** Updating comments but leaving byte-order logic unchanged
- **Mixing endian.h includes:** db/ code must use `db/util/endian.h`, relay/ code must use `relay/util/endian.h` (they are separate copies in separate build targets)

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| BE encoding | Manual bit shifts | `store_u32_be()` / `store_u64_be()` from endian.h | Consistent, tested, matches rest of codebase |
| BE decoding | Manual bit shifts | `read_u32_be()` from endian.h | Bounds-checked version available for span |
| Vector append BE | Manual push_back | `write_u32_be()` from endian.h | Matches pattern used elsewhere |

**Key insight:** Both `db/util/endian.h` and `relay/util/endian.h` already have every BE function needed. The codec.cpp site can use `store_u32_be`/`store_u64_be` (fixed buffer writes into stack arrays). The auth_helpers.h encode can use `write_u32_be` (vector append). The auth_helpers.h decode can use `read_u32_be` (span read). Only `tools/relay_uds_tap.cpp` might need manual shifts if it doesn't include an endian.h -- but it should include relay's.

## Common Pitfalls

### Pitfall 1: Forgetting the relay UDS auth path
**What goes wrong:** Node switches to BE auth payload, relay still sends LE -- every UDS handshake fails
**Why it happens:** CONTEXT.md D-06 says "no relay code changes needed" but relay has inline auth payload code
**How to avoid:** Change all 8 sites identified above, not just the 2 from CONTEXT.md
**Warning signs:** Relay fails to connect to node after rebuild

### Pitfall 2: Test hardcoded byte expectations
**What goes wrong:** Tests that check specific byte values (e.g., `REQUIRE(encoded[0] == 0x0A)`) will fail
**Why it happens:** The tests correctly verify byte layout -- they need updating to expect BE layout
**How to avoid:** Update every test that checks specific byte positions in auth payload or signing input
**Warning signs:** Catch2 assertion failures showing byte-order mismatches

### Pitfall 3: relay_test_helpers.h has its own build_signing_input copy
**What goes wrong:** Smoke test produces blobs with LE signing input, node rejects them
**Why it happens:** `tools/relay_test_helpers.h` has a standalone copy of build_signing_input that doesn't use db/wire/codec.cpp
**How to avoid:** Update the copy in relay_test_helpers.h to match the new BE encoding
**Warning signs:** Smoke test Data message failures (signature verification fails on node side)

### Pitfall 4: Not checking for hidden LE in comments that look like code
**What goes wrong:** Comments referencing "LE" remain, confusing future maintainers
**Why it happens:** Many LE sites are documented with explicit "IMPORTANT: LE" comments
**How to avoid:** grep -rn for LE/little-endian across entire codebase after changes
**Warning signs:** Any remaining "LE" or "little-endian" references in source files
