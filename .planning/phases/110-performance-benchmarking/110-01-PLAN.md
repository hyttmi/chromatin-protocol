FlatBuffer Blob schema (db/schemas/blob.fbs):
```
table Blob {
  namespace_id:[ubyte];   // 32 bytes: SHA3-256(pubkey)
  pubkey:[ubyte];         // 2592 bytes: ML-DSA-87 public key
  data:[ubyte];           // variable length: application data
  ttl:uint32;             // seconds until expiry, 0 = permanent
  timestamp:uint64;       // writer's unix timestamp
  signature:[ubyte];      // up to 4627 bytes: ML-DSA-87 signature
}
```

Signing input: SHA3-256(namespace_id || data || ttl_be32 || timestamp_be64)
- ttl is big-endian uint32, timestamp is big-endian uint64
- ML-DSA-87 signs the 32-byte SHA3-256 digest
- namespace_id = SHA3-256(pubkey)

HTTP Auth flow:
1. POST /auth/challenge -> {"nonce": "<hex>"}
2. POST /auth/verify with {"nonce": "<hex>", "public_key": "<5184 hex chars>", "signature": "<hex>"} -> {"token": "<hex>", "namespace": "<hex>"}
3. All subsequent requests: Authorization: Bearer <token>

Data endpoints:
- POST /blob (Content-Type: application/octet-stream, body = raw FlatBuffer Blob) -> {"type":"write_ack","blob_hash":"<hex>","seq_num":"<uint64_str>","status":"<name>"}
- GET /blob/{ns_hex}/{hash_hex} -> raw binary (application/octet-stream), 404 if not found
- DELETE /blob/{ns_hex}/{hash_hex} (body = raw FlatBuffer tombstone) -> JSON DeleteAck

Query endpoints (all return JSON):
- GET /stats/{ns_hex} -> namespace stats
- GET /exists/{ns_hex}/{hash_hex} -> {"type":"exists_response","exists":bool,"blob_hash":"<hex>"}
- GET /list/{ns_hex}?since=<ts>&limit=<n> -> blob list
- GET /node-info -> node info
- GET /storage-status -> storage status
</interfaces>

DEPENDENCIES: httpx, flatbuffers, liboqs-python (oqs). Script should check imports on startup and print clear error if missing ("pip install httpx" etc). Per D-01.

- --concurrency-levels (default "1,10,100") -- per D-05

CORE INFRASTRUCTURE:

1. **ML-DSA-87 Identity** (per D-02, D-03):
   - Generate ephemeral ML-DSA-87 keypair at startup using oqs.Signature("Dilithium5")
   - Derive namespace = SHA3-256(pubkey) using hashlib.sha3_256
   - sign(message_bytes) method returns signature

2. **Auth helper** (per D-02):
   - async authenticate(client: httpx.AsyncClient, base_url, identity) -> str (bearer token)
   - POST {base_url}/auth/challenge -> get nonce hex
   - Sign the nonce bytes (from_hex(nonce_hex)) with ML-DSA-87
   - POST {base_url}/auth/verify with JSON {"nonce": nonce_hex, "public_key": pubkey_hex, "signature": sig_hex}
   - Return token string
   - pubkey is 2592 bytes = 5184 hex chars

3. **FlatBuffer Blob builder** (per D-12):
   - build_blob(identity, data: bytes, ttl: int = 3600) -> bytes
   - Use flatbuffers.Builder to construct a Blob table
   - namespace_id = identity.namespace (32 bytes)
   - pubkey = identity.public_key (2592 bytes)
   - data = the payload bytes
   - ttl = given ttl
   - timestamp = int(time.time())
   - Build signing input: SHA3-256(namespace_id || data || ttl_be32 || timestamp_be64) where ttl is struct.pack(">I", ttl) and timestamp is struct.pack(">Q", timestamp)
   - signature = identity.sign(signing_digest)
   - IMPORTANT: FlatBuffer field order for building -- create all vectors first (CreateByteVector for each [ubyte] field), then start the table with BlobStart, add fields, end with BlobEnd, finish with builder.Finish(). Use the flatbuffers.Builder API directly since we don't have generated Python code. The Blob table vtable layout matches blob.fbs field order: namespace_id=field0, pubkey=field1, data=field2, ttl=field3, timestamp=field4, signature=field5. Use builder.PrependUOffsetTRelativeSlot for vector fields, builder.PrependUint32Slot for ttl, builder.PrependUint64Slot for timestamp.

4. **Latency measurement** (per D-10):
   - Use time.perf_counter_ns() for precision
   - Collect all latencies in a list, compute p50/p95/p99 using sorted + index
   - Report in milliseconds

BENCHMARK WORKLOADS:

**PERF-01: Throughput benchmark** (per D-06):
- For each concurrency level (1, 10, 100):
  - Create N httpx.AsyncClient instances (per D-14: HTTP keep-alive is default)
  - Each client authenticates (gets its own bearer token)
  - Each client sends --iterations POST /blob requests with 1 KB random data
  - Measure wall-clock time for all clients to complete
  - Report: blobs/sec (total ops / wall time), per-client ops/sec, p50/p95/p99 latency per request

**PERF-02: Latency overhead** (per D-07):
- NOTE: UDS direct requires constructing TransportMessage FlatBuffers + TrustedHello + HKDF + ChaCha20-Poly1305 AEAD. This is extremely complex in Python.
- If --skip-uds: measure relay-only latency for GET /blob and report "UDS baseline deferred to future phase"
- If not --skip-uds: attempt UDS direct measurement (best-effort). Use Unix domain socket, TrustedHello handshake (32-byte pubkey in AEAD context), HKDF-SHA256 key derivation, ChaCha20-Poly1305 encrypted transport frames. This is hard -- if it doesn't work, fall back to relay-only.
- Default: --skip-uds is ON (skip UDS baseline). The relay-only latencies from PERF-01 already give useful data.
- Operations to measure: single blob write (POST /blob), single blob read (GET /blob), exists check (GET /exists), stats query (GET /stats)

**PERF-03: Large blob benchmark** (per D-08):
- For each size (1, 10, 50, 100 MiB):
  - Build a FlatBuffer Blob with os.urandom(size) as data
  - Measure write time: POST /blob with raw binary body
  - Extract blob_hash from WriteAck JSON response
  - Measure read time: GET /blob/{ns}/{hash} and verify response length
  - Report: MiB/sec write, MiB/sec read, total time
  - Use a single client with HTTP keep-alive (per D-14)
  - NOTE: ML-DSA-87 signing of SHA3-256(namespace||100MiB data||...) will be slow. Include signing time separately from network time if feasible.

**PERF-04: Mixed workload** (per D-09):
- Concurrent workload:
  - "Small" pool: 10 clients doing GET /stats/{ns} and GET /exists/{ns}/{hash} in a loop
  - "Large" pool: 2 clients doing POST /blob with 10 MiB payloads
  - Run for 30 seconds (or --iterations total ops on small side)
  - Measure small-query p50/p95/p99 latency WITH and WITHOUT large blob load
  - Report: latency degradation percentage for small queries under large-blob load
