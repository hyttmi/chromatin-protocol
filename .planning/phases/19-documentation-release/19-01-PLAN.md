Create `db/PROTOCOL.md` as a protocol walkthrough document for developers building chromatindb-compatible clients. Target audience: "developer building a client in another language." Use a hybrid format: step-by-step narrative with ASCII diagrams for the handshake.

Structure:

1. **Header** -- "chromatindb Protocol Walkthrough" with a one-paragraph intro stating this covers the wire protocol for connecting to and interacting with a chromatindb node.

2. **Transport Layer** -- Document the AEAD frame format from `db/net/framing.h`:
   - Frame format: `[4 bytes: big-endian uint32 ciphertext_length][ciphertext_length bytes: AEAD ciphertext]`
   - AEAD parameters: ChaCha20-Poly1305, 32-byte key (from HKDF), 12-byte nonce (4 zero bytes + 8-byte BE counter), empty AD, 16-byte tag appended
   - Separate send/recv counters starting at 0, incrementing per frame
   - Max frame size: 110 MiB (115,343,360 bytes)
   - Plaintext is a FlatBuffers TransportMessage: `{ type: TransportMsgType (byte), payload: [ubyte] }`

3. **Connection Lifecycle** -- Three subsections:
   - **Step 1: TCP Connect** -- Connect to node's bind_address (default 4200)
   - **Step 2: PQ Handshake** -- ASCII diagram of the 4-step handshake from `db/net/handshake.h`:
     ```
     Initiator                         Responder
         |--- [raw] KemPubkey ----------->|  ML-KEM-1024 ephemeral pubkey (1568 B)
         |<-- [raw] KemCiphertext --------|  ML-KEM-1024 ciphertext (1568 B)
         |   Both derive session keys via HKDF
         |--- [encrypted] AuthSignature ->|  ML-DSA-87 pubkey (2592 B) + signature
         |<-- [encrypted] AuthSignature --|  ML-DSA-87 pubkey (2592 B) + signature
     ```
     Explain: KEM messages are raw (unencrypted), auth messages use the derived AEAD keys. Session keys derived via HKDF with labels "chromatin-init-to-resp-v1" and "chromatin-resp-to-init-v1". Session fingerprint for mutual authentication.
   - **Step 3: Encrypted Session** -- All subsequent messages are AEAD-encrypted TransportMessages. ACL check happens after handshake (if allowed_keys configured).

4. **Storing a Blob** -- Two subsections:
   - **Canonical Signing Input** -- Explain the signing format from `db/wire/codec.h`: `SHA3-256(namespace_id || data || ttl_le32 || timestamp_le64)`. The signature covers these fields in this exact order, NOT the raw FlatBuffer bytes.
   - **Data Message** -- Construct a FlatBuffers Blob (reference `schemas/blob.fbs` fields: namespace_id, pubkey, data, ttl, timestamp, signature), wrap in TransportMessage with type=Data (8), send as AEAD frame. Node responds by ingesting and replicating.

5. **Retrieving Blobs (Sync Protocol)** -- Document the Phase A/B/C sync flow from `db/sync/sync_protocol.h`:
   - **Phase A: Namespace Exchange** -- SyncRequest/SyncAccept, then NamespaceList wire format
   - **Phase B: Hash Diff** -- Per-namespace HashList exchange, compute diff
   - **Phase C: Blob Transfer** -- BlobRequest (max 64 hashes per request), BlobTransfer response, one blob at a time
   - SyncComplete, then inline PEX follows
   Include wire formats for each phase using the binary layouts from sync_protocol.h.

6. **Additional Interactions** -- Concise coverage (overview depth, not full walkthrough) of:
   - **Blob Deletion** -- Delete message (type=18) with tombstone format (0xDEADBEEF magic + 32-byte target hash), DeleteAck response
   - **Namespace Delegation** -- Delegation blob format (DELEGATION_MAGIC + delegate pubkey), written via standard Data message
   - **Pub/Sub Notifications** -- Subscribe/Unsubscribe (types 20/21) with namespace list payload, Notification (type=22) with 77-byte fixed payload
   - **Peer Exchange** -- PeerListRequest/PeerListResponse (types 16/17)
   - **Storage Signaling** -- StorageFull (type=23) sent when node at capacity

7. **Message Type Reference** -- Table listing all 24 TransportMsgType values from `schemas/transport.fbs` with their numeric values and one-line descriptions.

Reference the FlatBuffers schemas inline where appropriate (blob.fbs is 12 lines -- include it directly). Reference transport.fbs for the message type enum. Use byte-level descriptions, NOT C++ types.
