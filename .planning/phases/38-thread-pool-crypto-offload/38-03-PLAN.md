<objective>
Offload all Signer::verify() calls in Connection handshake code to the thread pool, so the event loop is never blocked during handshake authentication.

Purpose: Handshake auth verification (ML-DSA-87 verify) runs on a pool worker thread, freeing the event loop to accept and process other connections during the ~5ms verify time.
Output: All 4 handshake verify call sites offloaded. AEAD state untouched. All connection/handshake tests pass.
</objective>

  <action>
Add `#include "db/crypto/thread_pool.h"` to connection.cpp.

For each of the 4 Signer::verify call sites in the handshake methods, replace the synchronous call with a thread pool offload. The pattern is identical for all 4:

**Before:**
```cpp
bool valid = crypto::Signer::verify(
    session_keys_.session_fingerprint, resp_sig, resp_pk);
```

**After:**
```cpp
bool valid = co_await crypto::offload(*pool_, co_await asio::this_coro::executor,
    [&]() {
        return crypto::Signer::verify(
            session_keys_.session_fingerprint, resp_sig, resp_pk);
    });
```

Apply this transformation at exactly these 4 locations:

1. **do_handshake_initiator_trusted()** — verifies responder's auth after PQRequired fallback (the TrustedHello->PQ fallback path). Around line 297. The lambda captures session_keys_.session_fingerprint (read-only, already derived), resp_sig, resp_pk (local vectors on coroutine frame).

2. **do_handshake_initiator_pq()** — verifies responder's auth. Around line 387. Same capture pattern.

3. **do_handshake_responder_pq_fallback()** — verifies initiator's auth. Around line 511. Captures session_fingerprint, init_sig, init_pk.

4. **do_handshake_responder_pq()** — verifies initiator's auth. Around line 592. Same pattern.

**SAFETY ANALYSIS for each capture:**

- `session_keys_.session_fingerprint`: This is `std::array<uint8_t, 32>` — a fixed-size value inside session_keys_. By the time we reach the verify call, session_keys_ has been fully derived (via `hs.take_session_keys()` or `derive_lightweight_session_keys()`). Reading session_fingerprint from the pool worker is safe because: (a) it's immutable after derivation, (b) no other coroutine is writing to it, (c) the Connection coroutine is suspended at the co_await, not executing.

- `resp_sig`/`resp_pk`/`init_sig`/`init_pk`: Local vectors on the coroutine frame. The coroutine is suspended, so no one else can access them. Safe to read from pool.

- `send_counter_`, `recv_counter_`: NOT captured, NOT accessed. These are only used in send_encrypted/recv_encrypted which are NOT called from the offload lambda.

**IMPORTANT:** The do_handshake_responder_trusted() method does NOT call Signer::verify — it uses a lightweight key exchange with no signature verification. Leave it unchanged.

**Edge case — pool_ is nullptr:** This should never happen if Plan 01 plumbing is correct (set_pool called before run()). But add a defensive check in each offload site: if pool_ is null, fall back to synchronous verify. This avoids crashes if Connection is used in a context without a pool (e.g., old test code):

```cpp
bool valid;
if (pool_) {
    valid = co_await crypto::offload(*pool_, co_await asio::this_coro::executor,
        [&]() { return crypto::Signer::verify(...); });
} else {
    valid = crypto::Signer::verify(...);
}
```

However, if Plan 01 properly sets pool on all connections and all tests provide a pool, the nullptr path is truly dead code. Include it for defensive correctness but do not optimize around it.
