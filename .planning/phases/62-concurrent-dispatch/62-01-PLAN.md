<objective>
Fix the latent thread safety issue in Data and Delete handlers where `send_message`, `notify_subscribers`, `metrics_`, and `record_strike` are called from the thread pool after `engine_.ingest()` / `engine_.delete_blob()` resume the coroutine on a pool thread. Document the inline dispatch pattern for cheap operations.

Purpose: Enables safe concurrent request pipelining -- clients can send multiple requests without waiting for responses, and all AEAD nonce operations, shared state access, and metrics updates happen on the IO thread.
Output: Thread-safe Data/Delete handlers, documented inline dispatch pattern, verified by tests.
</objective>
