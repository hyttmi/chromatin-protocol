**Four changes to peer_manager.cpp. Read the full file first to get exact line numbers.**

**Change 1: Rewrite on_peer_message SyncRequest handler (lines 370-380)**

Replace the existing SyncRequest handler block with cooldown check + session limit check + proper rejection. The new block must:

1. Find peer (early return if null)
2. Step 0: If `sync_cooldown_seconds_ > 0`, compute elapsed ms since `peer->last_sync_initiated` using steady_clock. If elapsed < cooldown threshold, call `send_sync_rejected(conn, SYNC_REJECT_COOLDOWN)`, increment `metrics_.sync_rejections`, return.
3. Step 1: If `peer->syncing` (session already active), call `send_sync_rejected(conn, SYNC_REJECT_SESSION_LIMIT)`, increment `metrics_.sync_rejections`, return.
4. ONLY after both checks pass: update `peer->last_sync_initiated = steady_clock_ms_now()`, clear inbox, co_spawn responder.

Use `steady_clock::now().time_since_epoch().count()` in milliseconds for timestamp, same pattern as `try_consume_tokens`. The `steady_clock_ms_now()` helper already exists inline in try_consume_tokens -- extract the pattern directly (cast to uint64_t of duration_cast<milliseconds>).

CRITICAL: Do NOT update `last_sync_initiated` on rejection. Only update when sync is accepted (per Research anti-pattern guidance).

**Change 2: Universal sync byte accounting (lines 444-457)**

Replace the Data/Delete-only rate check with a universal byte accounting block at the TOP of `on_peer_message`, BEFORE any message-type dispatch (before line 370). This block:

1. If `rate_limit_bytes_per_sec_ > 0`, find peer and call `try_consume_tokens(*peer, payload.size(), ...)`.
2. If tokens consumed: continue to message-type dispatch (bucket just got decremented).
3. If bucket exhausted (try_consume_tokens returns false), behavior depends on message type:
   - `Data` or `Delete`: disconnect (existing Phase 18 behavior). Increment `metrics_.rate_limited`. Return.
   - `SyncRequest`: send `SyncRejected(SYNC_REJECT_BYTE_RATE)`, increment `metrics_.sync_rejections`. Return.
   - Any other sync message (SyncAccept, NamespaceList, ReconcileInit/Ranges/Items, BlobRequest, BlobTransfer, SyncComplete): route the message anyway (let the responder handle mid-sync cutoff at namespace boundary). Do NOT disconnect or reject mid-stream. Log at debug level.
   - PeerListRequest/Response, Subscribe/Unsubscribe, Notification, StorageFull, QuotaExceeded, etc.: route normally (these are lightweight control messages, don't block them).

Then DELETE the old Data/Delete-only rate check block at lines 444-457 (it is superseded by the universal check at the top). Update the comment accordingly.

IMPORTANT: Ping/Pong are handled inside Connection::message_loop and never reach on_peer_message -- they are not affected.

**Change 3: Initiator byte budget check at namespace boundary (run_sync_with_peer)**

In `run_sync_with_peer`, at the TOP of the `for (const auto& ns : all_namespaces)` loop (line 743), BEFORE namespace filter check, add:

```cpp
// Byte budget check: if token bucket is exhausted, stop starting new namespaces
if (rate_limit_bytes_per_sec_ > 0 && peer->bucket_tokens == 0) {
    spdlog::info("sync with {}: byte budget exhausted after {} namespaces, sending SyncComplete early",
                 conn->remote_address(), missing_per_ns.size());
    break;  // Fall through to SyncComplete
}
```

This is a PEEK (no consumption) -- actual consumption happens in on_peer_message when messages arrive. The check is intentionally simple: bucket_tokens == 0 means "fully exhausted."

**Change 4: Initiator SyncRejected handling (run_sync_with_peer)**

In `run_sync_with_peer`, after the SyncAccept check (line 649), add explicit handling for SyncRejected before the generic "no SyncAccept received" fallthrough:
