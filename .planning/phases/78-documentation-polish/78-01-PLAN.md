**Section structure (per D-02 through D-06):**

### 1. Overview subsection (`### Overview`)

Brief paragraph explaining the KEM-then-Wrap pattern (per D-05):
- Random 32-byte DEK encrypts data once with ChaCha20-Poly1305
- Per-recipient: ML-KEM-1024 encapsulation produces shared secret, HKDF-SHA256 derives KEK with info label `chromatindb-envelope-kek-v1`, KEK wraps DEK via ChaCha20-Poly1305 with zero nonce
- Sender is always auto-included as a recipient
- Stanzas sorted by pk_hash for O(log N) lookup during decryption
- Content-addressed deduplication breaks on encrypted data because identical plaintext produces different ciphertext (random DEK, random nonce) -- this is expected and correct

### 2. Binary Format subsection (`### Envelope Binary Format`)

Add three tables matching the existing PROTOCOL.md table style (per D-01, D-03):

**Fixed header (20 bytes):**

| Offset | Size | Field | Encoding | Description |
|--------|------|-------|----------|-------------|
| 0 | 4 | magic | `CENV` (0x43454E56) | Envelope format identifier |
| 4 | 1 | version | uint8 | Format version (0x01) |
| 5 | 1 | suite | uint8 | Cipher suite (0x01 = ML-KEM-1024 + ChaCha20-Poly1305) |
| 6 | 2 | recipient_count | big-endian uint16 | Number of recipient stanzas (1-256) |
| 8 | 12 | data_nonce | raw bytes | Random nonce for data AEAD encryption |

**Per-recipient stanza (1648 bytes each, sorted by pk_hash):**

| Offset | Size | Field | Description |
|--------|------|-------|-------------|
| 0 | 32 | kem_pk_hash | SHA3-256 of recipient's ML-KEM-1024 public key |
| 32 | 1568 | kem_ciphertext | ML-KEM-1024 encapsulation ciphertext |
| 1600 | 48 | wrapped_dek | AEAD-encrypted DEK (32-byte key + 16-byte Poly1305 tag) |

**Data payload (variable):**

| Offset | Size | Field | Description |
|--------|------|-------|-------------|
| 20 + N*1648 | variable | ciphertext | ChaCha20-Poly1305 encrypted data + 16-byte tag |

Add a formula line: `Total envelope size: 20 + (N * 1648) + len(plaintext) + 16 bytes`

And note per-recipient overhead: `Per-recipient overhead: 1648 bytes (32 hash + 1568 KEM ciphertext + 48 wrapped DEK)`

### 3. AEAD Parameters subsection (`### AEAD Parameters`)

Two sub-subsections (per D-06):

**Data Encryption:**
- Algorithm: ChaCha20-Poly1305 (IETF, RFC 8439)
- Key: Random 32-byte DEK (generated with `secrets.token_bytes(32)`)
- Nonce: Random 12 bytes (`secrets.token_bytes(12)`) -- NOT counter-based (contrast with transport AEAD nonces which are counter-based)
- Associated data: Full envelope header (fixed header + all recipient stanzas, 20 + N*1648 bytes)
- Tag: 16 bytes (appended to ciphertext)

**DEK Wrapping:**
- Algorithm: ChaCha20-Poly1305 (IETF, RFC 8439)
- Key: 32-byte KEK derived via HKDF-SHA256 from ML-KEM shared secret with info label `chromatindb-envelope-kek-v1` and empty salt
- Nonce: 12 zero bytes (0x000000000000000000000000) -- safe because the HKDF-derived KEK is unique per KEM encapsulation (the key is never reused)
- Associated data: Partial header (fixed 20 bytes) + all pk_hash (32 bytes each) + all kem_ciphertext (1568 bytes each) accumulated so far
- Output: 48 bytes (32-byte DEK + 16-byte Poly1305 tag)

### 4. HKDF Label Registry subsection (`### HKDF Label Registry`)

Per D-04, a single reference table with all four domain labels:

| Label | Purpose | Salt | IKM | Output | Source |
|-------|---------|------|-----|--------|--------|
| `chromatin-init-to-resp-v1` | Transport send key (initiator) / recv key (responder) | empty | ML-KEM shared secret | 32 bytes | Handshake (Step 2) |
| `chromatin-resp-to-init-v1` | Transport recv key (initiator) / send key (responder) | empty | ML-KEM shared secret | 32 bytes | Handshake (Step 2) |
| `chromatindb-dare-v1` | Data-at-rest encryption key (node-level) | empty | Node master key | 32 bytes | Node storage |
| `chromatindb-envelope-kek-v1` | Envelope KEK for DEK wrapping (per-recipient) | empty | ML-KEM shared secret (per encapsulation) | 32 bytes | Envelope encryption |

Add a note: "The session fingerprint shown in the handshake diagram (`chromatin-session-fp-v1`) is NOT an HKDF derivation. Both C++ and Python compute it as `SHA3-256(shared_secret + init_pk + resp_pk)`."

### 5. Decryption Process subsection (`### Decryption`)

Brief steps:
1. Parse fixed header (validate magic `CENV`, version 0x01, suite 0x01)
2. Compute `SHA3-256(own_kem_public_key)` to get own pk_hash
3. Binary search sorted stanzas for matching pk_hash
4. If not found, reject with NotARecipientError
5. Decapsulate kem_ciphertext with own KEM secret key to recover shared secret
6. HKDF-SHA256(ikm=shared_secret, salt=empty, info="chromatindb-envelope-kek-v1") to derive KEK
7. Decrypt wrapped_dek with KEK and zero nonce to recover DEK
8. Decrypt data ciphertext with DEK and data_nonce from header, using full header as AD
