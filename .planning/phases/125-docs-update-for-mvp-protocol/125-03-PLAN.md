## Overview

chromatindb is organized in three tiers:

1. **Storage** (libmdbx, 8 DBIs) — persistent state, ACID, single-writer strand.
2. **Engine** (ingest pipeline, validation, cascade) — pure business logic,
   offloads heavy crypto to a thread pool, serializes storage access via the
   io_context strand.
3. **Net** (peer/client transport, handshake, sync orchestration) — AEAD-
   framed connections, PQ handshake, Phase A/B/C reconciliation, push-then-
   fetch replication.

Concurrency invariant (see [§Storage Strand Model](#storage-strand-model)):
all Storage mutations execute on a single thread (the `io_context` worker). Any
code path that awaits on the crypto thread pool MUST post back to `ioc_` before
touching Storage. Verified under TSAN in Phase 121's concurrent-ingest ship
gate.
```

### Storage Strand Model (~40 lines, D-05)
Dedicated subsection. Required content:

1. **The rule (one sentence):** all Storage public methods execute on the io_context owner thread.
2. **ASCII diagram** (80-col safe):
```
        io_context (single-threaded owner)
              │
              ▼
        ┌────────────────┐    co_await asio::post(ioc_, …)

4. **The pattern contributors must follow** (reproduce the code block from `<interfaces>`): offload → post back → storage.
5. **Evidence:** Phase 121 TSAN ship-gate — the concurrent-ingest stress test confirmed zero findings. Reference `.planning/phases/121-storage-concurrency-invariant/121-VERIFICATION.md` and `121-TSAN-RESULTS.md`.

### Encryption at Rest (~30 lines)
- ChaCha20-Poly1305 envelope with HKDF-derived key.
- HKDF label: `"chromatindb-dare-v1"` (link to [PROTOCOL.md §HKDF Label Registry](PROTOCOL.md#hkdf-label-registry) — existing anchor).
- Master key at `data_dir/master.key`, 0600 permissions, rejected if found unencrypted or world-readable.
- Envelope format: `[version:1][nonce:12][ciphertext+tag:N]`. Version byte reserves space for future DARE v2.
- Associated data = libmdbx key (binds ciphertext to storage location — reuse across namespaces impossible).

### PUBK-First Enforcement (~30 lines)
- Implementation side of the protocol rule.
- On ingest, lookup `target_namespace` in `owner_pubkeys` DBI:
  - If missing AND incoming blob is PUBK → register signing_pk, continue.
  - If missing AND incoming blob is NOT PUBK → reject with `0x07` error.
  - If present → resolve signing_pk and continue to Step 5.
- Applies uniformly across direct writes, delegate writes, and sync-path blob transfers.
- Test coverage: `test_pubk_first.cpp` (direct), `test_pubk_first_sync.cpp` (sync path), `test_pubk_first_tsan.cpp` (concurrent registration race — verified no owner_pubkeys corruption under TSAN).
- Cross-link: [PROTOCOL.md §PUBK-First Invariant](PROTOCOL.md#pubk-first-invariant).

### signer_hint Resolution (~30 lines)
Two code paths:
- **Owner path:** `signer_hint` looked up in `owner_pubkeys` DBI directly. Hit → resolved signing_pk. Cached in-process? No (blob_hash dedup at Step 2 short-circuits duplicate traffic before resolution ever runs; owner_pubkeys reads are not hot).
- **Delegate path:** composite key `(target_namespace, signer_hint)` looked up in `delegation` DBI. Hit → resolved signing_pk (from the delegation blob's embedded delegate pk reference).
- On resolution failure in either path, fail with `0x08 pubk_mismatch` if owner_pubkeys row exists but pk doesn't match (cross-namespace race or forged signer_hint), or `0x07 pubk_first_violation` if no owner_pubkeys row at all and incoming blob is not PUBK.
- Cross-link: [PROTOCOL.md §signer_hint Semantics](PROTOCOL.md#signer_hint-semantics).

### BOMB Cascade Side-Effect (~50 lines)
- On accepting a BOMB blob (magic 0x424F4D42, see [PROTOCOL.md §BOMB Blob Format](PROTOCOL.md#bomb-blob-format)):
  - Engine parses target_count + N×32-byte hashes.
  - For each target, emit one single-target tombstone blob (ttl=0, magic 0x54434B4E or whatever codec.h defines — verify from `db/wire/codec.h`).
  - Tombstones replicate normally via the push path (BlobNotify then BlobFetch).
  - The BOMB blob itself is stored too (content-addressable like any other).
- **ttl=0 enforced at Step 0e** (engine.cpp:161). Non-zero BOMB ttl → `0x09 bomb_ttl_nonzero`.
- **Delegate rejection** at engine.cpp:311 → `0x0B bomb_delegate_not_allowed`.
- **Structural rejection** (malformed magic/count/truncation) at engine.cpp:398 → `0x0A bomb_malformed`.
- **Wire-path note (Phase 124 Plan 05 Rule-1 fix):** BOMBs ship under `TransportMsgType_BlobWrite = 64`, not `Delete = 17`, because the node's Delete dispatcher rejects non-tombstone-format payloads. The BOMB itself is a signed blob; the tombstone cascade is a side-effect of ingestion. See STATE.md Phase 124 Plan 05 decision.

**## Net Layer**

### Handshake State Machine (~40 lines)
- Two paths: PQ (ML-KEM-1024 + ML-DSA-87 mutual auth) and lightweight (ML-DSA-87 challenge-response over shared-nonces, localhost + trusted_peers only).
- Fallback: if initiator proposes TrustedHello but responder rejects (untrusted), responder returns PQRequired; initiator retries as full PQ.
- Post-999.1 role signalling: AuthSignature payload carries a 1-byte role enum (PEER, CLIENT, reserved 0x02-0x04). ACL routing splits by declared role.
- Cross-link: [PROTOCOL.md §PQ Handshake](PROTOCOL.md#pq-handshake), [PROTOCOL.md §Role Signalling](PROTOCOL.md#role-signalling).

### AEAD Framing + Chunked Sub-Frames (~40 lines)
- Frame format: 4-byte BE length prefix + AEAD ciphertext. Nonce = 4 zero bytes + 8-byte BE counter per direction.
- Chunked sub-frame encoding for payloads ≥ 1 MiB: 0x01 flag byte + 14-byte header + 1 MiB data sub-frames + 0-byte sentinel. Each sub-frame consumes one nonce.
- **Subtle invariant (called out for future contributors):** per-connection send queue serializes outgoing messages. Concurrent sends from different coroutines would desync the AEAD nonce counter (pre-v2.0 crash class, fixed by the send queue). Never bypass the queue.
- Cross-link: [PROTOCOL.md §Transport Layer](PROTOCOL.md#transport-layer), [PROTOCOL.md §Chunked Transport Framing](PROTOCOL.md#chunked-transport-framing).

### PeerManager Decomposition (~40 lines)
- 6 components (v2.2.0 refactor):
  - `ConnectionManager` — owns peer connections (accept/connect/disconnect/reconnect).
  - `MessageDispatcher` — routes incoming messages to handlers.
  - `SyncOrchestrator` — Phase A/B/C driver.
  - `MetricsCollector` — strand-confined counters (no atomics; TSAN clean).
  - `PexManager` — inline peer exchange after sync rounds.
  - `BlobPushManager` — BlobNotify/BlobFetch push path.
- Wiring: reference injection at construction; PeerManager facade preserves the old monolithic public API.

### Sync Orchestration (~40 lines)
- Phase A: NamespaceList exchange (both sides advertise {ns, latest_seq} pairs).
- Phase B: per-namespace range-based set reconciliation via XOR fingerprints (O(diff)).
- Phase C: blob transfer for the diff — each `BlobTransfer` payload is prefixed by a 32-byte `target_namespace` (post-Phase-122; see [PROTOCOL.md §Sync Protocol](PROTOCOL.md#sync-protocol)).
- Cursor DBI tracks per-peer per-namespace progress for O(new) incremental sync.
- Safety-net full reconciliation every Nth round (`full_resync_interval`, default 10).
- BlobNotify + BlobFetch is the primary path (push-then-fetch, low latency); sync is the backstop.

### PEX + Reconnect + Inactivity (~40 lines)
- Inline PEX after sync completes (serialized per-connection to avoid nonce desync).
- `pex_interval` config controls minimum spacing.
- Reconnect with jittered exponential backoff (1s-60s). ACL rejections get extended backoff (3 rejections → 600s).
- Receiver-side inactivity timeout (`inactivity_timeout_seconds`, default 120s). No sender-side Ping because bidirectional keepalive traffic could desync AEAD nonces.

### Role Classification (~20 lines)
- Post-999.1 AuthSignature role byte: PEER → peer-ACL list, CLIENT → client-ACL list.
- Implicit localhost + `trusted_peers` → lightweight handshake allowed.
- UDS connections always map to CLIENT role, local-trust.
