**2. §signer_hint Semantics (~25 lines)**
   - 32 bytes. Equals `SHA3-256(signer's ML-DSA-87 signing_pk)`. For owner writes this equals the target namespace; for delegate writes it equals the delegate's own namespace (different from target).

**1. §NAME Blob Format (~50 lines)**
   - 4-byte magic: `0x4E414D45` ("NAME", ASCII).
   - Purpose: a mutable-name pointer. Multiple NAME blobs may exist for the same (namespace, name_label); the resolver picks the winner.
   - Payload layout (derive byte-level from `db/wire/codec.h:160` + `cli/src/wire.cpp:382+`):
     ```
     [ 4 bytes : 0x4E 0x41 0x4D 0x45  "NAME" magic ]
     [ 1 byte  : name_label_len (max 64)            ]
     [ N bytes : name_label (UTF-8, N = name_label_len) ]
     [32 bytes : target_content_hash                ]
     ```
     (Verify exact byte layout from `cli/src/wire.cpp` encode_name_blob and `db/wire/codec.cpp` decode; write from the code, not from memory.)
   - Resolver winner rule: for a (namespace, name_label) pair, the winner is the NAME blob with the highest `timestamp`, with `blob_hash DESC` as tiebreak when timestamps tie.
   - Known quirk (STATE.md FLAG): within a single second, a `--replace` write can lose the tiebreak against the prior NAME blob. Add a boxed `> Note:` documenting the 1-second granularity contract. This is a pre-existing FLAG from Phase 123 D-15 — Plan 1 documents it; fix is deferred.
   - Cascade on delete: tombstoning a NAME blob severs the binding but does not delete the target blob.
   - Source: `db/wire/codec.h:160`, `cli/src/wire.h:354`, `cli/src/wire.cpp:382+`, `123-01-SUMMARY.md`, `123-02-SUMMARY.md`.

**2. §BOMB Blob Format (~60 lines)**
   - 4-byte magic: `0x424F4D42` ("BOMB", ASCII).
   - Purpose: batched tombstone. One BOMB blob carries N target content hashes; on ingest the node emits one single-target tombstone per target as a side-effect (the "cascade").
   - Payload layout (derive byte-level from `db/wire/codec.h:191` + `cli/src/wire.h:357`):
     ```
     [ 4 bytes : 0x42 0x4F 0x4D 0x42  "BOMB" magic ]
     [ 4 bytes : target_count (BE uint32)          ]
     [32 * target_count bytes: target content hashes, concatenated ]
     ```
   - **Invariants (all node-enforced at ingest):**
     a. `ttl == 0` (permanent, same as single-target tombstones). Non-zero ttl → error `0x09 ERROR_BOMB_TTL_NONZERO`.
     b. Structural decode must succeed (magic, count, length match). Failure → error `0x0A ERROR_BOMB_MALFORMED`.
     c. Delegates MAY NOT emit BOMB blobs — owner-only. Delegate attempts → error `0x0B ERROR_BOMB_DELEGATE_NOT_ALLOWED`.
   - **Wire path (important, per STATE.md Phase 124 Plan 05 "Rule-1 fix"):** BOMB blobs are structurally regular blobs and ship under `TransportMsgType_BlobWrite = 64`, NOT under `Delete = 17`. The Delete dispatcher rejects non-tombstone-format payloads; routing BOMB through BlobWrite bypasses that check and lets the engine's BOMB cascade emit the per-target tombstones.
   - Cascade side-effect: after accepting the BOMB, the engine writes one single-target tombstone per target hash. Those tombstones replicate normally. The BOMB blob itself is stored too (it's content-addressable like any other blob).
   - D-06 BOMB cascade for CPAR manifests (Phase 124): deleting a CPAR manifest hash via `cdb rm` (single-target or batched) triggers a cascade that includes all CDAT chunks referenced by the manifest. This is a CLI convenience; the node sees a BOMB with all target hashes already expanded.
   - Source: `db/wire/codec.h:191`, `db/engine/engine.cpp:161,311,398`, `cli/src/wire.h:357`, `123-03-SUMMARY.md`, `123-04-SUMMARY.md`, STATE.md Phase 124 Plan 05 decision.
