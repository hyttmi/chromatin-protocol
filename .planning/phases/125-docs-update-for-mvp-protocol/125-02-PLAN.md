**A. Section: "## Mutable Names (put --name, get <name>, --replace)"**
   Describe the user-visible behavior of Phase 123 NAME blobs; defer byte-level detail to PROTOCOL.md.

   Content:
   - `cdb put --name <name> <file>` uploads the file blob AND emits a NAME blob binding `name` in your namespace to the file's content hash. Future `cdb get <name>` resolves to that content hash.
   - `cdb put --name <name> --replace <file>` overwrites an existing binding. Behind the scenes it emits a new NAME blob with a later timestamp; the resolver picks the latest one.
   - `cdb get <name>` resolves the name via two paths:
     1. Local `~/.cdb/name_cache.json` fast-path (if seen before).
     2. Enumerate NAME blobs in your namespace from the node; the winner is `timestamp DESC`, with `blob_hash DESC` as tiebreak.
   - **Known 1-second quirk** (STATE.md Phase 123 D-15 FLAG): within a single second, a `--replace` write can lose the tiebreak against the prior NAME blob. If you need sub-second overwrite semantics, wait one second or re-issue.
   - Examples block: 4-5 realistic commands showing the full loop.
   - Cross-link: [PROTOCOL.md §NAME Blob Format](../db/PROTOCOL.md#name-blob-format) (anchor created by Plan 1 Task 3).

**B. Section: "## Batched Deletion and CPAR Cascade (rm)"**
   Describe the user-visible behavior of Phase 123 BOMB + Phase 124 D-06 CPAR cascade.

   Content:
   - `cdb rm <hash>` deletes a single blob by emitting a tombstone.
   - `cdb rm <hash1> <hash2> <hash3> ...` deletes many blobs in a single batch via a BOMB blob; the node expands the batch on ingest and emits one tombstone per target. One round-trip for arbitrarily many deletions.
   - **CPAR manifest cascade:** if any target hash resolves to a CPAR manifest (a chunked-upload manifest from `cdb put` on a file >400 MiB), the CLI automatically expands the delete to include ALL referenced CDAT chunk hashes. You get a summary line before submission. Example: deleting a 750 MiB file's CPAR hash removes 1 CPAR + ~3 CDAT chunks in a single BOMB.
   - **Delegates can't BOMB:** only owner identities can emit BOMB blobs. Delegate accounts get `Error: delegates cannot perform batch deletion on this node.`
   - Examples block: single-target, batched, and CPAR cascade scenarios.
   - Cross-link: [PROTOCOL.md §BOMB Blob Format](../db/PROTOCOL.md#bomb-blob-format).

**A. Section: "## Chunked Large Files" (~15 lines)**
   - `cdb put` auto-chunks any file larger than 400 MiB into CDAT chunk blobs plus a CPAR manifest blob.
   - `cdb get <cpar-hash>` automatically reassembles all referenced chunks into the original file. No user-visible difference from a regular download other than bandwidth/time.
   - `cdb rm <cpar-hash>` cascades to all CDAT chunks (see Batched Deletion section above).
   - If you want to tune the chunk size, use `--chunk-size <bytes>` on `put` (if exposed — verify via `cli/src/main.cpp` or omit).
   - Cross-link: [PROTOCOL.md §Chunked Transport Framing](../db/PROTOCOL.md#chunked-transport-framing) (existing anchor).

**B. Section: "## Request Pipelining" (~8 lines)**
   - Multi-hash `cdb get h1 h2 h3 h4` uses request pipelining over a single PQ connection. Default depth is 8 in-flight requests.
   - Transparent — no user-visible flag. Operator-visible only as "fast".
   - Cross-link: [PROTOCOL.md §Transport Layer](../db/PROTOCOL.md#transport-layer) (for request_id detail — existing anchor).

**C. Section: "## Auto-PUBK and First-Write Errors" (~15 lines)**
   - First write on a node: the CLI automatically emits a PUBK blob before your user blob, populating the node's owner_pubkeys registry. Invisible if it succeeds.
   - If auto-PUBK fails (network, ACL, misconfigured node), you'll see:
     > `Error: namespace not yet initialized on node <host>. Auto-PUBK failed; try running 'cdb publish' first.`
   - Running `cdb publish <host>` explicitly is always safe and gives clearer feedback.
   - Other PUBK-related errors (owned by a different key, delegate restrictions, malformed BOMB payloads) produce similar one-line messages; see [PROTOCOL.md §ErrorResponse](../db/PROTOCOL.md#errorresponse) for the full table.

   - `ls --raw` — Include infrastructure blobs with their 4-byte type prefix.
   - `ls --type <TYPE>` — Filter by 4-byte type (e.g. `--type BOMB`, `--type NAME`). Server-side; fast.
