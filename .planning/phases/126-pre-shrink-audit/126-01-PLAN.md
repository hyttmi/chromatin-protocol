Pin the send-side streaming invariant in code so that no future change can produce a single non-chunked AEAD frame exceeding STREAMING_THRESHOLD (plus envelope overhead) on either the node (db/) or the CLI (cli/) side of the wire. 1. Compile-time invariant tying MAX_FRAME_SIZE to STREAMING_THRESHOLD on the node side (D-05).
2. Runtime assertion at the narrow waist of the node-side non-chunked send path (D-04).
3. Symmetric compile-time invariant on the CLI side (D-14).

### Design rationale the executor must preserve

- **No shared helper.** The node side and CLI side each have their own `MAX_FRAME_SIZE` and `STREAMING_THRESHOLD` constants in separate compilation units (chromatindb::net::MAX_FRAME_SIZE vs file-local CLI constant). The assertions are one line each and the constants cannot be meaningfully shared without restructuring both codebases. Duplication is minimal and intentional — documented here so a later reader does not factor it out under the `feedback_no_duplicate_code.md` banner. That guidance targets non-trivial duplicated logic, not two parallel one-line invariants.
- **Runtime assertion uses `assert()`, not release-safe abort.** Debug-only per the researcher's Q1 recommendation. The streaming invariant breach is a programmer error caught by CI test runs (debug build), not a production runtime condition.
- **Assertions reference `STREAMING_THRESHOLD`, not `MAX_FRAME_SIZE`.** Per D-11, so the assertions and the test stay meaningful BEFORE and AFTER Phase 128 shrinks MAX_FRAME_SIZE to 2 MiB.
- **No "Phase 126" / "AUDIT-0X" / "v4.2.0" / "FRAME-01" tokens** in user-visible strings (assert messages, TEST_CASE names, code comments that ship in the binary). Per memory `feedback_no_phase_leaks_in_user_strings.md`. Tags like `[connection][streaming]` are fine.
- **No full-suite test run in tasks.** Per memory `feedback_delegate_tests_to_user.md` and `feedback_no_test_automation_in_executor_prompts.md`. Acceptance criteria reference `grep` and "file exists"; the user runs tests.
