The originating node responds with BlobFetchResponse (type 61). This is handled inline in the message loop -- no sync session handshake required. Pending fetch dedup: only one BlobFetch per hash is in-flight at a time per connection.
     - Wire format table for BlobFetch (type 60) -- 64-byte payload:
       | Field | Offset | Size | Encoding | Description |
       namespace_id: 0, 32, raw bytes, Target namespace
       blob_hash: 32, 32, raw bytes, Hash of the blob to fetch
     - Wire format table for BlobFetchResponse (type 61) -- variable-length payload:
       Found: [0x00][flatbuffer_encoded_blob]
       Not found: [0x01]
       Note: Status byte convention differs from ReadResponse (type 32), which uses 0x01 for found and 0x00 for not-found. BlobFetchResponse uses 0x00 for found, 0x01 for not-found.

This catches any blobs missed by the push-then-fetch path (e.g., due to transient errors). The interval is reloadable via SIGHUP.
