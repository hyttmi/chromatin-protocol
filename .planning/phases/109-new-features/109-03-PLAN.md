**Part B: Record in route_response (per D-04, Pitfall 6)**

In `relay/core/uds_multiplexer.cpp` `route_response()`, find the section where `request_id > 0` is handled. After `auto pending = router_.resolve_response(request_id);` succeeds (the `if (!pending)` early return), and BEFORE the `binary_to_json` translation call, add the WriteTracker recording:

```cpp
// FEAT-01: Record writer session for source exclusion on subsequent Notification
// WriteAck(30) and DeleteAck(18) both have blob_hash at offset 0, 32 bytes (per D-04, Pitfall 6)
if ((type == 30 || type == 18) && payload.size() >= 32) {
    BlobHash32 blob_hash{};
    std::memcpy(blob_hash.data(), payload.data(), 32);
    write_tracker_.record(blob_hash, pending->client_session_id);
}
```

This must be AFTER resolve_response (which gives us client_session_id) and BEFORE send_json (which consumes the response). The UDS ordering guarantee ensures WriteAck arrives before the corresponding Notification (see Research Pitfall 1).

**Part C: Filter in handle_notification (per D-04)**

In `relay/core/uds_multiplexer.cpp` `handle_notification()`, modify the fan-out loop. After getting `session_ids` from tracker and before the fan-out loop, add blob_hash extraction and writer lookup:

```cpp
if (payload.size() >= 64) {
```

Then modify the fan-out loop to skip the writer:

```cpp
for (uint64_t sid : session_ids) {

    auto session = sessions_.get_session(sid);
    if (session) {
        session->send_json(*json_opt);
    }
}
```

Add `using chromatindb::relay::core::BlobHash32;` at the top of the file or use the fully qualified type. Since the file is inside the `chromatindb::relay::core` namespace (check the cpp file's namespace), `BlobHash32` should be directly accessible after including write_tracker.h.
