Create three files:

**1. `.dockerignore`** with these exclusions:
```

```
Include `tests/` because the runtime image does not need test source files and excluding them reduces build context. Do NOT exclude `bench/` because `chromatindb_bench` is a build target.

**2. `Dockerfile`** -- Multi-stage build following all locked decisions from CONTEXT.md:

Build stage (FROM debian:bookworm AS builder):
- Install build deps: build-essential, cmake, git, libssl-dev, pkg-config (no ninja -- single-threaded build per user preference)
- WORKDIR /src
- COPY CMakeLists.txt, db/, bench/ into /src
- Single RUN command with `--mount=type=cache,target=/src/build/_deps` that:
  - Runs `cmake -S . -B build -DCMAKE_BUILD_TYPE=Release -DFETCHCONTENT_QUIET=ON -DCMAKE_CXX_FLAGS="-Wno-restrict"` (suppress GCC 12 Bug #105329 false positive)
  - Runs `cmake --build build --target chromatindb chromatindb_bench` (NOT chromatindb_loadgen -- it does not exist yet, Phase 28 will add it)
  - Do NOT use `--parallel` flag (user preference: eats all memory)
- Strip binaries: `strip build/chromatindb build/chromatindb_bench`

Runtime stage (FROM debian:bookworm-slim):
- Install runtime deps: `libssl3` only (for libcrypto.so.3), clean apt lists
- Create non-root user: `groupadd -r chromatindb && useradd -r -g chromatindb chromatindb`
- COPY --from=builder both binaries to /usr/local/bin/
- Create /data dir owned by chromatindb user, declare VOLUME /data
- USER chromatindb
- EXPOSE 4200
- HEALTHCHECK with interval=30s, timeout=5s, start_period=10s, retries=3 using bash /dev/tcp/127.0.0.1/4200
- ENTRYPOINT ["chromatindb"]
- CMD ["run", "--data-dir", "/data", "--log-level", "debug"]

Use `# syntax=docker/dockerfile:1` at the top for BuildKit cache mount support.

**3. `db/version.h`** -- Bump version to 0.6.0:
```cpp
#define VERSION_MAJOR "0"
#define VERSION_MINOR "6"
#define VERSION_PATCH "0"

static constexpr const char* VERSION = VERSION_MAJOR "." VERSION_MINOR "." VERSION_PATCH;
```

**Why --mount=type=cache instead of CMakeLists-first layer:** Per research Pitfall 1, copying only CMakeLists.txt and running cmake configure fails because CMake validates source file existence at configure time (db/main.cpp referenced in add_executable). Cache mount achieves the same goal -- source-only changes reuse cached FetchContent downloads.

**Why no chromatindb_loadgen:** Per research Open Question 1, the loadgen binary does not exist yet (Phase 28 creates it). Build what exists now. Phase 28 will update the Dockerfile to add the loadgen target.

**Why -Wno-restrict:** GCC 12.2 (bookworm default) has Bug #105329: false -Wrestrict warnings with C++20 + -O2/-O3. These are warnings not errors, but suppressing them keeps the build output clean. Does not affect correctness.
