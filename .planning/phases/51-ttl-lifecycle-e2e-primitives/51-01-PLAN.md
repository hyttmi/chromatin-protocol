**test_ttl01_tombstone_propagation.sh** -- TTL-01: Tombstone Propagation (100 blobs)

Topology: 3-node open-mode cluster using manual container creation (standalone docker run pattern, not compose). Use dedicated subnet 172.39.0.0/16 and unique container names chromatindb-ttl01-node{1,2,3}. Named volumes chromatindb-ttl01-node{N}-data. Config: sync_interval_seconds=5, full_resync_interval=9999, inactivity_timeout_seconds=0.

Flow:
1. Start node1, generate owner identity (--identity-save), ingest 100 blobs with --count 100 --size 256 --ttl 3600. Capture blob hashes from loadgen stdout JSON (.blob_hashes array).
2. Start node2 + node3 bootstrapping to node1. Wait for sync convergence to 100 blobs on all 3 nodes (wait_sync).
3. Owner deletes all 100 blobs via tombstones: pipe the 100 hashes (one per line) to loadgen --delete --hashes-from stdin --ttl 0 --identity-file.
4. Wait for tombstones to sync: wait_sync for blob count to reach 200 on all nodes (100 original + 100 tombstones). Use 120s timeout.
5. Check 1: All 3 nodes have 200 blobs (originals + tombstones).
6. Check 2: Verify tombstones are active -- use SIGUSR1 metrics dump and check integrity_scan log output for tombstone map entries >= 100 on node2. Alternatively, verify by attempting to re-ingest a blob with the same hash -- the tombstone should block it.
7. Check 3: Verify original blobs are "dead" -- the blob count metric from SIGUSR1 includes tombstones, but the tombstone_map entries in integrity scan logs confirm tombstones are tracked. Run integrity scan on node2 by restarting it (startup runs integrity_scan automatically) and grep for "tombstone=" in logs showing >= 100 entries.

Cleanup: remove containers, volumes, network, temp dirs.

**test_ttl03_permanent_blobs.sh** -- TTL-03: TTL=0 Permanent Blobs

Topology: 2-node standalone topology. Subnet 172.40.0.0/16, containers chromatindb-ttl03-node{1,2}. Same config as TTL-01.

Flow:
1. Start node1. Ingest 10 blobs with --ttl 0 (permanent) and 10 blobs with --ttl 60 (ephemeral). Use two separate identities (two loadgen runs with --identity-save).
2. Start node2 bootstrapping to node1. Wait for sync to 20 blobs.
3. Record blob counts on both nodes (should be 20 each).
4. Wait for ephemeral blobs to expire: sleep 70 (TTL=60s + safety margin). The expiry_scan_loop runs every 60s, so wait for one full cycle after expiry: sleep 130 total (60s TTL + 70s for GC scan to fire).
5. Trigger explicit check: send SIGUSR1 to get blob counts after GC.
6. Check 1: Blob count on node1 decreased from 20 to 10 (ephemeral blobs expired, permanent remain).
7. Check 2: Blob count on node2 also decreased to 10.
8. Check 3: Verify permanent blobs still exist -- the remaining 10 blobs are the TTL=0 ones.

Note: The expiry scan runs every 60 seconds (hardcoded in peer_manager.cpp). With TTL=60, blobs expire at t+60s. The scan at t+60s (or the next scan after) will catch them. Total wait: ~130s max (60s TTL + 60s scan cycle + 10s margin).

**test_ttl04_delegate_cannot_delete.sh** -- TTL-04: Delegate Cannot Delete

Topology: 3-node open-mode cluster via docker-compose.acl.yml (reuse existing). Container names chromatindb-test-node{1,2,3} on 172.28.0.0/16 subnet.

Flow:
1. Start 3-node cluster using docker-compose.acl.yml. Wait healthy on all 3.
2. Generate owner identity (--identity-save to temp dir), ingest 5 blobs to node1 with --ttl 3600.
3. Generate delegate identity, discover owner namespace hex from loadgen stderr.
4. Owner creates delegation blob for delegate on node1 (--delegate PUBKEY_HEX --ttl 0).
5. Wait for delegation to propagate (sleep 20 + wait_sync).
6. Delegate writes 3 blobs to owner's namespace on node2 (--namespace OWNER_NS_HEX). Verify 0 errors (pre-condition: delegation works).
7. Wait for delegate blobs to sync.
8. Delegate attempts tombstone: pipe one of the delegate-written blob hashes to loadgen --delete --hashes-from stdin --namespace OWNER_NS_HEX --identity-file /delegate-identity --ttl 0 targeting node1.
9. Check 1: Tombstone rejected -- grep node1 logs for "delegates cannot create tombstone" OR "SHA3-256(pubkey) != namespace_id".
10. Check 2: Owner tombstones the same blob successfully (using owner identity + --delete). Grep for "Blob deleted via tombstone" in node1 logs.
11. Check 3: Blob count stable -- delegate's tombstone attempt did not increase count, but owner's tombstone did (count = pre-count + 1).

This test is more thorough than ACL-03's check 3 because it also verifies the positive case (owner CAN delete) and count stability.
