Create test_dr01_dare_forensics.sh:
- Topology: Single node (docker run). Dedicated network (172.32.0.0/16, chromatindb-dr01-test-net). Named volume for data.
- Step 1: Start node, wait healthy.
- Step 2: Ingest blobs with known plaintext markers via loadgen. Use a distinctive marker string. After loadgen, also ingest a blob with well-known content by running loadgen with --size 1024 (the content will contain the namespace ID since loadgen generates deterministic data). The key verification is that NO plaintext appears in the database file.
- Step 3: Stop the node (docker stop). Copy data.mdb out of the volume:
  `docker run --rm -v VOLUME:/data -v /tmp:/out alpine cp /data/data.mdb /out/dr01_data.mdb`
- Step 4: Primary verification -- grep for known strings in the hex dump:
  - Get the node's namespace ID from logs (grep namespace: in logs).
  - `xxd /tmp/dr01_data.mdb | grep -i NAMESPACE_HEX_PREFIX` -- should find ZERO matches (first 8 chars of namespace hex).
  - The namespace is a 64-char hex string. Search for a 16-char substring of it in the raw binary: `grep -c "SUBSTRING" /tmp/dr01_data.mdb` should be 0.
  - Note: some metadata keys may contain namespace bytes as mdbx keys (the AD for AEAD), so focus on VALUE content. Use `strings /tmp/dr01_data.mdb | grep -i SUBSTRING` as the primary check -- no human-readable namespace substrings should appear in strings output.
- Step 5: Supplementary entropy check -- sample 1000 bytes from middle of file, count unique byte values. Encrypted data should have near-uniform distribution (200+ unique byte values out of 256 for a 1000-byte sample from the middle of the file).
- Cleanup temp files in trap.

Create test_dr02_dr03_master_key.sh:
- Topology: 2 nodes (docker run). Dedicated network (172.33.0.0/16, chromatindb-dr02-test-net). Named volumes for both nodes.
- DR-02 (master key dependency):
  - Step 1: Start Node A, ingest 20 blobs, stop it.
  - Step 2: Remove master.key from Node A's volume:
    `docker run --rm -v VOLUME_A:/data alpine rm /data/master.key`
  - Step 3: Try to start Node A. It should fail to start (exit non-zero or crash immediately). Check exit code != 0 or container not healthy after 30s.
  - Step 4: Restore master.key (from a backup or regenerate -- but regenerated key won't decrypt old data, so this is also a DR-03 negative test).
- DR-03 (master key isolation):
  - Step 5: Start Node B (separate volume, generates its own master.key). Ingest 10 blobs to Node B. Stop Node B.
  - Step 6: Copy Node A's data.mdb to Node B's volume (overwrite Node B's data.mdb with Node A's):
    `docker run --rm -v VOLUME_A:/src:ro -v VOLUME_B:/dst alpine cp /src/data.mdb /dst/data.mdb`
  - Step 7: Start Node B (has Node B's master.key but Node A's data.mdb). Expect AEAD decryption failure. Node B should either fail to start, crash, or start but serve 0 blobs (integrity scan should report errors). Check logs for "integrity" or "decrypt" or "AEAD" errors. Verify blob count is 0 via SIGUSR1 if node starts.

Create test_dr04_data_migration.sh:
- Topology: 2 nodes initially, then 3rd node from copied data. Dedicated network (172.34.0.0/16, chromatindb-dr04-test-net). Named volumes.
- Step 1: Start Node A and Node B. Wait healthy. Let them connect.
- Step 2: Ingest 100 blobs to Node A. Wait for sync to Node B (100 blobs).
- Step 3: Stop Node A.
- Step 4: Copy Node A's entire data_dir to a new volume (Node C):
  `docker run --rm -v VOLUME_A:/src:ro -v VOLUME_C:/dst alpine sh -c 'cp -a /src/* /dst/'`
  This copies data.mdb, master.key, node.key, and any other files.
- Step 5: Start Node C with the copied volume, same config but different container name, same IP as Node A (or new IP with Node B pointing to it). Use Node A's IP (stop Node A first, reuse IP). Node C's config: bootstrap to Node B.
- Step 6: Wait for Node C healthy. Verify:
  - Node C has peer connections (peer count >= 1 via SIGUSR1)
  - Node C blob count == 100 (existing blobs preserved)
  - Node C serves blobs via sync (ingest 20 new blobs to Node B, wait for sync to Node C -- proves Node C can receive new blobs)
- Step 7: Cursor test -- ingest 30 more blobs to Node B. Node C should sync only the 30 new blobs (cursor state preserved). Verify final blob count on Node C is 150 (100 original + 20 + 30).

Create test_dr05_crash_recovery_integrity.sh:
- This is separate from NET-05 per user decision. NET-05 tests sync convergence after crash. DR-05 tests MDBX transaction integrity and cursor resumption.
- Topology: 2-node standalone. Dedicated network (172.35.0.0/16, chromatindb-dr05-test-net). Named volumes.
- Step 1: Start 2 nodes, wait healthy, let them connect and establish sync.
- Step 2: Ingest 200 blobs to Node1. Wait for sync to Node2 (200 blobs).
- Step 3: Ingest 100 more blobs to Node1 (total 300 on Node1). Wait for Node2 to have 300.
- Step 4: Start slow background loadgen to Node1 (--count 500 --rate 10 --size 4096).
- Step 5: Wait 5s, then SIGKILL Node2 during active sync.
- Step 6: Wait for background loadgen to complete. Get Node1 total blob count.
- Step 7: Restart Node2. Wait healthy. Verify "integrity scan" in restart logs.
- Step 8: Cursor resumption check: Wait for sync convergence. The key is that Node2 does NOT do a full resync -- it resumes from its cursor. Verify via metrics: check full_resyncs counter in SIGUSR1 output. After restart, full_resyncs should be 0 (cursor state preserved, no full resync needed). Also verify blob count matches Node1.
- Step 9: Verify no data loss -- Node2 final count >= Node1 count (all committed data intact).
