Protocol messages:
- StorageFull (type 23): Empty payload. Sent when max_storage_bytes exceeded. Peers suppress sync pushes.
- QuotaExceeded (type 26): Empty payload. Sent when namespace quota exceeded. Only affects that namespace.

Engine ingest flow (from db/engine/engine.cpp):
```cpp
if (max_storage_bytes_ > 0 && !wire::is_tombstone(blob.data)) {
    if (storage_.used_bytes() >= max_storage_bytes_) {
        // Returns storage_full error
    }
}
```

Recovery mechanism:
- SIGHUP reloads config including max_storage_bytes
- After raising the limit, StorageFull state clears on next write attempt
- Peers resume pushing on next reconnection cycle

SIGHUP writable config pattern from ACL-05:
- Config file mounted without :ro
- Edit config on host, docker kill -s HUP

Create test_dos04_storage_full.sh:
- Topology: 2-node standalone. Dedicated network (172.39.0.0/16, chromatindb-dos04-test-net). Named volumes. Writable config for Node1 (SIGHUP recovery).
- Node1: max_storage_bytes set to a tight limit. Start with max_storage_bytes=0 (unlimited) initially to let some data in, then SIGHUP to a tight limit. Actually better: start with max_storage_bytes=204800 (200KB) so it fills quickly.
- Node2: bootstraps to Node1.
- Step 1: Start Node1 with max_storage_bytes=204800 (200KB). Start Node2. Wait healthy.
- Step 2: Ingest blobs (--size 4096 --rate 20 --count 100) to Node1. At ~4KB per blob, after ~50 blobs Node1 should be near 200KB. Loadgen may report some failures as storage fills -- that is expected.
- Step 3: Check Node1 logs for "storage" or "StorageFull" or "storage_full" evidence. Check SIGUSR1 metrics for evidence of rejections > 0.
- Step 4: Verify sync suppression -- Node2 should NOT have received all blobs that were rejected. The blob count on Node2 should be less than or equal to what Node1 successfully stored (not 100).
- Step 5: Recovery -- edit Node1 config to max_storage_bytes=10485760 (10MB, plenty of room). Send SIGHUP. Verify "SIGHUP received" in logs.
- Step 6: Ingest 20 more blobs to Node1 (--size 4096 --rate 20 --count 20). These should succeed now. Verify Node1 blob count increased. Wait for sync to Node2. Verify Node2 received the new blobs.

Create test_dos05_namespace_quotas.sh:
- Topology: Single node with 3 separate loadgen identities (3 namespaces). Dedicated network (172.40.0.0/16, chromatindb-dos05-test-net). Named volume.
- Node1 config: namespace_quota_count=5 as global default (tight: only 5 blobs per namespace). No per-namespace overrides initially.
- Step 1: Start Node1. Wait healthy.
- Step 2: Generate 3 key pairs by starting 3 temporary containers with separate volumes, extracting their namespace IDs, then stopping them. Or simpler: use run_loadgen 3 times with different --key-file options. Actually, loadgen generates its own key if none specified and uses its own namespace. So just run 3 separate loadgen invocations -- each creates its own namespace.
- Step 3: First namespace -- ingest 10 blobs (--count 10 --size 1024 --rate 10). After 5 blobs, quota should be hit. Loadgen may report errors. Check Node1 logs for "QuotaExceeded" or "quota" evidence. Check SIGUSR1 metrics for quota_rejections > 0.
- Step 4: Second namespace -- ingest 5 blobs (exactly at quota). These should all succeed (5 <= 5).
- Step 5: Third namespace -- ingest 3 blobs. These should all succeed (3 <= 5). Verify Node1 blob count shows blobs from all 3 namespaces (the accepted ones).
- Step 6: Verify namespace isolation -- the first namespace hitting quota did NOT prevent the second and third from writing. quota_rejections should be > 0 (from namespace 1) but namespace 2 and 3 blobs all accepted.

Create test_dos06_thread_pool_saturation.sh:
- Topology: Target node + 4-6 concurrent loadgen instances + 1 fresh connector node. Dedicated network (172.41.0.0/16, chromatindb-dos06-test-net). Named volumes.
- Node1 (target): standard config, no rate limits (we want to saturate crypto, not get rate-limited).
- Step 1: Start Node1. Wait healthy.
- Step 2: Launch 4 concurrent loadgen containers in background (--size 4096 --rate 200 --count 500 each, run as `docker run --rm ... &`). Each loadgen generates its own key, so ML-DSA-87 verify is called for every blob from every loadgen. 4 x 200/sec = 800 ingest attempts/sec, each requiring an ML-DSA-87 signature verification on the thread pool. Track their PIDs.
- Step 3: While loadgen is hammering the node, verify event loop is not starved:
  - Check 1 (new connection): Start Node2 (fresh node, bootstraps to Node1). Wait for it to become healthy AND verify it connects to Node1 (check Node1 peer count via SIGUSR1). Connection accepted = handshake completed = event loop responsive. Allow generous timeout (30s) since the node is under load.
  - Check 2 (metrics responsive): Send SIGUSR1 to Node1. Verify metrics line appears in logs within 5s. If metrics dump responds, event loop is processing signals.
- Step 4: Wait for loadgen containers to finish (wait on PIDs, 120s timeout). Kill any stragglers.
- Step 5: Final verification -- Node1 is still running and healthy. Blob count > 0 (some blobs were accepted despite saturation). Peer count shows Node2 connected.
- The key insight: we are NOT testing that all blobs succeed (some may be dropped under extreme load). We are testing that the event loop remains responsive (new connections + signal handling work even when thread pool is saturated with ML-DSA-87 verifications).
