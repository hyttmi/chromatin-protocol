# Interfaces and context needed for the tasks
From `db/peer/peer_manager.cpp`:
The `on_peer_message` function processes ~15 different message types.
Responses are sent from inside `asio::co_spawn` lambdas.
**Pattern:**
```cpp
// Update this:
asio::co_spawn(ioc_, [this, conn, payload = std::move(payload)]() -> asio::awaitable<void> {
    // ...
    co_await conn->send_message(wire::TransportMsgType_WriteAck, ...);
}, asio::detached);

// To this (capture request_id by value, pass to send_message):
asio::co_spawn(ioc_, [this, conn, request_id, payload = std::move(payload)]() -> asio::awaitable<void> {
    // ...
    co_await conn->send_message(wire::TransportMsgType_WriteAck, ..., request_id);
}, asio::detached);
```

Per D-06: All `send_message` calls echo the received `request_id`.
Per D-08: `StorageFull` and `QuotaExceeded` echo the `request_id` from the `Data` message that triggered them.
Per D-09: `Notification` messages are server-initiated, always `request_id = 0` (handled by default parameter).
</context>
