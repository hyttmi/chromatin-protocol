# Phase 999.2: Node Silent Failure on Malformed Requests - Research

**Researched:** 2026-04-11
**Domain:** C++ node wire protocol error handling + relay JSON translation
**Confidence:** HIGH

## Summary

The chromatindb node currently has approximately 39 code paths where a client-facing request triggers `record_strike_()` + `co_return` with no response sent back. The client (or relay's RequestRouter) waits indefinitely for a response that never arrives, eventually hitting a 60-second stale cleanup timeout. This phase adds ErrorResponse (type 63) as a new wire protocol message type so every client-facing request guarantees a response.

The implementation spans three layers: (1) the node's FlatBuffers schema and message dispatcher, (2) PROTOCOL.md documentation, and (3) the relay's type registry, message filter, JSON schema, and translator. The error response format is fixed at 2 bytes: `[error_code:1][original_type:1]`, using the transport envelope's existing `request_id` field for correlation.

**Primary recommendation:** Add ErrorResponse = 63 to transport.fbs, create a small `send_error_response()` helper in message_dispatcher.cpp, and systematically replace every `record_strike_(conn, ...); co_return;` with `record_strike_(conn, ...); co_await send_error_response(conn, ERROR_CODE, type, request_id); co_return;` for all client-facing handlers.

## User Constraints (from CONTEXT.md)

### Locked Decisions
- D-01: New ErrorResponse message type (type 63) added to the wire protocol. Fixed binary format: `[error_code:1][original_type:1]`. Echoes the client's `request_id` in the transport envelope so relay/client can match it to the pending request.
- D-02: The relay's type_registry and translator must be updated to handle type 63 (ErrorResponse) -- translate to JSON `{"type": "error", "code": "<name>", "original_type": "<name>"}` for WebSocket clients.
- D-03: Error code is a single byte from a fixed enum. No detail strings, no payload sizes, no internal state leaked. Codes: `malformed_payload(1)`, `unknown_type(2)`, `decode_failed(3)`, `validation_failed(4)`, `internal_error(5)`.
- D-04: The relay translates error code bytes to human-readable JSON names (e.g. code 1 -> `"malformed_payload"`). The node wire format stays compact.
- D-05: Only client-facing request types send ErrorResponse on failure: Data(8), Delete(17), ReadRequest(10), ListRequest(12), StatsRequest(14), ExistsRequest(24), BatchReadRequest(26), NodeInfoRequest(28), PeerInfoRequest(30), TimeRangeRequest(57), MetadataRequest(16), Subscribe(19), Unsubscribe(20).
- D-06: Peer-to-peer types (sync, PEX, BlobNotify/BlobFetch) stay silent -- peers don't wait for responses on these paths.
- D-07: Unknown/unrecognized message types also get ErrorResponse with code `unknown_type`.
- D-08: PROTOCOL.md must be updated with the new ErrorResponse type (63), its wire format, and the full error code enum table.
- D-09: Error codes must be documented in PROTOCOL.md alongside existing message type documentation.
- D-10: The relay must fully support ErrorResponse: type_registry entry, message_filter allowlist, JSON translation (`binary_to_json` for type 63), and forwarding to the originating WebSocket client via the request_router (using the echoed request_id).
- D-11: Relay JSON format for ErrorResponse: `{"type": "error", "code": "<code_name>", "original_type": "<type_name>"}`. The relay translates both the error code byte and original_type byte to human-readable names.

## Architecture Patterns

### ErrorResponse Wire Format
```
Transport envelope: [type:63][payload:2][request_id:client's_id]
Payload: [error_code:1][original_type:1]
Total payload: 2 bytes
```

### Error Code Enum (for db/peer/ header)
```cpp
// db/peer/error_codes.h (new file)
#pragma once
#include <cstdint>
#include <string_view>

namespace chromatindb::peer {

constexpr uint8_t ERROR_MALFORMED_PAYLOAD  = 0x01;
constexpr uint8_t ERROR_UNKNOWN_TYPE       = 0x02;
constexpr uint8_t ERROR_DECODE_FAILED      = 0x03;
constexpr uint8_t ERROR_VALIDATION_FAILED  = 0x04;
constexpr uint8_t ERROR_INTERNAL           = 0x05;

constexpr std::string_view error_code_string(uint8_t code) {
    switch (code) {
        case ERROR_MALFORMED_PAYLOAD:  return "malformed_payload";
        case ERROR_UNKNOWN_TYPE:       return "unknown_type";
        case ERROR_DECODE_FAILED:      return "decode_failed";
        case ERROR_VALIDATION_FAILED:  return "validation_failed";
        case ERROR_INTERNAL:           return "internal_error";
        default:                       return "unknown";
    }
}

} // namespace chromatindb::peer
```

This follows the exact pattern of `sync_reject.h` (constexpr byte constants + string mapping function).

### Helper Function Pattern
```cpp
// In message_dispatcher.cpp (private helper or lambda)
static asio::awaitable<void> send_error_response(
    net::Connection::Ptr conn,
    uint8_t error_code,
    wire::TransportMsgType original_type,
    uint32_t request_id)
{
    std::array<uint8_t, 2> payload = {
        error_code,
        static_cast<uint8_t>(original_type)
    };
    co_await conn->send_message(
        wire::TransportMsgType_ErrorResponse,
        std::span<const uint8_t>(payload.data(), payload.size()),
        request_id);
}
```

**Critical pattern:** The helper must be a free/static function or lambda, NOT a member coroutine. It is called from within existing coroutine lambdas that already capture `conn` and `request_id`. Making it static avoids SIGHUP stack-use-after-return issues (project key lesson).

### Insertion Pattern (for each silent path)
```cpp
// BEFORE:
if (payload.size() < 64) {
    record_strike_(conn, "ReadRequest too short");
    co_return;
}

// AFTER:
if (payload.size() < 64) {
    record_strike_(conn, "ReadRequest too short");
    co_await send_error_response(conn, ERROR_MALFORMED_PAYLOAD, type, request_id);
    co_return;
}
```

For catch blocks:
```cpp
// BEFORE:
} catch (const std::exception& e) {
    spdlog::warn("malformed ReadRequest from {}: {}", conn->remote_address(), e.what());
    record_strike_(conn, e.what());
}

// AFTER:
} catch (const std::exception& e) {
    spdlog::warn("malformed ReadRequest from {}: {}", conn->remote_address(), e.what());
    record_strike_(conn, e.what());
    co_await send_error_response(conn, ERROR_DECODE_FAILED, type, request_id);
}
```

**Catch block subtlety:** The `co_await` inside a catch block requires the catch to have access to `type` and `request_id`. All coroutine lambdas already capture these by value, so this works. However, `co_await` in a catch block is valid in C++20 coroutines (unlike C++17 where it was forbidden).

### Unrecognized Type Path (end of dispatch)
The unrecognized type handler at line 1248 is NOT inside a coroutine -- it's in the non-coroutine `on_peer_message()`. To send an ErrorResponse, it must co_spawn:
```cpp
// End of dispatch, after all type checks:
asio::co_spawn(ioc_, [this, conn, type, request_id]() -> asio::awaitable<void> {
    co_await send_error_response(conn, ERROR_UNKNOWN_TYPE, type, request_id);
}, asio::detached);
```

### Translator binary_to_json (relay/translate/translator.cpp)
ErrorResponse needs special handling because the flat decode path would produce `{"type":"error","code":1,"original_type":8}` but D-11 requires human-readable names.

Two options:
1. **Custom compound decoder** (recommended): Add a `decode_error_response()` that maps code byte and original_type byte to strings via `error_code_string()` and `type_to_string()`.
2. **Post-process:** Let flat decode run, then post-process to translate numeric fields.

Option 1 is cleaner:
```cpp
static std::optional<nlohmann::json> decode_error_response(
    std::span<const uint8_t> payload)
{
    if (payload.size() < 2) return std::nullopt;
    nlohmann::json j;
    j["type"] = "error";

    uint8_t code = payload[0];
    // Map code byte to name
    static constexpr std::pair<uint8_t, std::string_view> ERROR_CODES[] = {
        {1, "malformed_payload"}, {2, "unknown_type"},
        {3, "decode_failed"}, {4, "validation_failed"},
        {5, "internal_error"}
    };
    std::string_view code_name = "unknown";
    for (const auto& [c, name] : ERROR_CODES) {
        if (c == code) { code_name = name; break; }
    }
    j["code"] = std::string(code_name);

    uint8_t orig_type = payload[1];
    auto type_name = type_to_string(orig_type);
    j["original_type"] = type_name ? std::string(*type_name) : std::to_string(orig_type);

    return j;
}
```

Then in `binary_to_json()`, add before the compound check:
```cpp
// ErrorResponse: custom decode for human-readable names
if (type == 63) {
    return decode_error_response(payload);
}
```

### Request Router Compatibility
The relay's `route_response()` in `uds_multiplexer.cpp` routes responses by `request_id`. ErrorResponse uses the client's original request_id, so routing works without changes. The RequestRouter will match it to the pending request and forward to the correct WebSocket client.

## Common Pitfalls

### Pitfall 1: co_await in Catch Blocks
**What goes wrong:** Some compilers historically rejected `co_await` in catch blocks.
**Why it happens:** C++17 forbidden it, C++20 allowed it, but some older compilers lag.
**How to avoid:** This project uses C++20 coroutines with modern GCC/Clang. The existing codebase does NOT have any co_await in catch blocks currently, so this would be new. Alternative: save the error info, exit the catch, then send the error response after the catch.
**Recommendation:** Use the pattern `catch (...) { record_strike_(); }` followed by a check outside the try/catch. OR simply put `co_await send_error_response()` inside the catch block -- C++20 allows this with GCC 11+ and Clang 14+, both of which this project uses.

### Pitfall 2: Missing request_id in non-coroutine paths
**What goes wrong:** The unrecognized-type handler at the end of `on_peer_message()` is not in a coroutine. Sending an error response requires `co_await`, which means `co_spawn`.
**How to avoid:** Use `asio::co_spawn(ioc_, [conn, type, request_id]() -> asio::awaitable<void> { ... }, asio::detached)` just like the existing QuotaExceeded pattern at line 241.

### Pitfall 3: Overflow Guards Are Internal Errors, Not Client Errors
**What goes wrong:** Treating checked_mul/checked_add overflow as `malformed_payload` when it's actually the node failing to compute a response.
**How to avoid:** Use `internal_error` code for overflow guards. The client's request was valid; the node hit an internal arithmetic overflow (practically impossible, but defensively coded).

### Pitfall 4: Data/Delete Have Complex Error Semantics
**What goes wrong:** Adding ErrorResponse to ALL Data(8) failure paths when some already respond (StorageFull, QuotaExceeded) or are intentionally silent (timestamp_rejected, namespace filtering).
**How to avoid:** Only add ErrorResponse to the two specific paths per handler: (1) catch block, (2) the validation_failed/error path that currently does record_strike_ + no response. Do NOT touch the StorageFull, QuotaExceeded, timestamp_rejected, or namespace filter paths.

### Pitfall 5: send_error_response After Connection Close
**What goes wrong:** If the connection is closing/closed, `send_message()` returns false but doesn't crash. However, attempting to send after a disconnect wastes resources.
**How to avoid:** The existing `send_message()` already handles this gracefully (returns false, no crash). No special guard needed.

### Pitfall 6: FlatBuffers Enum Regeneration
**What goes wrong:** Editing transport.fbs but forgetting to regenerate transport_generated.h, or the regenerated header has a different FlatBuffers version check.
**How to avoid:** Run `flatc --cpp --gen-object-api -o wire/ schemas/transport.fbs` after editing. The project uses FlatBuffers 25.2.10 (per transport_generated.h line 11). Ensure flatc version matches.

### Pitfall 7: Relay ALLOWED arrays Must Stay Sorted
**What goes wrong:** Binary search fails if entries are not in sorted order.
**How to avoid:** ALLOWED_TYPE_STRINGS is sorted alphabetically ("error" goes between "exists_response" and "goodbye"). ALLOWED_WIRE_TYPES is sorted numerically (63 goes at the end).

## Open Questions

1. **co_await in catch blocks**
   - What we know: C++20 spec allows it, GCC 11+ and Clang 14+ support it
   - What's unclear: Whether the project's exact compiler version handles it correctly
   - Recommendation: Test with a simple case first. If it fails, restructure to try/catch without co_await, then send error after the catch using a saved error code.

2. **D-05 Type Numbering Mismatch**
   - What we know: CONTEXT.md D-05 lists "ReadRequest(10)" but the actual wire type for ReadRequest is 31. Similarly StatsRequest is 35 not 14, ExistsRequest is 37 not 24, etc. These appear to be CONTEXT.md typos (using old numbering).
   - Recommendation: Use the actual wire types from transport.fbs. The intent is clear from the type names.

3. **StorageStatusRequest(43) silent failure**
   - What we know: StorageStatusRequest has no payload validation (no size check), only a catch block (line 691). It's not in CONTEXT.md's D-05 explicit list.
   - What's unclear: Whether it should get ErrorResponse on catch.
   - Recommendation: Yes -- it's client-facing and its catch block is a silent failure. Include it.

4. **NamespaceListRequest(41), DelegationListRequest(51) not in D-05**
   - What we know: D-05 doesn't explicitly list these, but they are client-facing request types with silent failure paths.
   - Recommendation: Include them. D-05's list appears to be representative, not exhaustive. The principle "every client request gets a response" applies to all client-facing types.
