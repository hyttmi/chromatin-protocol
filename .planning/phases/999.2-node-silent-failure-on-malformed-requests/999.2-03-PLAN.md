**Test 1: Malformed ReadRequest (payload too short)**

Send a ReadRequest with a namespace but a truncated hash (only 16 bytes instead of 32). The relay will translate this to binary and forward it. The node should reject it with `malformed_payload` because the binary payload will be < 64 bytes.

```cpp
{
    // ReadRequest with truncated hash (16 hex chars instead of 64)
    // This produces a binary payload shorter than the required 64 bytes
    json malformed_read = {
        {"type", "read_request"},
        {"request_id", 900},
        {"namespace", ns_hex},
        {"hash", "00112233445566778899aabbccddeeff"}  // 16 bytes, need 32
    };
    ws_send_text(fd, malformed_read.dump());
    auto frame = ws_recv_frame(fd);
    bool ok = false;
    std::string detail;
    if (frame && frame->text) {
        auto resp = json::parse(*frame->text, nullptr, false);
        if (!resp.is_discarded() && resp.contains("type") && resp["type"] == "error") {
            bool code_ok = resp.contains("code") && resp["code"] == "malformed_payload";
            bool orig_ok = resp.contains("original_type") && resp["original_type"] == "read_request";
            bool rid_ok = resp.contains("request_id") && resp["request_id"] == 900;
            ok = code_ok && orig_ok && rid_ok;
            detail = ok ? "code=malformed_payload, original_type=read_request, request_id=900"
                        : "unexpected response: " + resp.dump();
        } else {
            detail = "expected error response, got: " + (frame->text ? *frame->text : "(binary)");
        }
    } else {
        detail = "no response received (would have hung before Phase 999.2)";
    }
    record("error_response_malformed_read", ok, detail);
}
```

NOTE: If the relay rejects the malformed JSON before forwarding (because the hash hex is too short for HEX_32 encoding), the relay itself will return an error. In that case, the test should check that the relay's client-side validation error is returned instead. Adjust the test logic: if the relay rejects at the JSON->binary stage, it returns `{"error":"..."}` or closes the connection. The test should handle BOTH outcomes (relay rejection OR node ErrorResponse) since either proves the client isn't left hanging.

A more reliable approach is to use the raw WebSocket to send a binary frame that bypasses the relay's JSON validation. However, the relay only accepts text (JSON) frames from clients. So instead, use a valid-looking JSON that the relay CAN translate but that produces a payload the NODE rejects.

**Better approach for reliable node-side ErrorResponse:** Send a request that the relay translates successfully but the node rejects. Examples:
- A `stats_request` with a 31-byte namespace (relay HEX_32 requires exactly 64 hex chars). Actually this would fail at relay too.
- Best approach: send a valid `read_request` with a valid namespace and hash. If the blob doesn't exist, the node returns ReadResponse with status=not_found (not an error). We need something the NODE will reject.

**Most reliable approach:** Send a request type that the node doesn't recognize. But the relay's type_from_string would reject unknown types.

**Actually reliable approach:** The node has validation beyond payload size. For example, `BatchExistsRequest` with count=0 triggers ERROR_VALIDATION_FAILED. The relay will translate this JSON to binary correctly (count is derived from the hashes array length), but if we send an empty hashes array, the relay encodes count=0, and the node rejects with validation_failed.

```cpp
{

        {"type", "batch_exists_request"},

        {"hashes", json::array()}  // empty -> count=0 -> node validation_failed
```

Important: If the relay's json_to_binary rejects any of these before forwarding (returns nullopt), the relay may not send anything or may send its own error. The test should handle this gracefully. If the relay rejects the request (e.g., empty hashes array produces a 0-byte payload that the relay itself drops), record the test as "skipped" rather than failed, with a note explaining why.

Check the relay's json_to_binary for batch requests: the HEX_32_ARRAY encoder writes a u32BE count prefix followed by the hashes. An empty array would produce just a 4-byte `\x00\x00\x00\x00` count prefix. For BatchExistsRequest this gives payload = [namespace:32][count:4 = 0] = 36 bytes. The node expects >= 36 bytes and checks count == 0, so it WILL reach the node and trigger validation_failed. This should work.

For the malformed ReadRequest test (Test 1 with truncated hash): the relay's HEX_32 encoder requires exactly 64 hex chars. A 32-char hex string will cause `from_hex` to return a 16-byte vector, and `bytes->size() != 32` check will fail, causing `encode_field` to return false and `json_to_binary` to return nullopt. The relay will NOT forward this to the node. So remove Test 1 (malformed read) and keep only Tests 2-3 which reliably reach the node.
