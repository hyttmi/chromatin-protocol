## Decisions Made
- **catch_error flag pattern:** GCC rejects co_await in exception handlers (error: "await expressions are not permitted in handlers") despite C++20 spec allowing it. Restructured all 16 catch blocks to set a uint8_t flag inside catch, then co_await send_error_response after the catch block.
- **Static free function:** send_error_response is a static free function (not a member) to avoid stack-use-after-return in coroutine lambdas per project key lessons.
- **Specific enum values:** Used wire::TransportMsgType_ReadRequest etc. instead of adding `type` to each lambda capture list. Cleaner since each handler already knows what type it processes.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] GCC rejects co_await in catch blocks**
- **Found during:** Task 1 (compilation)
- **Issue:** Plan specified co_await send_error_response() directly inside catch blocks. GCC emits "await expressions are not permitted in handlers" -- the C++20 spec allows it but GCC does not implement this yet.
- **Fix:** Restructured all 16 catch blocks to use a catch_error flag pattern: declare `uint8_t catch_error = 0` before try, set the flag inside catch, then `if (catch_error) co_await send_error_response(...)` after the catch block.
- **Files modified:** db/peer/message_dispatcher.cpp
- **Verification:** Build compiles cleanly, all runnable test suites pass
- **Committed in:** 91f1867 (Task 1 commit)
