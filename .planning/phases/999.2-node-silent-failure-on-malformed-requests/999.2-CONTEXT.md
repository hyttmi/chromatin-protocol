# Phase 999.2: Node Silent Failure on Malformed Requests - Context

**Gathered:** 2026-04-11
**Status:** Ready for planning

<domain>
## Phase Boundary

The node currently drops malformed client requests silently — `record_strike_()` + `co_return` with no response. Clients (and the relay) wait forever for a response that never comes. This phase adds an ErrorResponse message type so every client-facing request guarantees a response, even on failure.

</domain>

<decisions>
## Implementation Decisions

### Error Response Format
- **D-01:** New ErrorResponse message type (type 63) added to the wire protocol. Fixed binary format: `[error_code:1][original_type:1]`. Echoes the client's `request_id` in the transport envelope so relay/client can match it to the pending request.
- **D-02:** The relay's type_registry and translator must be updated to handle type 63 (ErrorResponse) — translate to JSON `{"type": "error", "code": "<name>", "original_type": "<name>"}` for WebSocket clients.

### Information Disclosure
- **D-03:** Error code is a single byte from a fixed enum. No detail strings, no payload sizes, no internal state leaked. Codes: `malformed_payload(1)`, `unknown_type(2)`, `decode_failed(3)`, `validation_failed(4)`, `internal_error(5)`. Enough to categorize the problem without revealing protocol expectations.
- **D-04:** The relay translates error code bytes to human-readable JSON names (e.g. code 1 → `"malformed_payload"`). The node wire format stays compact.

### Scope — Which Failures Get Responses
- **D-05:** Only client-facing request types send ErrorResponse on failure: Data(8), Delete(17), ReadRequest(10), ListRequest(12), StatsRequest(14), ExistsRequest(24), BatchReadRequest(26), NodeInfoRequest(28), PeerInfoRequest(30), TimeRangeRequest(57), MetadataRequest(16), Subscribe(19), Unsubscribe(20).
- **D-06:** Peer-to-peer types (sync, PEX, BlobNotify/BlobFetch) stay silent — peers don't wait for responses on these paths and adding responses would complicate the sync protocol.
- **D-07:** Unknown/unrecognized message types also get ErrorResponse with code `unknown_type` — this is client-facing since the relay might forward a type the node doesn't understand.

### Protocol Documentation
- **D-08:** PROTOCOL.md must be updated with the new ErrorResponse type (63), its wire format, and the full error code enum table. This is the authoritative spec — node and relay implementations must match it.
- **D-09:** Error codes must be documented in PROTOCOL.md alongside existing message type documentation.

### Relay Support
- **D-10:** The relay must fully support ErrorResponse: type_registry entry, message_filter allowlist, JSON translation (`binary_to_json` for type 63), and forwarding to the originating WebSocket client via the request_router (using the echoed request_id).
- **D-11:** Relay JSON format for ErrorResponse: `{"type": "error", "code": "<code_name>", "original_type": "<type_name>"}`. The relay translates both the error code byte and original_type byte to human-readable names.

### Claude's Discretion
- Error code enum values (exact byte assignments) — as long as they're documented in PROTOCOL.md
- Whether to add a metrics counter for error responses (recommended)
- Test approach for verifying all silent paths are covered

</decisions>

<code_context>
## Existing Code Insights

### Current Silent Failure Pattern
Every handler follows this pattern on malformed input:
```cpp
if (payload.size() < EXPECTED_SIZE) {
    record_strike_(conn, "RequestType too short");
    co_return;  // <-- CLIENT HANGS HERE
}
```
There are ~12 such paths across ReadRequest, ListRequest, StatsRequest, ExistsRequest, BatchReadRequest, NodeInfoRequest, PeerInfoRequest, TimeRangeRequest, MetadataRequest, Data, Delete, and the catch blocks.

### Reusable Patterns
- `conn->send_message(type, payload, request_id)` — awaitable send, used by all handlers
- `record_strike_()` — should continue to be called (strike tracking stays)
- `SyncRejected` with reason codes — prior art for error code enum pattern

### Integration Points
- `message_dispatcher.cpp` — each handler's catch/validation blocks need ErrorResponse send before `co_return`
- `wire/codec.h` — add `TransportMsgType_ErrorResponse = 63` to the FlatBuffer enum
- Relay type_registry — add entry for type 63
- Relay translator — add `binary_to_json` case for ErrorResponse

</code_context>

<specifics>
## Specific Ideas

- The motivation is debuggability without information leakage — the original silent design was intentional for privacy, but makes development/debugging extremely painful
- Error codes should be categorical, not descriptive — "malformed_payload" not "ReadRequest expected 64 bytes, got 32"
- The relay already has a request_router with pending request tracking and 60s stale cleanup — ErrorResponse will let it clear pending requests immediately instead of waiting for timeout

</specifics>
