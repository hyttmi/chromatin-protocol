## Item 1: SC#7 literal flow, both nodes

**Definition (CONTEXT D-08 #1):** `cdb publish` → `cdb put file` → `cdb get <hash>` → `cdb put --name foo file` → `cdb get foo` → `cdb rm <hash1> <hash2>` (batched) → `cdb ls`.

### Run 1a — local (127.0.0.1) — PASS (with Rule-1 fix)

First-attempt `cdb rm` on local failed with `Error: BOMB submission failed`
(exit 1). Root cause: `submit_bomb_blob` in `cli/src/commands.cpp` was sending
BOMB payloads via `MsgType::Delete=17`, whose node-side handler requires
tombstone format (36 bytes); BOMBs are 8+32N bytes and must ride
`MsgType::BlobWrite=64`. Node log evidence:

```
[warning] delete rejected from 127.0.0.1:42612: delete request data must be
tombstone format (4-byte magic + 32-byte hash)
[warning] strike 1/10 for peer 127.0.0.1:42612 (delete request data must be
tombstone format (4-byte magic + 32-byte hash))
```

Fix applied as GSD Rule-1 deviation (bug) — see commit `5d337da0 fix(124-05):
route BOMB submission via BlobWrite + surface ErrorResponse`. Rebuild of local
`cdb` succeeded; home `cdb` = same binary (workstation CLI only, not home
node). Item 1 then re-ran cleanly on local.

Rule-2 deviation also applied in the same commit: `ls --type BOMB` and
`ls --type NAME` were unrecognised filter names; both are post-Phase-123 types
that the node's type_filter already honours — added to the CLI filter
switch so the E2E matrix could enumerate them.

**Finding:** Both NAME blobs landed in the same second (`10:31:11`).

the tombstoned v1 content → `get` returns "blob not found".

This is NOT a Phase 124 CLI defect; it's a latent Phase 123 D-15
hazard: `cdb put --name --replace` emits NEW NAME with
`now = time(nullptr)`, not `max(old_ts + 1, now)`. When two
`cdb put --name` invocations run within a single second (routine in
scripted contexts), the replacement can lose the tiebreak.

**Flagged for Phase 125+ follow-up:** consider either
(a) bumping NAME timestamp to `max(seen_latest_ts + 1, now)` in the
`--replace` path, or (b) tiebreaking on `(timestamp DESC, target_hash
DESC)` so the RESOLUTION deterministically points at the live content,
or (c) documenting the second-granularity contract.
