## Item 6: `--replace` BOMB-of-1

### Run 6a — local, same-second tiebreak hazard found

First attempt with no sleep between `put --name` and `put --name --replace`:

```
$ cdb --node local put --name replace-test /tmp/124-e2e/item6/v1.bin
named: replace-test -> bc4efce9dcc590b6271353dcdabe47ed36a1bab1a9e200bdcdcbeb99d1e3e24f
(exit 0)

$ cdb --node local put --name replace-test --replace /tmp/124-e2e/item6/v2.bin
named: replace-test -> 000829ad16e1bf941407a2529bb937f597b5665d1390e1bc3c85eb0094ea9d1e
replaced: bc4efce...e24f tombstoned
(exit 0)

$ cdb --node local get replace-test -o /tmp/124-e2e/item6/recv
Error: blob not found: bc4efce9dcc590b6271353dcdabe47ed36a1bab1a9e200bdcdcbeb99d1e3e24f
exit=1
```

**Finding:** Both NAME blobs landed in the same second (`10:31:11`).
The tiebreak in `pick_name_winner` is `blob_hash DESC`, and the older
NAME blob's hash happened to be larger → winning tiebreak → pointing to
the tombstoned v1 content → `get` returns "blob not found".

This is NOT a Phase 124 CLI defect; it's a latent Phase 123 D-15
hazard: `cdb put --name --replace` emits NEW NAME with
`now = time(nullptr)`, not `max(old_ts + 1, now)`. When two
`cdb put --name` invocations run within a single second (routine in
scripted contexts), the replacement can lose the tiebreak.

**Flagged for Phase 125+ follow-up:** consider either
(a) bumping NAME timestamp to `max(seen_latest_ts + 1, now)` in the
`--replace` path, or (b) tiebreaking on `(timestamp DESC, target_hash
DESC)` so the RESOLUTION deterministically points at the live content,
or (c) documenting the second-granularity contract.
