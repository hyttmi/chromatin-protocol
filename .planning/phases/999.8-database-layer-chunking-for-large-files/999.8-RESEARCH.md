# Phase 999.8: Database Layer Chunking for Large Files - Research

**Researched:** 2026-04-11
**Domain:** MDBX blob storage, chunking conventions, engine-layer API design
**Confidence:** HIGH

## Summary

This phase adds chunking as a convention within the chromatindb engine layer. Large data is split into 1 MiB chunks (regular signed blobs), with a manifest blob (magic-prefixed data) listing chunk hashes in order. All chunks and the manifest live in the same namespace, use the same TTL/timestamp, and are stored atomically in a single MDBX write transaction. No new wire types, no new MDBX sub-databases, no relay changes.

The codebase already has a well-established pattern for magic-prefixed data blobs: tombstones (0xDE 0xAD 0xBE 0xEF), delegations (0xDE 0x1E 0x6A 0x7E), envelopes (0x43 0x45 0x4E 0x56 "CENV"), and user entries (0x55 0x45 0x4E 0x54 "UENT"). The manifest format (0x43 0x48 0x4E 0x4B "CHNK") fits this pattern exactly. The engine already handles encode/hash/sign/store for blobs -- chunking adds orchestration around existing primitives.

**Primary recommendation:** Add `store_chunked()` and `read_chunked()` to BlobEngine alongside existing `ingest()` and `get_blob()`. The critical design challenge is atomic multi-blob storage -- the current `store_blob()` opens its own write transaction per call, so a new storage-level method is needed that accepts multiple blobs in a single transaction.

### Pattern 1: Magic-Prefixed Data Convention

**What:** The codebase identifies special blob types by a 4-byte magic prefix in the `data` field. Chunks and manifests follow this established pattern.

**When to use:** Any time a blob's data field carries structured metadata rather than opaque user data.

**Existing precedents:**
```cpp
// db/wire/codec.h -- tombstone pattern
inline constexpr std::array<uint8_t, 4> TOMBSTONE_MAGIC = {0xDE, 0xAD, 0xBE, 0xEF};
inline constexpr size_t TOMBSTONE_DATA_SIZE = 36;  // 4 + 32
bool is_tombstone(std::span<const uint8_t> data);

// db/wire/codec.h -- delegation pattern
inline constexpr std::array<uint8_t, 4> DELEGATION_MAGIC = {0xDE, 0x1E, 0x6A, 0x7E};
inline constexpr size_t DELEGATION_DATA_SIZE = 4 + 2592;
bool is_delegation(std::span<const uint8_t> data);
```

**New manifest pattern (in chunking.h):**
```cpp
// db/engine/chunking.h
namespace chromatindb::engine {

inline constexpr std::array<uint8_t, 4> MANIFEST_MAGIC = {0x43, 0x48, 0x4E, 0x4B};  // "CHNK"
inline constexpr size_t CHUNK_SIZE = 1048576;  // 1 MiB

/// Check if blob data is a chunking manifest (magic prefix check).
bool is_manifest(std::span<const uint8_t> data);

/// Parse manifest data into ordered list of chunk hashes.
/// Returns nullopt if format is invalid (bad magic, wrong size, etc.).
std::optional<std::vector<std::array<uint8_t, 32>>> parse_manifest(
    std::span<const uint8_t> data);

/// Build manifest data: [CHNK magic:4][chunk_count:4 BE][hash_1:32][hash_2:32]...
std::vector<uint8_t> make_manifest_data(
    const std::vector<std::array<uint8_t, 32>>& chunk_hashes);

} // namespace chromatindb::engine
```

### Pattern 2: Atomic Multi-Blob Storage

**What:** The current `store_blob()` opens and commits its own MDBX write transaction. For atomic chunking, we need a new method that stores N blobs (chunks + manifest) in a single transaction.

**Why this matters:** If we call `store_blob()` in a loop, a crash after storing some chunks but before the manifest leaves orphaned chunks. MDBX's ACID transactions solve this -- all-or-nothing.

**Design: new Storage method:**
```cpp
// db/storage/storage.h
struct PrecomputedBlob {
    wire::BlobData blob;
    std::array<uint8_t, 32> content_hash;
    std::vector<uint8_t> encoded;
};

/// Store multiple blobs atomically in a single write transaction.
/// Used by store_chunked() -- all chunks + manifest stored together.
/// Checks dedup, capacity, and quota the same as store_blob().
/// Returns vector of StoreResults (one per blob, same order).
std::vector<StoreResult> store_blobs_atomic(
    const std::vector<PrecomputedBlob>& blobs,
    uint64_t max_storage_bytes = 0,
    uint64_t quota_byte_limit = 0,
    uint64_t quota_count_limit = 0);
```

**Implementation strategy:** The internals of `store_blob()` (lines 381-542 in storage.cpp) open a write txn, check dedup, check capacity/quota, encrypt, store, assign seq_num, update quota, commit. The new `store_blobs_atomic()` does the same loop but within ONE txn. Key: sum all encoded sizes for capacity/quota checks upfront.

### Pattern 3: Signature Callback for Engine-Level Signing

**What:** The current engine verifies signatures but doesn't sign. `store_chunked()` needs to sign each chunk. Since the engine doesn't hold keys, a callback is needed.

**Design:**
```cpp
// Signature callback type
using SignFn = std::function<std::vector<uint8_t>(std::span<const uint8_t> message)>;

/// Split data into chunks, sign each, create manifest, store atomically.
/// signature_fn is called once per chunk + once for the manifest.
asio::awaitable<IngestResult> store_chunked(
    std::span<const uint8_t, 32> namespace_id,
    std::span<const uint8_t> pubkey,
    std::span<const uint8_t> data,
    uint32_t ttl,
    uint64_t timestamp,
    SignFn signature_fn);
```

**Why callback, not key material:** The engine is a validation/storage layer, not a key owner. The caller (message dispatcher or internal handler) owns the identity and provides the signing function. This matches the existing separation of concerns where `NodeIdentity::sign()` is called at the caller level, not deep in the engine.

### Pattern 4: Read-Side Manifest Resolution

**What:** `read_chunked()` reads the manifest blob, parses it, fetches each chunk, and concatenates.

**Design:**
```cpp
/// Result of a chunked read operation.
struct ChunkedReadResult {
    bool success = false;
    std::vector<uint8_t> data;              // Reassembled data
    std::string error;                      // Error message if !success
    uint32_t chunks_found = 0;              // How many chunks were available
    uint32_t chunks_expected = 0;           // From manifest
};

/// Read a chunked file by manifest hash.
ChunkedReadResult read_chunked(
    std::span<const uint8_t, 32> namespace_id,
    std::span<const uint8_t, 32> manifest_hash);
```

### Discretion Recommendations

**1. Expose as transparent handling (NOT new wire type):**
Recommendation: Handle transparently in the Data(8) handler. When the engine's Data handler receives data > CHUNK_SIZE, it internally calls `store_chunked()`. The client doesn't need to know about chunking -- it sends a large blob, gets a WriteAck with the manifest hash. Read is similar: if a ReadResponse returns manifest-magic data, the client can either use it raw or request reassembly. This keeps the wire protocol unchanged and avoids relay modifications.

However, there is a subtlety: the node's current ingest path rejects blobs > MAX_BLOB_DATA_SIZE (100 MiB). For the transparent approach, the handler would need to intercept before the size check and route to `store_chunked()`. This works but requires careful ordering in the dispatch path.

**2. Add is_manifest() helper: YES**
It costs nothing and follows the existing is_tombstone()/is_delegation() pattern. Useful for read_chunked() dispatch and for any future code that needs to distinguish manifest blobs.

**3. read_chunked() should FAIL on missing chunks:**
Returning partial data is dangerous -- a 10 MiB file with 2 missing chunks in the middle would produce corrupt output. Better to fail clearly with the count of missing chunks so the caller can request BlobFetch for the gaps. The `ChunkedReadResult` struct above carries both `chunks_found` and `chunks_expected` for diagnostic purposes.

**4. Error handling for corrupt/incomplete manifests:**
- Magic mismatch: return error "not a manifest blob"
- Size mismatch (data size != 8 + chunk_count * 32): return error "corrupt manifest"
- chunk_count == 0: return error "empty manifest"
- Manifest data > MAX_BLOB_DATA_SIZE: impossible (already rejected at ingest)
- Maximum chunk_count: (MAX_BLOB_DATA_SIZE - 8) / 32 = 3,276,792 chunks = ~3.1 TiB. This exceeds practical limits. Consider a reasonable cap (e.g., 100,000 chunks = ~97.6 GiB) to prevent accidental memory exhaustion on read.

### Anti-Patterns to Avoid

- **Loop of store_blob() calls for atomicity:** Each store_blob opens its own txn. Crash between stores leaves orphans. Must use single-txn atomic store.
- **Engine holding signing keys:** The engine validates, it doesn't sign. Pass a callback.
- **New MDBX sub-databases for chunk metadata:** The entire point of the convention model is that chunks are regular blobs. No schema changes.
- **Changing the wire protocol for this:** Chunks and manifests travel as Data(8). No ChunkedData type. The relay doesn't need to know.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Signing input | Custom hash logic | `wire::build_signing_input()` | Already handles namespace+data+ttl+timestamp canonical form |
| FlatBuffer encoding | Manual serialization | `wire::encode_blob()` | Deterministic encoding with ForceDefaults |
| Content hashing | Custom hash | `wire::blob_hash()` | SHA3-256 of encoded blob, used for content addressing |
| BE encoding | Bit shifts | `chromatindb::util::store_u32_be()` | Consistent with endian.h utilities |
| MDBX transactions | Manual open/close | Existing `impl_->env.start_write()` pattern | RAII cleanup on exception |

## Common Pitfalls

### Pitfall 1: Non-Atomic Multi-Blob Store
**What goes wrong:** Storing chunks via repeated `store_blob()` calls means each chunk is its own MDBX write transaction. A crash after chunk 5 of 10 leaves orphaned chunks with no manifest.
**Why it happens:** The existing API assumes one-blob-per-call.
**How to avoid:** Implement `store_blobs_atomic()` in Storage that opens ONE write transaction for all blobs. The engine's `store_chunked()` calls this.
**Warning signs:** Tests pass individually but data integrity issues under crash/restart scenarios.

### Pitfall 2: Quota Accounting for Chunked Writes
**What goes wrong:** Checking quota once for the total data size, then storing N+1 blobs (N chunks + 1 manifest), each with its own FlatBuffer+encryption overhead.
**Why it happens:** Encoded blob size > raw data size due to FlatBuffer envelope, pubkey, signature, and DARE encryption overhead.
**How to avoid:** Pre-compute all encoded blobs before the transaction, sum their actual encrypted sizes, check quota once with the real total. The existing quota check in `store_blob()` uses `precomputed_encoded.size()` -- do the same for the sum.
**Warning signs:** Quota slightly exceeded after chunked writes because overhead wasn't accounted for.

### Pitfall 3: Signature Cost Amplification
**What goes wrong:** ML-DSA-87 signing is expensive. A 100 MiB file = 100 chunks + 1 manifest = 101 ML-DSA-87 sign operations.
**Why it happens:** Each chunk must be independently verifiable (D-06).
**How to avoid:** Offload signing to the thread pool (same as existing `crypto::offload()` pattern). Consider batching: sign all chunks in parallel on the thread pool rather than sequentially. The engine already uses `asio::thread_pool` for crypto offload.
**Warning signs:** store_chunked() taking orders of magnitude longer than expected due to sequential signing.

### Pitfall 4: Manifest Size Limit
**What goes wrong:** A manifest for a very large file (e.g., 50 GiB = 50,000 chunks) has data size 8 + 50000*32 = 1,600,008 bytes (1.5 MiB). This itself exceeds CHUNK_SIZE but is well under MAX_BLOB_DATA_SIZE.
**Why it happens:** Manifest is a regular blob -- it needs to fit in MAX_BLOB_DATA_SIZE.
**How to avoid:** Maximum theoretical: (100 MiB - 8) / 32 = 3.2M chunks = 3.1 TiB. In practice, set a reasonable cap (e.g., MAX_CHUNK_COUNT = 100000, supporting ~97.6 GiB files). This prevents read_chunked from allocating unbounded memory.
**Warning signs:** OOM on read_chunked for a corrupted manifest with huge chunk_count.

### Pitfall 5: Chunk Hash Ordering in Manifest
**What goes wrong:** Chunks are stored and later retrieved independently. If chunk hashes in the manifest are wrong or reordered, read_chunked reassembles garbage.
**Why it happens:** Hash computation timing -- must hash after encode, and manifest must list hashes in order.
**How to avoid:** store_chunked() flow: (1) split data into chunk payloads, (2) build BlobData for each chunk, (3) encode each with encode_blob(), (4) compute blob_hash() for each, (5) build manifest with hashes in order, (6) encode manifest, (7) store all atomically. The order is enforced by vector index.

### Pitfall 6: TTL Expiry of Chunks vs Manifest
**What goes wrong:** If chunks and manifest have the same TTL/timestamp (D-05), they should expire at the same time. But MDBX expiry scan might delete some chunks before the manifest (or vice versa) within the same scan cycle due to iteration order.
**Why it happens:** Expiry keys are [expiry_ts:8][hash:32], and hash varies per chunk.
**How to avoid:** This is actually fine -- if some chunks expire, read_chunked fails with "missing chunks." The manifest also expires, so the whole logical file becomes unreadable. This is correct behavior. No special handling needed.

### Pitfall 7: store_chunked Bypasses Engine Validation
**What goes wrong:** The current `ingest()` does expensive validation (namespace ownership, signature verify). If store_chunked builds and signs blobs internally, it must either (a) skip validation (trusted path) or (b) run each chunk through the full ingest pipeline.
**Why it happens:** Tension between "chunks are regular blobs" and "we're building them ourselves."
**How to avoid:** Since store_chunked is called by the node on behalf of an authenticated client, and the node is signing with a validated identity, skip the ingest validation pipeline. Store directly via the new atomic storage method. The blobs are self-consistent by construction (we built them). This is analogous to how the node internally stores BlobNotify-received blobs after verification.

## Code Examples

### Manifest Format Construction
```cpp
// Source: D-03 from CONTEXT.md, following endian.h patterns
std::vector<uint8_t> make_manifest_data(
    const std::vector<std::array<uint8_t, 32>>& chunk_hashes) {
    // [CHNK:4][chunk_count:4 BE][hash_1:32][hash_2:32]...
    std::vector<uint8_t> data;
    data.reserve(8 + chunk_hashes.size() * 32);
    
    // Magic prefix
    data.insert(data.end(), MANIFEST_MAGIC.begin(), MANIFEST_MAGIC.end());
    
    // Chunk count (BE u32)
    chromatindb::util::write_u32_be(data, static_cast<uint32_t>(chunk_hashes.size()));
    
    // Chunk hashes in order
    for (const auto& hash : chunk_hashes) {
        data.insert(data.end(), hash.begin(), hash.end());
    }
    
    return data;
}
```

### Manifest Parsing
```cpp
std::optional<std::vector<std::array<uint8_t, 32>>> parse_manifest(
    std::span<const uint8_t> data) {
    // Minimum: 4 (magic) + 4 (count) = 8 bytes
    if (data.size() < 8) return std::nullopt;
    
    // Check magic
    if (!std::equal(data.begin(), data.begin() + 4,
                    MANIFEST_MAGIC.begin())) {
        return std::nullopt;
    }
    
    uint32_t count = chromatindb::util::read_u32_be(data.subspan(4, 4));
    
    // Validate size: 8 + count * 32
    if (data.size() != 8 + static_cast<size_t>(count) * 32) {
        return std::nullopt;
    }
    
    // Reasonable cap to prevent memory exhaustion
    if (count > 100000) return std::nullopt;
    
    std::vector<std::array<uint8_t, 32>> hashes;
    hashes.reserve(count);
    for (uint32_t i = 0; i < count; ++i) {
        std::array<uint8_t, 32> hash;
        std::memcpy(hash.data(), data.data() + 8 + i * 32, 32);
        hashes.push_back(hash);
    }
    
    return hashes;
}
```

### store_chunked Flow (Engine Level)
```cpp
asio::awaitable<IngestResult> BlobEngine::store_chunked(
    std::span<const uint8_t, 32> namespace_id,
    std::span<const uint8_t> pubkey,
    std::span<const uint8_t> data,
    uint32_t ttl,
    uint64_t timestamp,
    SignFn signature_fn)
{
    // 1. Split data into chunk payloads
    std::vector<std::vector<uint8_t>> chunk_payloads;
    for (size_t offset = 0; offset < data.size(); offset += CHUNK_SIZE) {
        size_t len = std::min(CHUNK_SIZE, data.size() - offset);
        chunk_payloads.emplace_back(data.begin() + offset, data.begin() + offset + len);
    }
    
    // 2. Build BlobData for each chunk, sign, encode, hash
    std::vector<storage::PrecomputedBlob> precomputed;
    std::vector<std::array<uint8_t, 32>> chunk_hashes;
    
    for (auto& payload : chunk_payloads) {
        wire::BlobData chunk;
        std::memcpy(chunk.namespace_id.data(), namespace_id.data(), 32);
        chunk.pubkey.assign(pubkey.begin(), pubkey.end());
        chunk.data = std::move(payload);
        chunk.ttl = ttl;
        chunk.timestamp = timestamp;
        
        auto signing_input = wire::build_signing_input(
            chunk.namespace_id, chunk.data, chunk.ttl, chunk.timestamp);
        chunk.signature = co_await crypto::offload(pool_, [&]() {
            return signature_fn(signing_input);
        });
        
        auto encoded = wire::encode_blob(chunk);
        auto hash = wire::blob_hash(encoded);
        chunk_hashes.push_back(hash);
        
        precomputed.push_back({std::move(chunk), hash, std::move(encoded)});
    }
    
    // 3. Build manifest
    auto manifest_data = make_manifest_data(chunk_hashes);
    wire::BlobData manifest;
    std::memcpy(manifest.namespace_id.data(), namespace_id.data(), 32);
    manifest.pubkey.assign(pubkey.begin(), pubkey.end());
    manifest.data = std::move(manifest_data);
    manifest.ttl = ttl;
    manifest.timestamp = timestamp;
    
    auto manifest_si = wire::build_signing_input(
        manifest.namespace_id, manifest.data, manifest.ttl, manifest.timestamp);
    manifest.signature = co_await crypto::offload(pool_, [&]() {
        return signature_fn(manifest_si);
    });
    
    auto manifest_encoded = wire::encode_blob(manifest);
    auto manifest_hash = wire::blob_hash(manifest_encoded);
    precomputed.push_back({std::move(manifest), manifest_hash, std::move(manifest_encoded)});
    
    // 4. Atomic store (all chunks + manifest in single txn)
    auto [byte_limit, count_limit] = effective_quota(namespace_id);
    auto results = storage_.store_blobs_atomic(
        precomputed, max_storage_bytes_, byte_limit, count_limit);
    
    // 5. Return manifest hash as the handle
    // Check last result (manifest) for success
    // ...build IngestResult with manifest_hash...
}
```

## Key Technical Findings

### MDBX Transaction Model (HIGH confidence)
- Source: storage.cpp direct code inspection
- Each `store_blob()` call opens its own `impl_->env.start_write()` transaction
- MDBX supports a single active write transaction at a time (serialized)
- A single write transaction can do multiple upserts across multiple sub-databases
- `store_blobs_atomic()` is achievable by looping the inner store logic within one txn
- Transaction abort on any failure rolls back ALL changes (ACID)

### Encryption at Rest Impact (HIGH confidence)
- Source: storage.cpp lines 232-268
- Every blob value is encrypted before MDBX storage: [version:1][nonce:12][ciphertext+tag]
- 29 bytes overhead per blob (ENVELOPE_OVERHEAD constant)
- `encrypt_value()` is a private Impl method -- `store_blobs_atomic()` must be a Storage member (not free function) to access it
- The AD (associated data) for AEAD is the 64-byte MDBX key (namespace:32 + hash:32)

### Sequence Number Assignment (HIGH confidence)
- Source: storage.cpp line 476
- `impl_->next_seq_num(txn, namespace)` assigns monotonically increasing per-namespace seq_nums
- Within a single transaction, calling this N+1 times gives N+1 consecutive seq_nums
- Chunks will get sequential seq_nums, which is fine for replication ordering

### Quota Aggregation (HIGH confidence)
- Source: storage.cpp lines 510-525
- Quota is incremented per-blob within the write transaction
- For N+1 blobs in one transaction, the quota update must account for all of them
- Current code reads quota once, increments by encrypted size, writes back
- For atomic store: read once, accumulate total, write once at end

### Existing Magic Prefix Registry (HIGH confidence)
- Source: codec.h, PROTOCOL.md
- 0xDE 0xAD 0xBE 0xEF -- tombstone
- 0xDE 0x1E 0x6A 0x7E -- delegation
- 0x43 0x45 0x4E 0x56 -- envelope ("CENV")
- 0x55 0x45 0x4E 0x54 -- user entry ("UENT")
- 0x43 0x48 0x4E 0x4B -- manifest ("CHNK") [NEW -- no collision]

### BlobEngine Thread Safety (HIGH confidence)
- Source: engine.h comment "Thread safety: NOT thread-safe"
- BlobEngine is single-threaded (event loop)
- store_chunked() is safe to call from the event loop
- Crypto operations offloaded to thread_pool via co_await crypto::offload()

### Where Chunking Code Should Live (HIGH confidence)
- Source: project structure analysis
- Manifest format utilities (is_manifest, parse_manifest, make_manifest_data) belong in `db/engine/chunking.h/cpp` -- they are engine-layer conventions, not wire-format primitives like tombstone/delegation in codec.h
- The engine owns the chunking semantics; storage just stores blobs
- New test file: `db/tests/engine/test_chunking.cpp`

## Open Questions

1. **Transparent chunking at Data handler level vs explicit API only**
   - What we know: D-15 says Claude's discretion. The handler could intercept large Data(8) messages and route to store_chunked automatically.
   - What's unclear: Does this require the Data handler to have access to the signing key? The handler receives already-signed blobs. For transparent chunking, the handler would need to RE-sign chunks.
   - Recommendation: Start with explicit API only (store_chunked/read_chunked as engine methods). Transparent handler integration requires the UDS client to send unsigned data + auth proof, which is a protocol change. Instead, make store_chunked available for internal callers (e.g., UDS tool, future CLI) and document it. The relay/wire protocol can be extended later if needed.

2. **Parallel chunk signing for performance**
   - What we know: 100 MiB = 100 chunks = 100 ML-DSA-87 sign operations. Each takes ~ms on modern hardware.
   - What's unclear: Whether to sign sequentially (simpler) or batch-parallel (faster for large files).
   - Recommendation: Start sequential. ML-DSA-87 signing is ~1-5ms per call. Even 100 sequential signs = ~500ms. Optimize to parallel only if profiling shows this is a bottleneck. The thread pool exists and could be used, but adds complexity to the flow.

3. **Chunk count cap**
   - What we know: Theoretical max is ~3.2M chunks (100 MiB manifest). Practical files are orders of magnitude smaller.
   - Recommendation: Cap at 100,000 chunks (~97.6 GiB). This prevents memory exhaustion on read_chunked while being far above any practical use case. Return error for manifests exceeding this cap.
