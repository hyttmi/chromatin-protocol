# Phase 999.8: Database Layer Chunking for Large Files - Context

**Gathered:** 2026-04-12
**Status:** Ready for planning

<domain>
## Phase Boundary

Large files may exceed practical limits for single-blob operations (current max 100 MiB). Add chunking as a convention at the node layer: split large data into fixed-size chunk blobs, create a manifest blob that lists chunk hashes in order, reassemble on read. Chunks and manifests are regular signed blobs — no new wire types, no new storage tables, no relay changes.

</domain>

<decisions>
## Implementation Decisions

### Chunk Size
- **D-01:** Fixed 1 MiB (1,048,576 bytes) chunk size. Simple, matches benchmark sweet spot (~33 blobs/sec at 1 MiB). Not configurable per-blob — one global constant.
- **D-02:** Last chunk may be smaller than 1 MiB (remainder).

### Manifest Model
- **D-03:** A manifest is a regular signed blob whose `data` field contains a magic prefix + ordered list of chunk hashes. Format: `[MANIFEST_MAGIC:4][chunk_count:4 BE][chunk_hash_1:32][chunk_hash_2:32]...`. Magic bytes: `0x43 0x48 0x4E 0x4B` ("CHNK").
- **D-04:** Manifest blob lives in the same namespace as its chunks. The manifest's own blob_hash is the handle clients use to reference the chunked file.
- **D-05:** Manifest's TTL and timestamp apply to the logical file. Chunks inherit the same TTL and timestamp from the writer.

### Signature Strategy
- **D-06:** Each chunk is a regular signed blob — independently verifiable with standard `build_signing_input()`. No new crypto primitives.
- **D-07:** The manifest blob is also a regular signed blob. Its data field (the chunk list) is signed like any other blob data.
- **D-08:** All chunks and the manifest must be signed by the same key (same namespace). Cross-namespace chunking is out of scope.

### Storage Model
- **D-09:** Chunks are stored as regular blobs in MDBX. No new tables, no new indexes. The node has no concept of "chunk" vs "blob" — it's all blobs.
- **D-10:** The manifest is also a regular blob. The chunking convention is understood by the node's engine layer for write/read API convenience, but storage is unchanged.

### Node API
- **D-11:** New engine methods: `store_chunked(namespace, data, ttl, timestamp, signature_fn)` splits data into chunks, signs each, creates manifest, stores all atomically. Returns manifest blob_hash.
- **D-12:** New engine method: `read_chunked(namespace, manifest_hash)` reads manifest, fetches all chunks in order, concatenates data. Returns reassembled data or error if any chunk is missing.
- **D-13:** Regular `store_blob()` and `get_blob()` remain unchanged — small blobs don't need chunking.

### Wire Protocol
- **D-14:** No new message types. Chunks and manifests are sent as regular Data(8) messages. The relay sees them as normal blobs.
- **D-15:** The node's `store_chunked` is called internally when receiving a large Data request, or exposed as a new request type (Claude's discretion on which approach is simpler).

### Replication
- **D-16:** Chunks replicate as normal blobs via existing sync protocol and BlobNotify/BlobFetch. No special ordering — chunks can arrive in any order.
- **D-17:** Manifest replicates like any blob. A reader that receives the manifest before all chunks can fetch missing chunks on demand via BlobFetch.

### Pre-MVP
- **D-18:** No backward compat. No migration. Clean implementation.

### Claude's Discretion
- Whether to expose chunking as a new wire request type (e.g., ChunkedData) or handle it transparently when Data exceeds chunk size
- Whether to add a `is_manifest()` helper to detect manifest blobs by magic prefix
- Whether read_chunked should fail or return partial data when some chunks are missing
- Exact error handling for corrupt/incomplete manifests

</decisions>

<code_context>
## Existing Code Insights

### Current Limits
- MAX_BLOB_DATA_SIZE = 100 MiB (db/net/framing.h)
- Engine rejects blobs > MAX_BLOB_DATA_SIZE at ingest
- blob_size field is uint32 in BlobNotify (4 GB theoretical max)

### Reusable Assets
- `build_signing_input()` — signs chunks exactly like regular blobs
- `encode_blob()` / `decode_blob()` — FlatBuffer encode/decode works for chunks
- `store_blob()` with atomic write txn — can be called in a loop for each chunk + manifest

### Integration Points
- `engine.cpp` — Add store_chunked() / read_chunked() alongside existing store_blob() / get_blob()
- No relay changes — relay translates Data(8) for all blobs regardless
- No sync changes — chunks and manifests replicate as regular blobs

</code_context>

<specifics>
## Specific Ideas

- Chunking is a convention, not a protocol change — keeps the system simple
- 1 MiB chunks at 33 blobs/sec means ~33 MiB/sec chunked throughput
- Manifest format is self-describing (magic prefix) — future-proofs detection
- Atomic store: all chunks + manifest stored in a single MDBX write transaction

</specifics>

<deferred>
## Deferred Ideas

- Streaming chunk upload (store chunks as they arrive, not all at once) — separate phase if needed
- Cross-namespace chunking — would require multi-key signing, too complex for now
- Chunk deduplication across files — content-defined chunking (CDC) is a separate feature

</deferred>
