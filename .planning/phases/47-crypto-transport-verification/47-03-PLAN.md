Key transport protocol facts:
- PQ handshake: initiator sends ClientHello with ephemeral ML-KEM-1024 encapsulation key. Responder encapsulates, sends ServerHello with ciphertext. Both derive session key from KEM shared secret. Both sign session fingerprint with ML-DSA-87 identity key.
- Session fingerprint = hash of (KEM shared secret + both identity pubkeys + nonces). MITM substituting KEM keys produces different shared secrets on each side -> fingerprint mismatch.
- Trusted peers: initiator sends TrustedHello (no KEM). Responder checks IP against trusted_peers. If trusted, replies with signed identity proof. If NOT trusted, replies PQRequired (forcing full PQ handshake). Wrong identity key = rejected even if IP is trusted.
- All handshake events logged at debug level (TrustedHello, PQRequired, fingerprint verification, handshake success/failure).
- Forward secrecy: even with long-term identity keys compromised, past session traffic is undecryptable because session key derived from ephemeral KEM shared secret.

Docker networking:
- Bridge network allows tcpdump capture
- NET_ADMIN capability needed for tcpdump in containers
- Container names resolve on bridge network (node1, node2, node3)

Let me re-read the handshake protocol more carefully.

The handshake works like this:
1. Initiator generates ephemeral KEM keypair, sends public key + identity signature
2. Responder encapsulates with the KEM public key, sends ciphertext + identity signature
3. Both derive session key from KEM shared secret
4. Both verify each other's identity signature over the session fingerprint (which includes KEM shared secret)

A MITM would need to:
1. Intercept initiator's KEM public key, replace with MITM's KEM public key
2. Forward modified message to responder
3. Intercept responder's ciphertext (encapsulated to MITM's key), decapsulate, re-encapsulate to initiator's original key
4. The result: MITM has the shared secret, but when identity signatures are verified, the session fingerprint (which includes the KEM shared secret) differs on each side because initiator derived secret from MITM's re-encapsulation, not responder's original.
