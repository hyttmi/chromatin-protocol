Key crypto facts:
- blob_hash = SHA3-256(full FlatBuffer encoded bytes) -- content addressing
- signing_digest = SHA3-256(namespace_id || data || ttl_le32 || timestamp_le64) -- what gets signed
- ML-DSA-87 signature signs the signing_digest (32 bytes), NOT the raw data
- DARE: HKDF-derived key with context "chromatindb-dare-v1", envelope [version][nonce][ciphertext+tag]
- Blobs stored encrypted via ChaCha20-Poly1305; bit flip causes AEAD auth failure on decryption

**loadgen/loadgen_main.cpp** -- Add `--verbose-blobs` flag. When set, for each blob sent, write a line to stderr with the blob's raw fields:
```
BLOB_FIELDS:{"index":0,"namespace_id":"<hex>","data_hex":"<hex if <=4096 bytes else omitted>","data_sha3":"<hex>","ttl":3600,"timestamp":1234567890123,"pubkey_hex":"<hex>","signature_hex":"<hex>","blob_hash":"<hex>","signing_digest":"<hex>"}
```
For small blobs (<=4096 bytes), include data_hex. For larger blobs, omit data_hex but include data_sha3. This is ~20 lines of code in the blob-send loop.

**tools/verify_main.cpp** -- Add two new subcommands (in addition to existing `hash` and `sig` from Plan 01):
- `hash-fields --namespace-hex HEX --data-hex HEX --ttl N --timestamp N` -- computes SHA3-256(namespace||data||ttl_le32||timestamp_le64) and outputs `{"signing_digest":"<hex>"}`. This allows fully independent recomputation from raw fields without needing FlatBuffer encoding.
- `sig-fields --digest-hex HEX --signature-hex HEX --pubkey-hex HEX` -- calls Signer::verify(digest, signature, pubkey), outputs `{"valid":true/false}`, exit 0 if valid, 1 if invalid.

**test_crypt03_tamper_detection.sh** -- Verifies CRYPT-03: single bit flip in data.mdb causes AEAD failure and corrupted blob is never served.

Test flow:
1. Source helpers.sh, check_deps, build_image, cleanup, start_nodes (node1 only -- single node test)
2. Run loadgen to ingest 5 blobs to node1: `--count 5 --size 1024 --ttl 3600`
3. Verify node1 has 5 blobs (SIGUSR1 metrics)
4. Stop node1: `$COMPOSE stop node1`
5. Flip a single bit in data.mdb:
   - Find the data.mdb file in the container's volume: `docker run --rm -v node1-data:/data debian:bookworm-slim ls -la /data/data.mdb`
   - Use xxd + dd to flip a bit past the mdbx header (first 3 pages, page size 4096). Use offset 12288+ to target data pages containing high-entropy DARE envelopes:
   ```bash
   docker run --rm -v node1-data:/data debian:bookworm-slim sh -c '
     OFFSET=12288
     ORIG=$(xxd -s $OFFSET -l 1 -p /data/data.mdb)
     FLIPPED=$(printf "%02x" $(( 0x$ORIG ^ 0x01 )))
     printf "\x$(echo $FLIPPED)" | dd of=/data/data.mdb bs=1 seek=$OFFSET conv=notrunc 2>/dev/null
     echo "Flipped byte at offset $OFFSET: 0x$ORIG -> 0x$FLIPPED"
   '
   ```
6. Restart node1: `$COMPOSE start node1`, wait_healthy
7. Check node1 logs for AEAD-related error messages. The node should detect AEAD authentication failure during startup integrity scan or first blob access:
   ```bash
   docker logs chromatindb-test-node1-1 2>&1 | grep -i "aead\|decrypt\|corrupt\|integrity\|authentication"
   ```
   Expect AEAD authentication failure or integrity check messages.
8. Start node2 (which bootstraps from node1). Node1 should NOT serve the corrupted blob.
9. Wait for sync between node1 and node2.
10. Check node2 blob count: it should be less than 5 (corrupted blob not served) OR the sync should show errors for the corrupted blob.

**Edge case handling**: If the flipped bit hits mdbx metadata rather than blob data, mdbx itself may detect corruption and refuse to open. This is also acceptable behavior (database integrity works). The test should handle both cases: mdbx corruption = OK (database integrity works), AEAD failure = OK (DARE integrity works). Either way, corrupted data is not served.

Script should be chmod +x, source helpers.sh, trap cleanup on EXIT. Use the standard 2-node compose but start only node1 first, then add node2 for the sync check:

```bash
$COMPOSE up -d node1
wait_healthy chromatindb-test-node1-1
# ... ingest, stop, corrupt, restart ...
$COMPOSE up -d node2
wait_healthy chromatindb-test-node2-1
```
