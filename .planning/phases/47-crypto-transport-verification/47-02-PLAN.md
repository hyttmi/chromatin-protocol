- signing_digest = SHA3-256(namespace_id || data || ttl_le32 || timestamp_le64) -- what gets signed
- ML-DSA-87 signature signs the signing_digest (32 bytes), NOT the raw data
- DARE: HKDF-derived key with context "chromatindb-dare-v1", envelope [version][nonce][ciphertext+tag]
- Blobs stored encrypted via ChaCha20-Poly1305; bit flip causes AEAD auth failure on decryption

**test_crypt03_tamper_detection.sh** -- Verifies CRYPT-03: single bit flip in data.mdb causes AEAD failure and corrupted blob is never served.

Test flow:
1. Source helpers.sh, check_deps, build_image, cleanup, start_nodes (node1 only -- single node test)
2. Run loadgen to ingest 5 blobs to node1: `--count 5 --size 1024 --ttl 3600`
3. Verify node1 has 5 blobs (SIGUSR1 metrics)
4. Stop node1: `$COMPOSE stop node1`
5. Flip a single bit in data.mdb:
   - Find the data.mdb file in the container's volume: `docker run --rm -v node1-data:/data debian:bookworm-slim ls -la /data/data.mdb`
   - Use xxd + sed/printf to flip a bit. The data.mdb contains DARE-encrypted blobs. Flip a byte somewhere in the middle of the file (past the mdbx header, which is first 4KB):
   ```bash
   # Read a byte at offset 8192, flip bit 0, write back
   docker run --rm -v node1-data:/data debian:bookworm-slim sh -c '
     OFFSET=8192
     ORIG=$(xxd -s $OFFSET -l 1 -p /data/data.mdb)
     FLIPPED=$(printf "%02x" $(( 0x$ORIG ^ 0x01 )))
     printf "\x$(echo $FLIPPED)" | dd of=/data/data.mdb bs=1 seek=$OFFSET conv=notrunc 2>/dev/null
     echo "Flipped byte at offset $OFFSET: 0x$ORIG -> 0x$FLIPPED"
   '
   ```
   Note: The flip offset must be past the mdbx metadata pages. mdbx meta pages are in the first 3 pages (page size typically 4096). So offset 8192+ should be in data pages. If 8192 happens to be in a free page or mdbx B-tree node, the corruption may manifest differently. To maximize chance of hitting encrypted blob data, use a larger offset like 16384 or scan for non-zero data regions.
   Better approach: find a region with non-zero data (high-entropy DARE envelope):
   ```bash
   docker run --rm -v node1-data:/data debian:bookworm-slim sh -c '
     # Find first 256-byte block of high-entropy data (DARE envelope) past header
     xxd -s 12288 -l 4096 -p /data/data.mdb | head -c 64
   '
   ```
   Then flip a bit in that region.

6. Restart node1: `$COMPOSE start node1`, wait_healthy
7. Check node1 logs for AEAD-related error messages. The node should detect AEAD authentication failure when it tries to decrypt blobs during startup integrity scan or first access:
   ```bash
   docker logs chromatindb-test-node1-1 2>&1 | grep -i "aead\|decrypt\|corrupt\|integrity\|authentication"
   ```
   Expect to see AEAD authentication failure or integrity check messages.
8. Start node2 (which bootstraps from node1). Node1 should NOT serve the corrupted blob.
9. Wait for sync between node1 and node2.
10. Check node2 blob count: it should be less than 5 (corrupted blob not served) OR the sync should show errors for the corrupted blob.
11. Alternatively, verify that node1's blob count after restart is less than 5 (corrupted blob rejected during integrity scan).

**Key consideration**: The startup integrity scan (added in v0.9.0) should detect DARE corruption and quarantine/skip the corrupted entry. The test verifies this behavior.

**Edge case**: If the flipped bit happens to be in mdbx metadata rather than blob data, mdbx itself may detect corruption and refuse to open. This is also acceptable behavior (the database detects corruption). The test should handle both cases: mdbx corruption = OK (database integrity works), AEAD failure = OK (DARE integrity works). Either way, corrupted data is not served.

Script should be chmod +x, source helpers.sh, trap cleanup on EXIT.
