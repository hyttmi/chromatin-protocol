# Phase 999.3: Relay Request Timeout with Error Feedback - Context

**Gathered:** 2026-04-11
**Status:** Ready for planning

<domain>
## Phase Boundary

The relay's RequestRouter silently drops pending requests after 60s via `purge_stale()` — clients never learn their request timed out. This phase adds configurable per-request timeouts that send ErrorResponse(63) with a new `timeout` error code back to the client before cleaning up the pending entry.

</domain>

<decisions>
## Implementation Decisions

### Timeout Value & Config
- **D-01:** Default request timeout is 10 seconds. Configurable via `request_timeout_seconds` field in relay config JSON. SIGHUP-reloadable like other relay config fields.
- **D-02:** Config value 0 means disabled (no timeout enforcement, revert to current 60s silent cleanup behavior). Minimum non-zero value: 1 second.

### Error Code
- **D-03:** New error code `timeout = 6` added to the node's error_codes.h enum. The relay generates this ErrorResponse locally (relay-originated, not from node) when a pending request exceeds the configured timeout.
- **D-04:** The relay constructs the ErrorResponse binary payload itself: `[code:1=6][original_type:1]` where original_type comes from the PendingRequest. This means PendingRequest must store the original message type byte.

### Timeout Granularity
- **D-05:** Single global timeout for all request types. No per-type timeout configuration — all requests share one value. Per-type adds complexity with no demonstrated need.

### Cleanup Loop Behavior
- **D-06:** Cleanup interval changes from fixed 60s to `timeout / 2` (e.g., 5s for 10s timeout). This ensures requests are caught within ~1.5x the configured timeout, much more responsive than the current fixed 60s scan.
- **D-07:** When timeout is disabled (0), the cleanup loop reverts to the existing 60s interval with silent purge (backward compatible).

### PendingRequest Extension
- **D-08:** PendingRequest struct gains a `uint8_t original_type` field to record what type of request the client sent. Required for constructing the ErrorResponse original_type byte.

### Metrics
- **D-09:** Existing `errors_total` relay Prometheus counter incremented on timeout. No new counter needed — timeout is just another error type.

### Claude's Discretion
- Whether to add a `request_timeouts_total` counter in addition to incrementing errors_total
- Whether `purge_stale` should be renamed to better reflect its new behavior (sends errors, not just purges)
- Test strategy for verifying timeout behavior (timer mocking vs short timeouts in tests)

</decisions>

<specifics>
## Specific Ideas

- The relay generates ErrorResponse locally for timeouts -- the node is not involved (it didn't respond, that's the whole point)
- PendingRequest already has `created` timestamp which purge_stale uses -- no new timing infrastructure needed
- Phase 999.2's deferred ideas explicitly noted this phase: "999.3 (relay request timeout) -- complementary feature. ErrorResponse reduces its urgency since most timeouts were caused by silent failures."
- With ErrorResponse from 999.2, most "timeout" scenarios are actually eliminated (node now responds to malformed requests). This phase catches the remaining case: node genuinely slow or unresponsive.

</specifics>
