# Phase 999.3: Relay Request Timeout with Error Feedback - Research

**Researched:** 2026-04-11
**Domain:** Relay RequestRouter timeout + ErrorResponse generation
**Confidence:** HIGH

## Summary

This phase adds configurable per-request timeouts to the relay's RequestRouter, replacing the current 60-second silent cleanup with active error feedback to clients. The change is well-scoped: it modifies 5 existing files (RequestRouter, UdsMultiplexer, RelayConfig, error_codes.h, translator.cpp) and adds a new config field. All patterns needed already exist in the codebase -- the bulk_fail_all callback pattern, SIGHUP-reloadable config via std::atomic, and ErrorResponse binary construction/translation.

The key technical challenge is making the cleanup interval dynamic (timeout/2) while keeping it SIGHUP-reloadable. The UdsMultiplexer's cleanup_loop coroutine currently has a hardcoded 60s timer. The interval must read from an atomic or be recalculated each iteration. The ErrorResponse payload construction ([code:1=6][original_type:1]) is trivial -- 2 bytes, locally generated by the relay without node involvement.

**Primary recommendation:** Extend PendingRequest with `original_type`, add callback-based `purge_stale` overload, make cleanup interval dynamic, and wire timeout errors through the existing send_json path. This is a 4-5 file change with clear test boundaries.

### Locked Decisions
- D-01: Default request timeout is 10 seconds. Configurable via `request_timeout_seconds` field in relay config JSON. SIGHUP-reloadable like other relay config fields.
- D-02: Config value 0 means disabled (no timeout enforcement, revert to current 60s silent cleanup behavior). Minimum non-zero value: 1 second.
- D-03: New error code `timeout = 6` added to the node's error_codes.h enum. The relay generates this ErrorResponse locally (relay-originated, not from node) when a pending request exceeds the configured timeout.
- D-04: The relay constructs the ErrorResponse binary payload itself: `[code:1=6][original_type:1]` where original_type comes from the PendingRequest. This means PendingRequest must store the original message type byte.
- D-05: Single global timeout for all request types. No per-type timeout configuration.
- D-06: Cleanup interval changes from fixed 60s to `timeout / 2` (e.g., 5s for 10s timeout). When timeout is disabled (0), the cleanup loop reverts to the existing 60s interval with silent purge.
- D-07: When timeout is disabled (0), the cleanup loop reverts to the existing 60s interval with silent purge (backward compatible).
- D-08: PendingRequest struct gains a `uint8_t original_type` field.
- D-09: Existing `errors_total` relay Prometheus counter incremented on timeout. No new counter needed.

### Pattern 1: PendingRequest Extension

**What:** Add `uint8_t original_type` to PendingRequest, store at register time.
**Current struct:**
```cpp
struct PendingRequest {
    uint64_t client_session_id;
    uint32_t client_request_id;
    std::chrono::steady_clock::time_point created;
};
```
**After:**
```cpp
struct PendingRequest {
    uint64_t client_session_id;
    uint32_t client_request_id;
    std::chrono::steady_clock::time_point created;
    uint8_t original_type;  // D-08: for ErrorResponse construction
};
```
**Impact:** `register_request()` signature changes to accept the type byte. All callers (WsSession's handle_client_message) must pass it.

### Pattern 2: Callback-based purge_stale

**What:** Add a callback overload that fires before each stale entry is erased.
**Current:**
```cpp
size_t purge_stale(std::chrono::seconds timeout = std::chrono::seconds(60));
```
**Recommended approach:** Change purge_stale to accept an optional callback:
```cpp
size_t purge_stale(std::chrono::seconds timeout,
                   std::function<void(const PendingRequest&)> on_timeout = nullptr);
```
When `on_timeout` is non-null, it is invoked for each stale entry before erasure. The UdsMultiplexer's cleanup_loop wires this callback to construct and send the ErrorResponse JSON. When null (or when timeout=0/disabled), behavior is identical to current silent purge.

This follows the same callback pattern as `bulk_fail_all(std::function<void(session_id, client_rid)>)`.

### Pattern 3: SIGHUP-reloadable Config via Atomic

**What:** The relay uses `std::atomic<uint32_t>` for SIGHUP-reloadable fields. The `rate_limit_messages_per_sec` pattern in relay_main.cpp is the exact template.
**Established pattern (from relay_main.cpp):**
```cpp
// Declaration
std::atomic<uint32_t> rate_limit_rate{cfg.rate_limit_messages_per_sec};
// SIGHUP reload
rate_limit_rate.store(new_cfg.rate_limit_messages_per_sec, std::memory_order_relaxed);
```
**For request_timeout:** Same pattern. Declare `std::atomic<uint32_t> request_timeout_seconds{cfg.request_timeout_seconds}` in main, pass pointer to UdsMultiplexer. The cleanup_loop reads this atomic each iteration to determine both the timeout threshold and the sleep interval.

### Pattern 4: Dynamic Cleanup Interval

**What:** cleanup_loop currently sleeps 60s. Per D-06, interval = timeout/2 (minimum 1s to avoid busy-loop). Per D-07, when disabled (0), use 60s.
**Current:**
```cpp
asio::awaitable<void> UdsMultiplexer::cleanup_loop() {
    while (connected_) {
        asio::steady_timer timer(ioc_);
        timer.expires_after(std::chrono::seconds(60));
        co_await timer.async_wait(...);
        auto purged = router_.purge_stale();
    }
}
```
**After:** Read the atomic timeout value each iteration. Compute interval = (timeout > 0) ? max(timeout/2, 1) : 60. Pass timeout (or 60 if disabled) to purge_stale. Wire callback when timeout > 0.

### Pattern 5: Relay-Originated ErrorResponse

**What:** The relay constructs the ErrorResponse binary itself (it does NOT go through the node). The binary payload is 2 bytes: `[error_code:1][original_type:1]`. This gets translated by the existing `decode_error_response()` in translator.cpp which already handles the [code:1][type:1] format.
**Flow:**
1. cleanup_loop detects stale entry
2. Constructs JSON directly (no need for binary round-trip since relay already speaks JSON to clients):
   ```json
   {"type": "error", "code": "timeout", "original_type": "read_request", "request_id": 42}
   ```
3. Sends via session->send_json()

**Important design choice:** The relay should construct the JSON error directly rather than building binary and translating. The binary construction path (D-04) describes the logical payload, but since the relay is the originator and the client gets JSON, the relay can skip the binary->JSON round-trip and build JSON directly. The translator's decode_error_response already maps code 6 -> "timeout" and original_type -> name, so this logic should be reused or inlined.

**Recommendation:** Use type_to_string() from the type registry to map original_type byte to name, then build JSON directly in the callback. This is simpler and avoids unnecessary encode/decode.

### Anti-Patterns to Avoid
- **Encoding binary then decoding:** The relay generates the error -- skip the binary->JSON translation and build JSON directly. The only reason to go through binary would be if the error also gets sent over UDS, which it does not.
- **Per-request timers:** Do NOT create individual asio timers per pending request. The periodic sweep is the correct pattern at this scale. Individual timers would waste resources and complicate cancellation.
- **Renaming purge_stale:** The name is fine. Adding a callback does not change its primary purpose. Document the new behavior in the docstring instead.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Type name lookup | Manual switch/map for original_type | `translate::type_to_string(uint8_t)` | Already exists in type_registry, returns optional<string_view> |
| Error code name | Manual string for "timeout" | Add to error_codes.h + translator ERROR_CODES table | Consistent with all other error codes |
| Config parsing | Manual JSON field extraction | Extend existing relay_config.cpp pattern | j.value("field", default) pattern established |
| SIGHUP propagation | New reload mechanism | Copy rate_limit_rate atomic pattern from relay_main.cpp | Battle-tested, already in production |

## Common Pitfalls

### Pitfall 1: cleanup_loop Restart on Reconnect
**What goes wrong:** cleanup_loop is spawned after each successful handshake (line 172-175 of uds_multiplexer.cpp). If the UDS disconnects and reconnects, a NEW cleanup_loop is spawned. The old one exits when `connected_` goes false, but there is a brief window where both could run.
**Why it happens:** The loop checks `connected_` which is set to false on disconnect, causing the old loop to exit on next timer wake. No actual concurrency issue since both run on the same io_context (single-strand).
**How to avoid:** No action needed -- the existing design handles this correctly. The old coroutine exits, the new one starts fresh. Just ensure the atomic timeout is read each iteration (not captured at spawn time).

### Pitfall 2: Timeout 0 Still Needs Cleanup
**What goes wrong:** If someone sets timeout=0 and expects no cleanup at all.
**Why it happens:** D-07 says 0 = disabled (revert to 60s silent purge). The cleanup loop must still run for the 60s stale sweep.
**How to avoid:** When timeout=0, pass 60s to purge_stale with no callback. Document this clearly.

### Pitfall 3: register_request Signature Change Breaks Callers
**What goes wrong:** Adding `original_type` parameter to register_request changes the signature. All call sites must be updated.
**Why it happens:** WsSession::handle_client_message calls register_request. Need to find all callers.
**How to avoid:** Grep for `register_request` before implementation. There should be exactly 1 call site in WsSession (the AUTHENTICATED message handling path).
**Warning signs:** Compile error on missed call site.

### Pitfall 4: Session Already Gone When Timeout Fires
**What goes wrong:** A client disconnects between the request registration and the timeout sweep. The session is gone when the callback tries to send.
**Why it happens:** Normal -- clients disconnect. The `sessions_.get_session()` call returns nullptr.
**How to avoid:** Already handled by the existing pattern in bulk_fail_pending_requests: `if (!session) return;`. The timeout callback must do the same null check.

### Pitfall 5: Metrics Pointer May Be Null
**What goes wrong:** The cleanup_loop callback tries to increment errors_total but metrics pointer is null.
**Why it happens:** UdsMultiplexer does not currently hold a metrics pointer (unlike WsSession which has metrics_).
**How to avoid:** Either pass RelayMetrics* to UdsMultiplexer (like WsSession's pattern) or pass it through the atomic/callback wiring. The simplest approach is adding a `RelayMetrics*` member to UdsMultiplexer, set during construction or via setter like set_tracker().

### Pitfall 6: request_id 0 in ErrorResponse
**What goes wrong:** If the client sent request_id=0 (fire-and-forget per D-12), including request_id=0 in the timeout error is misleading.
**Why it happens:** Fire-and-forget requests conceptually don't expect responses, but they still sit in the pending map.
**How to avoid:** Only include `"request_id"` in the JSON when client_request_id != 0. Follow the pattern from bulk_fail_pending_requests: `if (client_rid != 0) err["request_id"] = client_rid;`.

## Discretion Recommendations

### request_timeouts_total Counter
**Recommendation: Add it.** A dedicated `request_timeouts_total` counter is cheap (one more atomic<uint64_t>) and provides operational value. Operators can distinguish timeouts from other errors without log parsing. Still increment `errors_total` as well (D-09). Add to Prometheus output alongside existing counters.

### purge_stale Naming
**Recommendation: Keep the name.** `purge_stale` accurately describes what it does -- purge stale entries. The callback is an optional side effect. Update the docstring to mention the callback sends errors. Renaming would break the audit trail from Phase 103's D-11 references.

### Test Strategy
**Recommendation: Short timeouts (not timer mocking).** Use 1-second timeouts in tests with `std::this_thread::sleep_for` to age entries past the threshold. This matches the existing `purge_stale(std::chrono::seconds(0))` pattern in test_request_router.cpp which uses 0s to force immediate staleness. Timer mocking adds framework complexity with no benefit for these tests.

Test plan:
1. **Unit tests (test_request_router.cpp):** Test purge_stale with callback -- verify callback invoked per stale entry with correct session_id, client_rid, and original_type. Test that non-stale entries are preserved. Test with null callback (silent purge).
2. **Unit tests (test_relay_config.cpp):** Test request_timeout_seconds parsing -- default 10, explicit value, value 0, minimum 1 validation.
3. **Integration concern:** The cleanup_loop timing behavior is hard to unit-test (coroutine + timer). Rely on the unit tests for purge_stale correctness and manual/smoke testing for the loop interval behavior.

## Code Examples

### ErrorResponse JSON Construction (in timeout callback)
```cpp
// Inside cleanup_loop's purge callback (UdsMultiplexer)
auto session = sessions_.get_session(pending.client_session_id);
if (!session) return;

nlohmann::json err;
err["type"] = "error";
err["code"] = "timeout";

// Map original_type byte to human-readable name
auto type_name = translate::type_to_string(pending.original_type);
err["original_type"] = type_name
    ? std::string(*type_name)
    : std::to_string(pending.original_type);

if (pending.client_request_id != 0) {
    err["request_id"] = pending.client_request_id;
}

session->send_json(err);
```

### Dynamic Cleanup Loop
```cpp
asio::awaitable<void> UdsMultiplexer::cleanup_loop() {
    while (connected_) {
        uint32_t timeout_sec = request_timeout_->load(std::memory_order_relaxed);
        bool timeout_enabled = (timeout_sec > 0);

        // D-06: interval = timeout/2, D-07: disabled = 60s
        uint32_t interval = timeout_enabled
            ? std::max(timeout_sec / 2, 1u)
            : 60u;

        asio::steady_timer timer(ioc_);
        timer.expires_after(std::chrono::seconds(interval));
        auto [ec] = co_await timer.async_wait(use_nothrow);
        if (ec || !connected_) break;

        auto threshold = std::chrono::seconds(timeout_enabled ? timeout_sec : 60);

        if (timeout_enabled) {
            auto purged = router_.purge_stale(threshold,
                [this](const PendingRequest& pending) {
                    // Send timeout error to client
                    send_timeout_error(pending);
                });
            if (purged > 0) {
                spdlog::info("cleanup: {} request(s) timed out", purged);
            }
        } else {
            auto purged = router_.purge_stale(threshold);
            if (purged > 0) {
                spdlog::debug("cleanup: purged {} stale requests", purged);
            }
        }
    }
}
```
