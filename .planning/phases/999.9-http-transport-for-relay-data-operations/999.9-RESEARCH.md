# Phase 999.9: HTTP Transport for Relay Data Operations - Research

**Researched:** 2026-04-13
**Domain:** HTTP/1.1 server, SSE, session token auth -- replacing WebSocket transport in chromatindb relay
**Confidence:** HIGH

## Summary

This phase replaces the entire WebSocket transport layer in the chromatindb relay with HTTP/1.1 endpoints for request/response operations and Server-Sent Events (SSE) for pub/sub notifications. The relay already has a working coroutine-based HTTP server pattern in MetricsCollector (accept loop, header parsing, route matching, response writing) that scales to a full HTTP server. The core components (UdsMultiplexer, Translator, Authenticator, RequestRouter, SubscriptionTracker, WriteTracker, RateLimiter, MetricsCollector) stay untouched -- only the transport layer changes.

The migration path is straightforward because the existing WsSession message flow (receive -> parse JSON -> check type -> check rate limit -> translate -> UDS send -> route response -> translate back -> send JSON) maps directly to HTTP handler functions. Each WsSession handler block becomes an HTTP endpoint handler. The challenge-response auth stays identical at the crypto level but changes from an in-band WS message exchange to a two-step HTTP POST flow that issues a bearer token. SSE replaces the WS notification push channel.

**Primary recommendation:** Build the HTTP server core first (parser, router, TLS acceptor, connection lifecycle), then migrate auth to token-based, then port each endpoint category (data, query, pub/sub), then add SSE, then delete all WS code. The MetricsCollector pattern is the foundation -- extend it rather than build from scratch.

### Pattern 1: Coroutine-Per-Connection HTTP Server
**What:** Each accepted TCP connection gets a coroutine that reads HTTP requests in a loop (keep-alive). One coroutine handles the full request lifecycle: read headers, read body, route, process, respond.
**When to use:** Always -- this is the MetricsCollector pattern extended.
**Example:**
```cpp
// Based on existing relay/core/metrics_collector.cpp:90-160
asio::awaitable<void> HttpServer::handle_connection(TlsOrTcpStream stream) {
    while (!stopping_) {
        // 1. Read HTTP request headers until \r\n\r\n
        auto request = co_await read_request(stream);
        if (!request) co_return;  // Connection closed or parse error

        // 2. Read body based on Content-Length (if present)
        if (request->content_length > 0) {
            co_await read_body(stream, *request);
        }

        // 3. Route to handler
        auto response = co_await router_.dispatch(*request, stream);

        // 4. Send response
        co_await send_response(stream, response);

        // 5. Check Connection: close
        if (!request->keep_alive) co_return;
    }
}
```

### Pattern 2: Token-Based Session Management
**What:** Two-step auth (POST /auth/challenge, POST /auth/verify) produces an opaque bearer token. All subsequent requests include `Authorization: Bearer <token>`. The relay stores a `token -> SessionState` mapping with the client's public key, namespace hash, rate limiter, and subscription set.
**When to use:** Every authenticated endpoint (everything except /auth/* and /health).
**Example:**
```cpp
struct HttpSessionState {
    uint64_t session_id;
    std::vector<uint8_t> client_pubkey;
    std::array<uint8_t, 32> client_namespace;
    RateLimiter rate_limiter;
    std::chrono::steady_clock::time_point last_activity;
    // SSE connection state (if active)
    SseWriter* sse_writer = nullptr;  // Non-owning; SSE coroutine owns lifetime
};

class TokenStore {
    std::unordered_map<std::string, HttpSessionState> tokens_;  // token_hex -> state
    std::unordered_map<uint64_t, std::string> id_to_token_;     // session_id -> token
    uint64_t next_session_id_ = 1;
public:
    std::string create_session(std::vector<uint8_t> pubkey,
                               std::array<uint8_t, 32> ns_hash);
    HttpSessionState* lookup(const std::string& token);
    void remove_session(uint64_t session_id);
};
```

### Pattern 3: SSE Notification Stream
**What:** `GET /events?token=<token>` establishes a long-lived HTTP response with `Content-Type: text/event-stream`. The server writes events as `data: <json>\n\n` lines. The connection stays open indefinitely. Client reconnects with `Last-Event-ID` header.
**When to use:** Replaces WS notification push.
**Example:**
```cpp
asio::awaitable<void> handle_sse_stream(TlsOrTcpStream& stream,
                                         HttpSessionState& session) {
    // Send SSE response headers
    std::string headers =
        "HTTP/1.1 200 OK\r\n"
        "Content-Type: text/event-stream\r\n"
        "Cache-Control: no-cache\r\n"
        "Connection: keep-alive\r\n"
        "\r\n";
    co_await async_write(stream, asio::buffer(headers));

    // Register SSE writer with session
    SseWriter writer(stream);
    session.sse_writer = &writer;

    // Block until disconnection -- events pushed by UdsMultiplexer fan-out
    co_await writer.wait_for_close();

    session.sse_writer = nullptr;
}

// SSE event format:
// id: <seq_num>\n
// data: {"namespace":"<hex>","blob_hash":"<hex>",...}\n
// \n
```

### Pattern 4: Binary Blob Pass-Through (D-07, D-08, D-09)
**What:** Blob write/read/delete endpoints bypass JSON translation entirely. The HTTP body IS the raw FlatBuffer binary that the node expects. The relay wraps it in a TransportCodec envelope and sends it over UDS. The response is similarly raw binary (for reads) or JSON (for write/delete acks).
**When to use:** POST /blob, GET /blob/{ns}/{hash}, DELETE /blob/{ns}/{hash}.
**Example:**
```cpp
// POST /blob handler
asio::awaitable<HttpResponse> handle_blob_write(
    const HttpRequest& req, HttpSessionState& session) {
    // req.body is already raw FlatBuffer bytes -- no JSON parsing needed
    // Wire type 8 (Data), no translation
    auto transport_msg = TransportCodec::encode(
        TransportMsgType_Data, req.body, relay_rid);
    uds_mux_->send(std::move(transport_msg));
    // Wait for WriteAck via RequestRouter
    auto ack = co_await wait_for_response(relay_rid);
    // Translate WriteAck binary -> JSON for response
    auto json = translate::binary_to_json(30, ack.payload);
    co_return HttpResponse::json(200, *json);
}
```

### Pattern 5: Prefix-Match HTTP Router
**What:** Simple string prefix matching for routing. No regex, no trie. The relay has ~20 fixed routes. A sorted vector of (method, prefix) pairs with linear scan is fast enough for this endpoint count and avoids external dependencies.
**When to use:** HTTP request routing.
**Recommendation:** Use prefix-based matching with a sorted route table. For parameterized paths like `/blob/{namespace}/{hash}`, extract path segments after the prefix match. This is simpler and faster than regex for fixed-structure routes.
**Example:**
```cpp
struct Route {
    std::string_view method;   // "GET", "POST", "DELETE"
    std::string_view prefix;   // "/blob", "/list/", "/auth/challenge"
    HandlerFn handler;
};

// Route matching: exact prefix match, then parse remaining path segments
// Example: "GET /blob/abc123.../def456..." matches Route{"GET", "/blob/"}
// remaining = "abc123.../def456..."  -> split on '/' for namespace and hash
```

### Anti-Patterns to Avoid
- **HTTP/2 complexity:** Do not implement HTTP/2. HTTP/1.1 with keep-alive is sufficient. The relay serves ~20 endpoints for a limited client set. HTTP/2's multiplexing is unnecessary and adds massive implementation complexity.
- **Connection: Upgrade for SSE:** SSE does NOT use HTTP Upgrade. It is a normal HTTP response that never ends. Do not confuse SSE with WebSocket upgrade semantics.
- **Pipelining:** Do NOT implement HTTP request pipelining (multiple requests before receiving responses). It is rarely used by clients and adds significant complexity. Handle one request at a time per connection with keep-alive.
- **Chunked request bodies:** Clients sending blobs should always use Content-Length, not chunked encoding. The server can reject Transfer-Encoding: chunked on request bodies to simplify implementation. Chunked encoding is only needed for SSE responses (server -> client).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| HTTP parser | Full RFC 7230 parser | Minimal line-based parser for known endpoints | Relay only serves its own API. Don't parse arbitrary HTTP. MetricsCollector already parses first-line + headers until CRLFCRLF. Extend this. |
| URL decoding | percent-decode library | Inline hex decode for path segments | Only need to decode hex namespace/hash from URL paths. No query string encoding issues (all hex chars). |
| Session tokens | JWT library | RAND_bytes(32) + hex encode | Opaque tokens are simpler, more secure (no signature verification overhead per request), and the relay is the only validator. No interoperability needed. |
| HTTP date formatting | date library | Static "Connection: close" or omit Date header | RFC 7231 says Date is only SHOULD for origin servers. A relay proxy can omit it. |
| Multipart responses | multipart library | JSON array with base64 for batch reads | Multipart/mixed adds parser complexity on both client and server. For batch reads, JSON with base64-encoded blobs is simpler and the blobs are small enough (batch reads are capped). |

**Key insight:** The relay is NOT a general-purpose HTTP server. It serves exactly its own API to authenticated clients. Every "standard" HTTP feature (content negotiation, range requests, caching, etags) is irrelevant. Build the minimum parser that handles the known request shapes.

## Common Pitfalls

### Pitfall 1: SSE Connection Memory Leak
**What goes wrong:** SSE connections are long-lived. If the SSE coroutine holds a shared_ptr to session state but the client disconnects silently (no TCP FIN), the session state leaks indefinitely.
**Why it happens:** TCP keepalive is off or too infrequent. The SSE write coroutine blocks on a signal and never detects the peer is gone.
**How to avoid:** (1) Set TCP_KEEPALIVE on SSE connections with short intervals (30s). (2) Send SSE comments (`: heartbeat\n\n`) every 30 seconds -- a failed write detects disconnection. (3) Idle timeout on token store -- reap sessions with no activity for configurable duration. (4) When SSE write fails, remove session from TokenStore and SubscriptionTracker.
**Warning signs:** Growing session count in /metrics that never decreases.

### Pitfall 2: UdsMultiplexer Depends on ws::SessionManager
**What goes wrong:** UdsMultiplexer currently takes a `ws::SessionManager&` in its constructor and calls `sessions_.for_each()` and `sessions_.get_session()` to route responses. Replacing SessionManager with TokenStore breaks compilation everywhere.
**Why it happens:** Tight coupling between UDS response routing and WS session lookup.
**How to avoid:** Introduce a session lookup interface (abstract base or std::function callbacks) that UdsMultiplexer uses. TokenStore implements it. This decouples UDS from the transport layer. Alternatively, change UdsMultiplexer to accept lambdas for: (1) get_session_by_id, (2) for_each_session, (3) send_json_to_session. The TokenStore + HttpSession can provide these.
**Warning signs:** Compilation errors in uds_multiplexer.cpp referencing ws::WsSession.

### Pitfall 3: Request/Response Correlation in HTTP
**What goes wrong:** In WebSocket, a single connection handles both requests and responses asynchronously. The WsSession sends a request and later receives the response via send_json() called from UdsMultiplexer. In HTTP, each request blocks its coroutine until the response arrives.
**Why it happens:** HTTP is synchronous per-request. The coroutine handling POST /blob must co_await the response from the node before it can send the HTTP response.
**How to avoid:** Use a per-request awaitable pattern: (1) Register request with RequestRouter, get relay_rid. (2) Create a one-shot `asio::steady_timer` (used as a signal). (3) Store the timer pointer in a pending map keyed by relay_rid. (4) When UdsMultiplexer routes the response, look up the timer and cancel it (signaling completion). (5) The HTTP handler coroutine co_awaits the timer -- when cancelled, the response data is available.
**Warning signs:** HTTP requests that hang forever. Missing responses. Race conditions between timer expiry and response arrival.

### Pitfall 4: Binary Body Size Limits
**What goes wrong:** A client sends a 100 MiB blob via POST /blob. The server tries to allocate 100 MiB in memory before processing.
**Why it happens:** The naive implementation reads the entire Content-Length body into a vector before forwarding.
**How to avoid:** For blob write, the Content-Length is known upfront (from HTTP header). Check against max_blob_size BEFORE reading the body. For reading the body, stream it in chunks to the UDS -- but this requires chunked UDS writes which the current UdsMultiplexer does not support. For pre-MVP, reading the full body into memory is acceptable given the existing 100 MiB node limit. Add the Content-Length check as the first step.
**Warning signs:** OOM kills under load with large blobs.

### Pitfall 5: CORS and Preflight
**What goes wrong:** Browser clients send OPTIONS preflight requests. The server returns 404 or 405, blocking all subsequent requests.
**Why it happens:** If any clients are browser-based (JS EventSource for SSE), CORS preflight is mandatory.
**How to avoid:** Add a catch-all OPTIONS handler that returns 204 with Access-Control-Allow-Origin, Access-Control-Allow-Headers (Authorization, Content-Type), and Access-Control-Allow-Methods. Also add CORS headers to every response. This is simple (5-10 lines in the router) and costs nothing.
**Warning signs:** Browser console showing CORS errors.

### Pitfall 6: Session Token in SSE URL Query String
**What goes wrong:** GET /events?token=<token> puts the session token in the URL. This is visible in server logs, browser history, and referrer headers.
**Why it happens:** SSE EventSource API in browsers cannot set custom headers. The only way to authenticate SSE is via query params or cookies.
**How to avoid:** This is an accepted trade-off per D-04. Mitigate by: (1) keeping tokens short-lived, (2) not logging full URLs at info level (log path only, mask token), (3) using TLS so the URL is encrypted on the wire. Token in query string is the standard pattern for SSE auth.
**Warning signs:** Tokens appearing in log files.

### Pitfall 7: Asio Coroutine Stack-Use-After-Return with std::visit
**What goes wrong:** std::visit with a coroutine lambda causes ASAN stack-use-after-return when the coroutine suspends.
**Why it happens:** std::visit returns the result of the visitor, but if the visitor is a coroutine lambda, the coroutine frame outlives the visit call's stack frame.
**How to avoid:** Use `std::get_if` branching instead of `std::visit` for TLS/plain dual-mode streams, exactly as WsSession already does (audited in Phase 106 FIX-02). Copy this pattern to the new HTTP server.
**Warning signs:** ASAN stack-use-after-return in TLS/plain read/write paths.

### Pitfall 8: Send Queue Design Change
**What goes wrong:** The existing core::Session has a deque-based send queue with a drain coroutine, designed for WS frame serialization. HTTP responses are synchronous per-request -- the handler coroutine writes the response directly. The send queue model doesn't map to HTTP.
**Why it happens:** WS needed a send queue because multiple async events (notifications, responses, pongs) could trigger sends on the same connection concurrently. HTTP requests are sequential.
**How to avoid:** For request/response, the handler coroutine writes the response directly -- no send queue needed. For SSE, a send queue IS needed because notifications arrive asynchronously. Use a simple deque + signal pattern (similar to core::Session) only for the SSE writer. The core::Session class may need modification or a new SSE-specific send queue.
**Warning signs:** Deadlocks or races on SSE event writes from multiple notification sources.

### HTTP Response Builder
```cpp
struct HttpResponse {
    uint16_t status;
    std::string status_text;
    std::unordered_map<std::string, std::string> headers;
    std::vector<uint8_t> body;

    static HttpResponse json(uint16_t status, const nlohmann::json& j) {
        auto body_str = j.dump();
        return {status, status_text_for(status),
                {{"Content-Type", "application/json"},
                 {"Content-Length", std::to_string(body_str.size())}},
                {body_str.begin(), body_str.end()}};
    }

    static HttpResponse binary(uint16_t status, std::vector<uint8_t> data) {
        return {status, status_text_for(status),
                {{"Content-Type", "application/octet-stream"},
                 {"Content-Length", std::to_string(data.size())}},
                std::move(data)};
    }

    static HttpResponse not_found() {
        return json(404, {{"error", "not_found"}});
    }
};
```

### Request/Response Awaitable Pattern
```cpp
// In HTTP handler coroutine:
asio::awaitable<HttpResponse> handle_list_request(
    const HttpRequest& req, HttpSessionState& session,
    UdsMultiplexer& uds, RequestRouter& router) {
    // Build JSON for translation
    nlohmann::json j = {{"type", "list_request"},
                         {"namespace", extract_namespace(req.path)}};
    auto since = parse_query_param(req.query, "since_seq");
    if (since) j["since_seq"] = *since;

    auto result = translate::json_to_binary(j);
    if (!result) co_return HttpResponse::json(400, {{"error", "bad_request"}});

    uint32_t relay_rid = router.register_request(session.session_id, 0, result->wire_type);

    // Create response promise
    ResponsePromise promise(co_await asio::this_coro::executor);

    // Register promise with a pending response map
    pending_responses_[relay_rid] = &promise;

    // Send to node
    auto transport_msg = wire::TransportCodec::encode(
        static_cast<wire::TransportMsgType>(result->wire_type),
        result->payload, relay_rid);
    uds.send(std::move(transport_msg));

    // Wait for response (with timeout)
    auto [type, payload] = co_await promise.wait(std::chrono::seconds(10));

    pending_responses_.erase(relay_rid);

    auto json_resp = translate::binary_to_json(type, payload);
    if (!json_resp) co_return HttpResponse::json(502, {{"error", "translation_failed"}});
    co_return HttpResponse::json(200, *json_resp);
}
```

### SSE Event Writer
```cpp
class SseWriter {
    TlsOrTcpStream& stream_;
    asio::steady_timer signal_;
    std::deque<std::string> events_;
    bool closed_ = false;
public:
    void push_event(const nlohmann::json& data, uint64_t event_id) {
        std::string event = "id: " + std::to_string(event_id) + "\n"
                          + "data: " + data.dump() + "\n\n";
        events_.push_back(std::move(event));
        signal_.cancel();  // Wake drain loop
    }

    asio::awaitable<void> drain_loop() {
        while (!closed_) {
            while (!events_.empty()) {
                auto& ev = events_.front();
                auto [ec, _] = co_await async_write(stream_, asio::buffer(ev));
                if (ec) { closed_ = true; co_return; }
                events_.pop_front();
            }
            // Wait for next event
            signal_.expires_at(asio::steady_timer::time_point::max());
            auto [ec] = co_await signal_.async_wait(
                asio::as_tuple(asio::use_awaitable));
            // ec is asio::error::operation_aborted when new event arrives
        }
    }
};
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| WebSocket (RFC 6455) | HTTP + SSE | This phase | Full transport rewrite |
| In-band challenge-response (WS text frames) | HTTP POST + Bearer token | This phase | Separate auth flow, stateless requests |
| WS binary frames for blob read | Raw binary HTTP response body | This phase | Simpler, no base64 overhead |
| WS text frames for all JSON | HTTP JSON responses | This phase | Standard REST-like API |
| WS notification push | SSE text/event-stream | This phase | Standard browser-compatible push |
| Implicit session via WS connection | Explicit token-based sessions | This phase | Multiple HTTP connections share one session |

**Key architectural shift:** WebSocket ties session state to connection lifetime. HTTP decouples them. A session token can be used across multiple HTTP connections (one for SSE, others for requests). This is actually more resilient -- a client can make requests and receive notifications on different connections.

## Discretion Recommendations

### HTTP Request Routing: Prefix Match (Recommended)
Use a sorted vector of `{method, prefix, handler}` tuples with linear scan. For 20 routes, this is faster than a trie or regex and trivially debuggable. The path parameter extraction (namespace, hash) happens inside the handler after the prefix match.

### HTTP/2: Do NOT implement
HTTP/2 adds frame multiplexing, HPACK header compression, flow control, and server push. None of these benefit the relay's use case. HTTP/1.1 with keep-alive is sufficient. The implementation complexity is enormous (HPACK alone is hundreds of lines).

### Batch Read Format: JSON with Base64 (Recommended)
Multipart/mixed adds parser complexity on both sides and is poorly supported in many HTTP client libraries. Since batch reads are already capped by the BatchReadResponse truncation mechanism, blob data is bounded. Base64 encoding adds ~33% overhead but avoids binary-in-JSON framing issues. The blobs in batch reads are typically small (metadata queries).

### SSE Reconnection: Support Last-Event-ID (Recommended)
Implement event IDs based on a monotonically increasing counter per SSE connection. Store a small ring buffer of recent events (last 1000). On reconnect with `Last-Event-ID`, replay missed events from the ring buffer. If the ID is too old, send all current subscription state as a catchup burst. This is simple to implement and prevents notification loss during brief disconnections.

### Metrics/Health: Merge into Main HTTP Server (Recommended)
The MetricsCollector currently runs on a separate port. Merging /metrics and /health into the main HTTP server simplifies configuration (one bind address/port) and reduces code duplication (the MetricsCollector HTTP handler is the prototype for the full server). Keep the separate metrics_bind option as a secondary endpoint for monitoring systems that need a different port.

## UdsMultiplexer Decoupling Strategy

The UdsMultiplexer currently depends on `ws::SessionManager` and `ws::WsSession`:

| Dependency | Location | Purpose |
|------------|----------|---------|
| `ws::SessionManager& sessions_` | Constructor, field | Look up sessions for response routing |
| `sessions_.get_session(id)` | route_response() | Get WsSession to call send_json() |
| `sessions_.for_each()` | route_response() | Broadcast StorageFull/QuotaExceeded |
| `session->send_json()` | route_response(), handle_notification() | Send translated JSON to client |

**Strategy:** Replace with an abstract callback interface:
```cpp
// New interface in core/ (transport-agnostic)
struct SessionSender {
    std::function<void(uint64_t session_id, const nlohmann::json& msg)> send_json;
    std::function<void(const nlohmann::json& msg)> broadcast;
};
```

UdsMultiplexer takes `SessionSender` instead of `ws::SessionManager&`. For the HTTP transport, TokenStore provides the callbacks that route JSON to either: (a) a pending response promise (for request/response), or (b) an SSE writer (for notifications/broadcasts).

However, there is a subtlety: in HTTP, request/response pairs are correlated by the handler coroutine, not by send_json(). The UdsMultiplexer's route_response() currently calls send_json() on the WsSession. In HTTP, the response should wake up the waiting handler coroutine instead. This requires the response dispatch mechanism (ResponsePromise pattern from Pitfall 3) to be wired into UdsMultiplexer.

## Open Questions

1. **Response promise mechanism -- timer-cancel vs channel**
   - What we know: The handler coroutine needs to block until the UDS response arrives. Timer-cancel pattern (create timer, cancel when response arrives) is a known Asio pattern.
   - What's unclear: Is there a simpler mechanism? Asio experimental::channel could work but may not be available in standalone Asio 1.38.
   - Recommendation: Use the timer-cancel pattern. It is well-tested in the existing codebase (auth_timer, idle_timer use the same cancel-as-signal pattern).

2. **core::Session send queue fate**
   - What we know: The send queue was designed for WS frame serialization. HTTP responses are synchronous. SSE needs a send queue.
   - What's unclear: Should core::Session be repurposed for SSE only, or should a new SseWriter own its own queue?
   - Recommendation: Create a new SseWriter with its own queue. Keep core::Session until WS is deleted, then remove it. This avoids modifying core::Session during the transition.

3. **Concurrent requests on same keep-alive connection**
   - What we know: HTTP/1.1 allows keep-alive but requests are sequential (no pipelining). The coroutine-per-connection model handles this naturally -- read request, process, respond, loop.
   - What's unclear: What if a client opens multiple TCP connections to the same relay for parallel requests? This is fine -- each connection is independent, and they all share the same session token.
   - Recommendation: No special handling needed. One coroutine per connection, sequential request processing.
