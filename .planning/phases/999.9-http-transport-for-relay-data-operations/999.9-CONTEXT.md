# Phase 999.9: HTTP Transport for Relay Data Operations - Context

**Gathered:** 2026-04-13
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace the entire WebSocket relay protocol with HTTP + SSE. All request/response operations (write, read, list, stats, exists, batch, etc.) become HTTP endpoints with raw binary bodies for blob data. Pub/sub notifications use Server-Sent Events. WebSocket code is deleted entirely. This is a full relay transport rewrite — the UDS multiplexer, translator, authenticator, request router, subscription tracker, and metrics stay; the WS transport layer is replaced with HTTP.

</domain>

<decisions>
## Implementation Decisions

### Transport Architecture
- **D-01:** Full HTTP + SSE. No WebSocket. All request/response is HTTP. Notifications are SSE (text/event-stream).
- **D-02:** Delete all WebSocket code: ws_frame.h/cpp, ws_handshake.h/cpp, ws_session.h/cpp, ws_acceptor.h/cpp, and all WS tests. Clean break, pre-MVP.
- **D-03:** Standalone Asio HTTP server built on the existing io_context. Coroutine-based request handling. Support chunked transfer encoding, keep-alive, content-length, streaming for large bodies.

### Authentication
- **D-04:** Challenge-response → session token flow:
  1. `POST /auth/challenge` → relay returns `{"nonce": "<hex>"}`
  2. Client signs nonce with ML-DSA-87
  3. `POST /auth/verify` with `{"public_key": "<hex>", "signature": "<hex>"}` → relay verifies, returns `{"token": "<opaque>", "namespace": "<hex>"}`
  4. All subsequent requests include `Authorization: Bearer <token>` header
  5. SSE connection also authenticated via token query param: `GET /events?token=<token>`
- **D-05:** Session token is opaque (random bytes, hex-encoded). Relay stores token → session state mapping. Token expires on disconnect or after configurable idle timeout.
- **D-06:** Reuse existing Authenticator class for ML-DSA-87 verification and namespace derivation.

### API Endpoints — Data Operations
- **D-07:** `POST /blob` — Write a blob. Body: raw binary FlatBuffer-encoded blob (same as node wire format). Content-Type: application/octet-stream. Response: JSON WriteAck `{"blob_hash": "<hex>", "seq_num": N, "status": "stored"|"duplicate"}`.
- **D-08:** `GET /blob/{namespace}/{hash}` — Read a blob. Response: raw binary FlatBuffer-encoded blob with Content-Type: application/octet-stream. 404 if not found.
- **D-09:** `DELETE /blob/{namespace}/{hash}` — Delete (tombstone). Body: raw binary FlatBuffer-encoded tombstone blob. Response: JSON DeleteAck.

### API Endpoints — Query Operations
- **D-10:** `GET /list/{namespace}?since_seq=N&limit=N` — List blobs. Response: JSON array of {hash, seq_num}.
- **D-11:** `GET /stats/{namespace}` — Namespace stats. Response: JSON {blob_count, total_bytes, quota_bytes}.
- **D-12:** `GET /exists/{namespace}/{hash}` — Exists check. Response: JSON {exists: true/false}.
- **D-13:** `POST /batch/exists` — Batch exists. Body: JSON {namespace, hashes: [...]}. Response: JSON array of booleans.
- **D-14:** `POST /batch/read` — Batch read. Body: JSON {namespace, hashes: [...], cap_bytes: N}. Response: multipart or JSON with base64 (for small blobs) — Claude's discretion on format.
- **D-15:** `GET /node-info` — Node info. Response: JSON.
- **D-16:** `GET /peer-info` — Peer info (trust-gated). Response: JSON.
- **D-17:** `GET /storage-status` — Storage status. Response: JSON.
- **D-18:** `GET /namespace-stats/{namespace}` — Per-namespace stats. Response: JSON.
- **D-19:** `GET /metadata/{namespace}/{hash}` — Blob metadata. Response: JSON.
- **D-20:** `GET /delegations/{namespace}` — Delegation list. Response: JSON.
- **D-21:** `GET /time-range/{namespace}?start=N&end=N&limit=N` — Time range query. Response: JSON.

### API Endpoints — Pub/Sub
- **D-22:** `POST /subscribe` — Subscribe to namespaces. Body: JSON {namespaces: ["<hex>", ...]}. Response: 200 OK.
- **D-23:** `POST /unsubscribe` — Unsubscribe. Body: JSON {namespaces: [...]}. Response: 200 OK.
- **D-24:** `GET /events?token=<token>` — SSE notification stream. Content-Type: text/event-stream. Each notification is a JSON event: `data: {"namespace": "<hex>", "blob_hash": "<hex>", "seq_num": N, "blob_size": N, "is_tombstone": false}\n\n`.

### Existing Components — Keep
- **D-25:** UdsMultiplexer — stays as-is, handles UDS communication with node.
- **D-26:** Translator (json_to_binary / binary_to_json) — stays for query operations. Blob write/read bypass translation (raw binary pass-through).
- **D-27:** RequestRouter — stays for request_id mapping.
- **D-28:** SubscriptionTracker — stays for pub/sub.
- **D-29:** Authenticator — stays, reused for challenge-response verification.
- **D-30:** MetricsCollector — stays, /metrics and /health move into the main HTTP server.
- **D-31:** RateLimiter — stays, applied per-session on HTTP requests.
- **D-32:** WriteTracker — stays for source exclusion on SSE notifications.

### Existing Components — Delete
- **D-33:** Delete: ws_frame.h/cpp, ws_handshake.h/cpp, ws_session.h/cpp, ws_acceptor.h/cpp, session_manager.h/cpp (replace with HTTP session manager).
- **D-34:** Delete all WS-specific tests.

### HTTP Server Details
- **D-35:** Coroutine-based HTTP handler on shared io_context. One coroutine per connection.
- **D-36:** Support HTTP/1.1 keep-alive (Connection: keep-alive). Parse Content-Length for request body. Stream response body for large blobs (chunked transfer encoding).
- **D-37:** TLS via the same cert/key config as current WS (cert_path, key_path). Plain HTTP when TLS not configured.
- **D-38:** Same bind_address/bind_port config fields.

### Claude's Discretion
- HTTP request routing implementation (string matching vs trie vs regex)
- Whether to support HTTP/2 (probably not needed for pre-MVP)
- Batch read response format (multipart vs JSON with embedded base64 vs sequential binary)
- SSE reconnection protocol (Last-Event-ID header for resume)
- Whether /metrics and /health become part of the main HTTP server or stay on separate port

</decisions>

<code_context>
## Existing Code Insights

### MetricsCollector HTTP Pattern
The relay already has a working coroutine HTTP server in MetricsCollector:
- `accept_loop()` coroutine accepts TCP connections
- Reads HTTP headers until `\r\n\r\n`
- Parses first line for route matching
- Sends response with proper HTTP headers
- This pattern scales to a full HTTP server with more routes and body handling

### WsSession Message Flow (to replicate in HTTP)
Current flow: WS text frame → JSON parse → type check → filter → translate → UDS send → route response → translate back → JSON send
HTTP equivalent: HTTP request → route match → auth check → body parse → translate → UDS send → route response → translate back → HTTP response

### SSE Pattern
SSE is simpler than WebSocket: HTTP response with `Content-Type: text/event-stream`, then write `data: ...\n\n` lines as events occur. Connection stays open. Client auto-reconnects on disconnect. `Last-Event-ID` header enables resume.

</code_context>

<specifics>
## Specific Ideas

- Blob write/read should be raw binary pass-through — the relay doesn't need to understand blob contents, just forward to/from UDS. No JSON wrapping, no base64.
- Query operations keep JSON because the response data is small and structured.
- The HTTP server is NOT a general-purpose web server — it only needs to handle the chromatindb API routes. Minimal HTTP parsing.
- SSE notifications carry the same 77-byte notification payload as before, just JSON-formatted in the event stream.

</specifics>

<deferred>
## Deferred Ideas

- HTTP/2 support — not needed pre-MVP, HTTP/1.1 is sufficient
- WebSocket fallback for environments that block SSE — can add later if needed
- gRPC — too heavy for this use case
- GraphQL — inappropriate for binary blob operations

</deferred>
