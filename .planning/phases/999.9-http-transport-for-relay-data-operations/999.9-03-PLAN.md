    **relay/http/http_connection.h/cpp**: Per D-35: one coroutine per connection.

    `HttpConnection` is a class that takes a TLS/plain stream variant (same dual-mode pattern as WsSession), a reference to HttpRouter, TokenStore, and references to rate_limiter atomics.

    Use `using Stream = std::variant<asio::ip::tcp::socket, asio::ssl::stream<asio::ip::tcp::socket>>` (same as WsSession::Stream).

    The main coroutine `handle()` is a keep-alive request loop:
    1. Read bytes until `\r\n\r\n` found (header-end sentinel)
    2. Parse request line + headers using http_parser functions
    3. If Content-Length > 0, read exactly that many body bytes
    4. If Content-Length > max_body_size (e.g., 110 MiB), respond 413 immediately
    5. Call `router_.dispatch(request, *this)` to get HttpResponse
    6. Serialize and send response
    7. If keep_alive, loop; else close

    For TLS/plain dual-mode reads and writes, use `std::get_if<TlsStream>` branching (NOT std::visit -- per Phase 106 FIX-02 and RESEARCH Pitfall 7, ASAN stack-use-after-return).

    TLS handshake: after accept, if TLS configured, perform async_handshake before entering the request loop. 5s handshake timeout (same as WsAcceptor).

    **relay/http/http_server.cpp**: Accept loop follows MetricsCollector pattern:
    - Accept TCP socket
    - Check connection cap (reject with 503 if full)
    - If TLS: create ssl::stream, wrap in variant, spawn HttpConnection with TLS stream
    - If no TLS: spawn HttpConnection with plain socket
    - Increment active_connections_ on accept, decrement on coroutine exit (RAII guard)

    **relay/http/http_router.h**:
    ```cpp
    class HttpRouter {
    public:
        using Handler = std::function<HttpResponse(const HttpRequest&, HttpSessionState*)>;
        // Handler receives HttpSessionState* which is non-null for authenticated endpoints, null for public

        /// Register a route. public_route=true means no auth check.
        void add_route(std::string_view method, std::string_view prefix, Handler handler, bool public_route = false);

        /// Dispatch a request. Handles auth middleware for non-public routes.
        HttpResponse dispatch(const HttpRequest& req, TokenStore& token_store);

    private:
        struct Route {
            std::string method;
            std::string prefix;
            Handler handler;
            bool public_route;
        };
        std::vector<Route> routes_;  // Scanned linearly; ~25 routes, fast enough
    };
    ```

    Route matching: exact method match, then check if request path starts with route prefix. For exact routes like "/auth/challenge", the prefix IS the full path. For parameterized routes like "/blob/", the prefix is "/blob/" and the remaining path contains namespace/hash.

    If method matches but no prefix matches -> 404. If prefix matches but method doesn't -> 405.

    **Auth middleware** (inside dispatch): For non-public routes, extract `Authorization` header, parse "Bearer <token>", look up in TokenStore. If missing or invalid -> return 401. If valid -> pass HttpSessionState* to handler. Also apply rate limiter check per D-31: if rate_limiter.try_consume() fails, return 429.

    **Auth handlers** (registered in router setup, public_route=true):
    - `POST /auth/challenge`: Call `authenticator_.generate_challenge()`, store challenge in a pending map (nonce_hex -> challenge bytes). Return `{"nonce": "<hex>"}`.
    - `POST /auth/verify`: Parse JSON body `{"public_key": "<hex>", "signature": "<hex>"}`. Look up nonce (from a header or a "nonce" field in the body -- match the challenge). Call `authenticator_.verify()` with challenge, pubkey, signature (per D-06). On success: call `token_store.create_session()`, return `{"token": "<hex>", "namespace": "<hex>"}`. On failure: return 401 with error details.

    Challenge storage: `std::unordered_map<std::string, std::array<uint8_t, 32>> pending_challenges_` in the router or a dedicated auth handler object. Challenges expire after 30s (reap on each new challenge request).

    The auth flow is two HTTP requests (not an in-band WS exchange), per D-04. The verify request must include the nonce so the server knows which challenge to verify against. Use `"nonce"` field in the verify JSON body.

    **Health handler** (public_route=true): `GET /health` -- return same JSON as current MetricsCollector /health handler. This is public (no auth required), per FEAT-03.

    **Test file**: Test route matching (exact, prefix, 404, 405) and auth middleware (missing token -> 401, valid token -> passes through). Test auth handlers with mock authenticator if feasible, or at minimum test route dispatch.
