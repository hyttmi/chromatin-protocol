<threat_model>
Phase 123 Plan 03 threat surface is CLI logic (not on-the-wire — CLI stays pre-122). Mitigations:

- **T-123-05 (NAME squatting deterministic resolution):** If two writers race to write NAME blobs with the same name in the same namespace, the resolution logic in `cdb get` must pick the same winner regardless of which client enumerates first. Mitigated by D-01 (blob.timestamp DESC) + D-02 (content_hash DESC tiebreak). Both are strictly ordered on bytes — no client-local state influences the outcome. Tested: manual verification; the underlying engine-level test `test_name_overwrite.cpp` (Plan 02) already proves the ordering is strict. CLI side simply applies the ordering; any CLI bug shows up as test-failure or wrong-content-retrieved.
- **T-123-05b (CLI selects stale winner due to enumeration race):** `cdb get foo` enumerates via ListRequest; a concurrent `cdb put --name foo` might or might not appear in the snapshot. Accepted: "eventual" semantics — `cdb get` returns whatever was visible at enumeration time; a subsequent `cdb get` picks up the newer NAME. This is consistent with the eventually-consistent nature of the blob store; no coordination required.
- **No new on-the-wire surface** — CLI uses existing Phase 117 ListRequest + type_filter endpoint, existing Phase 122 BlobWrite envelope (node side; CLI side still pre-122 per scope constraint). Thus no new transport threats introduced by this plan.
</threat_model>

  <objective>Extend `cmd_put` to emit a NAME blob (and optionally a BOMB-of-1 for `--replace`) alongside the content blob in `cli/src/commands.{h,cpp}`.</objective>

  </read_first>
  <action>
    Extend `cmd_put` signature to accept an optional `std::string name` and `bool replace`. Implementation flow:

    0. **(if replace && name.has_value())** **Look up the prior NAME's target content_hash** via `cmd_get_by_name(name)` — store as `old_target_hash`. Do NOT emit anything yet. If no prior NAME exists, set `old_target_hash = nullopt` and warn "no prior binding to replace".
    1. **Content blob:** existing path — write the file as a normal content blob, capture its content_hash on response.
    2. **NAME blob (if name.has_value()):** construct NAME payload bytes: `[magic:4=NAME_MAGIC][name_len:2 BE][name bytes][content_hash:32]` — use `wire::make_name_data(name, content_hash)` from Plan 01. Build a BlobData (pre-122 shape: `{namespace_id, pubkey, signer, data=name_payload, ttl=0 or user-specified, timestamp=now}`) and submit it via the existing pipelining path. NAME timestamp MUST be equal to or greater than any prior NAME for the same name so this write wins resolution (D-01).
    3. **BOMB-of-1 (if replace && old_target_hash.has_value())** — AFTER steps 1+2 succeed. Construct BOMB payload: `[magic:4=BOMB_MAGIC][count:4 BE = 1][target_content_hash:32=old_target_hash]` using `wire::make_bomb_data({old_target_hash})`. Build + submit BOMB with ttl=0 (non-negotiable). If steps 1 or 2 errored, do NOT emit the BOMB — old content stays reachable by hash. This is the write-before-delete ordering fix from plan-checker iteration 1.

    **Critical:** use `wire::make_name_data` and `wire::make_bomb_data` from Plan 01 — do NOT hand-roll payload construction here. This enforces `feedback_no_duplicate_code.md`.

    **Submission ordering: content → NAME → BOMB** (write-before-delete). Rationale: if a partial failure occurs and BOMB hasn't been issued yet, the old content is still reachable by hash AND the new content is reachable by either hash or name — the worst case is a user re-runs `--replace` and the BOMB catches up. In the reverse ordering (BOMB first) a partial failure where BOMB propagates but content/NAME fail DELETES the old content with no new pointer, which is irrecoverable data loss. The plan-checker flagged this — adopt write-before-delete.
  </action>

  <action>
    **`cmd_get_by_name(const std::string& name)`:**
    1. Send `ListRequest(namespace=current_namespace, type_filter=NAME_MAGIC, limit=<reasonable cap, 1000>)` via existing ListRequest pipelining path.
    2. Parse response: list of `[hash, seq, type, size, timestamp]` entries for NAME-type blobs.
    3. For each candidate, send `ReadRequest(hash)` to get the full BlobData.
    4. For each BlobData, parse the NAME payload via `wire::parse_name_payload(data)`. Filter to entries whose `.name == requested_name`. 
    5. Sort filtered by `(blob.timestamp DESC, content_hash DESC)`. Pick the winner (first entry post-sort).
    6. If no winner → error "name not found", exit 1.
    7. Otherwise: send `ReadRequest(winner.target_content_hash)` and write the resulting content bytes to stdout (or to file if --out given).

    **`cmd_rm_batch(const std::vector<std::array<uint8_t, 32>>& target_hashes)`:**
    1. Construct BOMB payload: `[magic:4=BOMB_MAGIC][count:4 BE = N][target_hash:32 × N]` — use `wire::make_bomb_data(target_hashes)`.
    2. Build BlobData (pre-122 shape): `{namespace_id=current_ns, pubkey=our_pk, signer=our_id, data=bomb_payload, ttl=0, timestamp=now}`. Sign.
    3. Submit via existing pipelining path. One BOMB for the whole vector — per D-07 and plan `must_haves`.
    4. Report success (count of targets) or relay the server's error code verbatim (e.g. ERROR_BOMB_DELEGATE_NOT_ALLOWED if this client was configured as a delegate identity — but that's the server's call).

    Both functions reuse existing ListRequest / ReadRequest / write-blob pipelining helpers. No new transport plumbing.
