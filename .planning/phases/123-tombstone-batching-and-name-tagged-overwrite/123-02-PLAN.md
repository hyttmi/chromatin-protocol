<threat_model>
Phase 123 Plan 02 threat surface is the BOMB ingest path. Mitigations tested by plan tasks:

- **T-123-01 (Delegate BOMB attempt):** Delegate emits BOMB to delete owner's data. Mitigated by Step 2 delegate-reject extension (is_bomb added to existing `is_tombstone` / `is_delegation` list). Tested: test_bomb_validation.cpp `bomb_delegate_rejected`.
- **T-123-02 (Malformed BOMB flood):** Attacker sends BOMB with declared count=1000000 but data.size() small (would overflow on multiplication). Mitigated by Step 1.7 structural check: reject if `data.size() != 8 + count*32` OR if `count*32` overflows uint32. Check is cheap (integer arithmetic) and runs BEFORE crypto::offload, matching Pitfall #6 "adversarial flood" defense from RESEARCH. Tested: test_bomb_validation.cpp `bomb_malformed_rejected` + overflow boundary case.
- **T-123-03 (Expiring BOMB re-propagation):** BOMB with ttl>0 would let peers prune the BOMB while still holding the supposedly-deleted content. Non-negotiable per memory `project_phase123_bomb_ttl_zero.md`. Mitigated by Step 1.7 ttl check + Step 0e max_ttl exemption (BOMB is permanent, no "grace"). Tested: test_bomb_validation.cpp `bomb_ttl_nonzero_rejected`.
- **T-123-04 (Empty BOMB flood):** Attacker spams BOMB count=0. Accepted as no-op per Plan 01 A2 recommendation — the structural check passes, Step 3.5 iterates zero targets, storage unchanged. Cost to attacker: full PQ signature per empty BOMB (~2420 bytes signed). Economic ratio is unattractive vs just spamming random content blobs. Tested: test_bomb_validation.cpp `bomb_count_zero_noop`.
- **T-123-06 (Delegate NAME write in owner's namespace):** Permitted per D-11. Audit property: the NAME blob carries the delegate's `signer_hint`, so the owner can identify who named what via ListRequest + signer_hint inspection. Tested: test_name_delegate.cpp `delegate_name_accepted` documents the signer_hint preservation.
</threat_model>
