```bash
#!/usr/bin/env bash
# =============================================================================
# FILT-01/FILT-02: Namespace Filtering via SyncNamespaceAnnounce
#
# Verifies:
#   Phase 1: All 3 nodes connect and exchange SyncNamespaceAnnounce
#   Phase 2: Blob in NS_A is received by all 3 nodes (all replicate NS_A or all)
#   Phase 3: Blob in NS_B is received by Node2 and Node3 only (Node1 doesn't replicate NS_B)
#
# Topology: 3-node standalone
#   Node1 (172.31.0.2): sync_namespaces=[NS_A]

#   Node3 (172.31.0.4): sync_namespaces=[] (all)
# =============================================================================
```

**Phase 1: Setup and verify announce exchange**
1. Create network, start all 3 nodes
2. Wait for `"announced.*sync namespaces"` in each node's logs (confirms SyncNamespaceAnnounce exchange)
3. Verify Node1 log shows it announced 1 namespace
4. Verify Node2 log shows it announced 2 namespaces

**Phase 2: Ingest blob in NS_A, verify all 3 receive it**
1. Use loadgen to ingest a blob into Node1 with namespace NS_A
2. Wait for the blob to appear on Node2 (via sync or BlobNotify+BlobFetch)
3. Wait for the blob to appear on Node3
4. All 3 should have the blob (verified via SIGUSR1 metrics or log)

**Phase 3: Ingest blob in NS_B, verify only Node2 and Node3 receive it**
1. Use loadgen to ingest a blob into Node2 with namespace NS_B
2. Wait for the blob to appear on Node3
3. Verify Node1 does NOT receive the blob (check blob count via SIGUSR1 metrics -- should still be 1 from Phase 2)
4. This is the key FILT-02 validation: BlobNotify for NS_B was not sent to Node1

**Cleanup:** Remove containers, volumes, network on EXIT trap.

Use `FAILURES` counter pattern. Exit 1 if any phase fails.

Note: The loadgen tool needs to support specifying a namespace. If it doesn't, use the SDK or a direct write via the existing ingest mechanism. Check `tools/loadgen.cpp` for namespace support. If loadgen writes to a fixed namespace, adjust the test to use that namespace as NS_A and pick a different one for NS_B. Alternatively, use two different identity keys (different namespace_id = SHA3-256(pubkey)) to produce blobs in different namespaces.

**Important:** The namespace in chromatindb is SHA3-256(pubkey) of the blob signer. To write to a specific namespace, you need a key whose SHA3-256 hash matches. Since this is impractical, the test should instead:
- Start Node1 with identity key K1 (namespace = SHA3-256(K1.pub))
- Start Node2 with identity key K2 (namespace = SHA3-256(K2.pub))
- Configure sync_namespaces on each node using the actual namespace hashes
- OR: Use the simpler approach of empty sync_namespaces on writer nodes and checking log output for BlobNotify filtering

**Simpler approach (recommended):** Configure Node1 with sync_namespaces containing ONLY its own namespace hash. Configure Node2 with sync_namespaces containing both its own and Node1's namespace. Configure Node3 with empty sync_namespaces. Then:
1. Ingest via Node1 (blob goes to Node1's namespace) -- all nodes replicate this namespace
2. Ingest via Node2 (blob goes to Node2's namespace) -- only Node2 and Node3 replicate this

The key verification: After step 2, check Node1's blob_count (via SIGUSR1 metrics dump) to confirm it did NOT receive Node2's blob. Check Node3's blob_count to confirm it DID receive it.

**Getting namespace hashes:** Each node's namespace is SHA3-256(pubkey). The node logs its namespace at startup. Parse it from the startup log: `"Node namespace: <hex>"` or similar. Use this to construct the sync_namespaces config dynamically:
1. Start nodes with empty sync_namespaces first
2. Extract namespace hashes from logs
3. Stop nodes
4. Write configs with correct sync_namespaces
5. Restart nodes

This is the robust approach. Follow it.

**Test script** (tests/integration/test_ops01_max_peers_sighup.sh):

```bash
#!/usr/bin/env bash
# =============================================================================
# OPS-01: max_peers SIGHUP Hot Reload
#
# Verifies:
#   Phase 1: Node1 accepts 2 peers (max_peers=2), both connect
#   Phase 2: SIGHUP reduces max_peers to 1 -- existing 2 peers stay connected
#            (no mass disconnect), but warning logged about excess
#   Phase 3: If Node2 disconnects and reconnects, Node1 should accept it
#            (count dropped to 1 which equals max_peers=1). But if both are
#            still connected and a NEW peer tries to connect, it would be refused.
#   Phase 4: SIGHUP increases max_peers to 5 -- new connections accepted again
#
# Topology: 3-node standalone
#   Node1 (172.32.0.2): writable config, max_peers=2 initially
#   Node2 (172.32.0.3): bootstraps to Node1
#   Node3 (172.32.0.4): bootstraps to Node1
# =============================================================================
```

**Phase 1: Verify initial connection acceptance**
1. Create network `chromatindb-ops01mp-test-net` (172.32.0.0/16)
2. Start Node1 with max_peers=2 (writable config mount)
3. Start Node2 and Node3
4. Wait for both to connect to Node1 (verify `"Connected to peer"` appears twice in Node1 logs)
5. Verify Node1 has 2 peers (SIGUSR1 metrics dump)

**Phase 2: Reduce max_peers via SIGHUP**
1. Edit Node1's config file: change `"max_peers": 2` to `"max_peers": 1`
2. Send SIGHUP: `docker kill -s HUP <node1_container>`
3. Wait for `"excess will drain naturally"` in Node1 logs
4. Verify BOTH peers are still connected (Node1 still has 2 peers -- D-12: no mass disconnect)
5. This is the key D-12 validation

**Phase 3: Verify new connections are refused**
1. Stop Node3
2. Restart Node3 (it will try to reconnect to Node1)
3. Since Node1 has 1 peer (Node2) and max_peers=1, Node3's connection should be refused
4. Verify Node3 does NOT appear as connected in Node1 logs after the restart
   (Note: This step may be tricky since Node1 still has Node2. Check that Node1's peer count stays at 1 after Node3 tries to reconnect. The `should_accept_connection()` check fires before the handshake.)

**Alternative for Phase 3 (simpler):** After Phase 2, stop Node2 AND Node3. Then restart only Node2. Verify Node2 connects (count goes to 1, which is the limit). Then restart Node3. Verify Node3 cannot connect (Node1 already at max_peers=1).

**Phase 4: Increase max_peers via SIGHUP**
1. Edit Node1's config: change `"max_peers": 1` to `"max_peers": 5`
2. Send SIGHUP: `docker kill -s HUP <node1_container>`
3. Wait for `"config reload: max_peers=5"` in Node1 logs
4. Restart Node3 if needed -- verify it can now connect
5. Verify Node1 has 2 peers again

**Cleanup:** Remove containers, volumes, network on EXIT trap.
