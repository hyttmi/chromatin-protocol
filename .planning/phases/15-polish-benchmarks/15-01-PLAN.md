## Project Facts for README Content

- Binary name: `chromatindb`
- Commands: `run`, `keygen`, `version`
- Run options: `--config <path>`, `--data-dir <path>`, `--log-level <lvl>`
- Keygen options: `--data-dir <path>`, `--force`
- Default bind: 0.0.0.0:4200
- Default data dir: ./data
- Config keys: bind_address, data_dir, bootstrap_peers, allowed_keys, max_peers, sync_interval_seconds, log_level
- Protocol constants: BLOB_TTL = 7 days (604800s), not configurable
- Crypto: ML-DSA-87 (signing), ML-KEM-1024 (key exchange), ChaCha20-Poly1305 (transport encryption), SHA3-256 (hashing), HKDF-SHA256 (key derivation)
- Build: CMake + FetchContent (all deps fetched automatically)
- Dependencies: liboqs, libsodium, FlatBuffers, libmdbx, Standalone Asio, Catch2, spdlog, nlohmann/json
- v3.0 features: blob deletion via tombstones, namespace delegation, pub/sub notifications
- Identity: ML-DSA-87 keypair, namespace = SHA3-256(pubkey)
- Storage: libmdbx (ACID key-value store), content-addressed by SHA3-256 hash
- Sync: peer-to-peer hash-list diff protocol
- Transport: ML-KEM-1024 handshake + ChaCha20-Poly1305 AEAD encryption
- ACL: implicit closed mode from non-empty allowed_keys list

Create README.md at project root with the following sections:

1. **Header**: Project name (chromatindb), one-line description ("A decentralized, post-quantum secure database node with signed blob storage and peer-to-peer replication"), brief value proposition paragraph.

2. **Crypto Stack** (surface this prominently, not buried -- per user's specific idea):
   - ML-DSA-87: signs all blobs and tombstones (FIPS 204 post-quantum digital signature)
   - ML-KEM-1024: key exchange during peer handshake (FIPS 203 post-quantum KEM)
   - ChaCha20-Poly1305: encrypts all peer-to-peer traffic (AEAD, via libsodium)
   - SHA3-256: content-addressed blob hashing and namespace derivation
   - One paragraph explaining: "Every blob is cryptographically signed by its author. Peers establish encrypted channels using post-quantum key exchange. No data travels in plaintext."

3. **Architecture** (brief high-level paragraph):
   - Namespaces: each node identity derives a namespace via SHA3-256(pubkey). Blobs belong to namespaces.
   - Blobs: signed data units with TTL, replicated across nodes
   - Sync: hash-list diff protocol -- peers exchange what they have, request what they're missing
   - Transport: ML-KEM handshake establishes AEAD-encrypted channels

4. **Building**:
   - Prerequisites: C++20 compiler (GCC 12+ / Clang 15+), CMake 3.20+, Git
   - Build steps: mkdir build && cd build && cmake .. && make -j$(nproc)
   - Note: all dependencies are fetched automatically via CMake FetchContent (first build takes longer)
   - Test: cd build && ctest --output-on-failure

5. **Usage**:
   - `chromatindb keygen [--data-dir <path>] [--force]` -- generate identity keypair
   - `chromatindb run [--config <path>] [--data-dir <path>] [--log-level <lvl>]` -- start daemon
   - `chromatindb version` -- print version

6. **Configuration**: Example JSON config block with all supported keys and brief inline comments. No reference table.
   ```json
   {
     "bind_address": "0.0.0.0:4200",
     "data_dir": "./data",
     "bootstrap_peers": ["192.168.1.10:4200"],
     "allowed_keys": [],
     "max_peers": 32,
     "sync_interval_seconds": 60,
     "log_level": "info"
   }
   ```

7. **Scenarios**:
   - Single node: generate keys, start with default config, node accepts inbound connections
   - Two-node sync: Node A starts, Node B uses bootstrap_peers pointing to Node A, blobs replicate automatically
   - Closed mode with ACLs: populate allowed_keys with namespace hashes, only listed peers can connect, SIGHUP reloads config

8. **v3.0 Features** (brief descriptions, no detailed scenarios):
   - Blob Deletion: namespace owners can permanently delete blobs via signed tombstones
   - Namespace Delegation: owners can grant write access to other identities via signed delegation blobs
   - Pub/Sub Notifications: connected peers can subscribe to namespaces and receive real-time notifications when blobs are ingested or deleted

9. **Performance**: Section header with a note saying "See benchmark instructions below" (placeholder for plan 15-02 to fill in results).
