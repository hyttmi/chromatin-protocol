<objective>
Close the single gap blocking Phase 119's end-to-end goal: `Connection::recv()` consumes replies off the wire but never decrements `in_flight_`, so every arrival-order drain loop leaks pipeline slots. After 8 replies land, `send_async`'s backpressure loop calls `pump_one_for_backpressure`, which stashes replies into `pending_replies_` but also never decrements `in_flight_` — the client stalls forever. Observed in live-node testing: a 420 MiB upload hung after 46 minutes with exactly 8 chunks persisted and 0 WriteAcks flowing back.

The fix is reviewer option (b) from 119-REVIEW.md CR-01: add a `pipeline::pump_recv_any` helper (arrival-order counterpart of the existing `pump_recv_for`), wrap it in a new public `Connection::recv_next()` method, and migrate all 5 call sites (put_chunked / rm_chunked / get_chunked / cmd::put / cmd::get Phase-B drains) from `conn.recv()` to `conn.recv_next()`. `Connection::recv()` itself is preserved — it remains the low-level wire reader invoked by both pumps. The single-sender / single-reader invariant (PIPE-02, load-bearing per project memory) is preserved because `pump_recv_any` never touches `send_counter_` and does not introduce a thread or strand.

Same plan closes three co-located regressions the reviewer flagged and one project-memory rule violation:

1. **WR-02** — `get_chunked` retry backoff pre-increments `attempts[idx]` before indexing `RETRY_BACKOFF_MS`, so the first retry sleeps 1000 ms instead of the D-15-specified 250 ms (and `RETRY_BACKOFF_MS[0]` is dead code). Fix: post-index (match `put_chunked`).
2. **WR-03** — `get_chunked` closes the output fd and then runs `verify_plaintext_sha3`; a `std::bad_alloc` between the close and the verify would leave the partial file on disk, violating D-12 "unlink on any failure". Fix: introduce a TU-local `UnlinkGuard` RAII struct covering the whole `get_chunked` body; `release()` only on success.
3. **WR-04** — `cmd::put` / `cmd::get` single-blob drain loops have the same `in_flight_` leak; a `cdb put f1..f9` of small files would hang on file 9. Closed by the same migration as CR-01.
4. **IN-03** — `main.cpp` `catch (...)` silently swallows malformed `config.json`, violating the project-memory rule "Don't suppress errors with || true". Fix: narrow to `catch (const std::exception& e)`, print a warning to stderr, still fall back to defaults.

Append the following template to `cli/src/pipeline_pump.h` immediately after the existing `pump_recv_for` template (before the closing `} // namespace chromatindb::cli::pipeline`). Use the EXACT body below — no edits:

```cpp
// =============================================================================
// Pump one reply in arrival order (not correlated to a specific rid)
// =============================================================================
//
// Counterpart of pump_recv_for for callers that drain replies in arrival order
// (D-08 — chunked uploads/downloads, cmd::put batches, cmd::get batches).
//
// Fast path: if anything was stashed by a prior send_async backpressure cycle
// into `pending`, deliver one of those entries first (no wire I/O). Else call
// `source()` for the next wire reply.
//
// On ANY non-nullopt return: decrement `in_flight` by one (both fast-path and
// source-path represent one pipeline slot freeing). Guarded so `in_flight = 0`
// never underflows — defense-in-depth against the CR-01 class of bug where an
// errant caller could otherwise decrement past zero.
//
// On source error (nullopt) with empty `pending`: return nullopt and leave
// `in_flight` UNTOUCHED (matches pump_recv_for's error semantic).
//
// Single-sender invariant (PIPE-02): this template only invokes `source()`
// (which wraps recv()) — it never touches the send path, so send_counter_
// and AEAD nonce monotonicity are preserved by construction.
//
// Note on the WR-01 window (carried forward from 119-REVIEW.md, out of scope
// for this template): stale entries in `pending` left behind by a failed
// send_async + pump_one_for_backpressure exchange are not cleaned up here —
// in that narrow case the connection is already dead and the caller exits.
template <typename Source>
std::optional<DecodedTransport> pump_recv_any(
    Source&& source,
    std::unordered_map<uint32_t, DecodedTransport>& pending,
    std::size_t& in_flight) {
    // Fast path: deliver a stashed reply before touching the wire.
    if (!pending.empty()) {
        auto it = pending.begin();
        DecodedTransport msg = std::move(it->second);
        pending.erase(it);
        if (in_flight > 0) --in_flight;
        return msg;
    }

    auto msg = source();
    if (!msg) return std::nullopt;
    if (in_flight > 0) --in_flight;
    return msg;
}
```
