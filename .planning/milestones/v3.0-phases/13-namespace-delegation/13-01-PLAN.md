  <action>
    **Wire codec (db/wire/codec.h and codec.cpp):**

    Add delegation utilities following the tombstone pattern exactly:

    1. `DELEGATION_MAGIC` -- pick a 4-byte constant (e.g., `{0xDE, 0x1E, 0x6A, 0x7E}` -- "delegate" mnemonic). Must be distinct from TOMBSTONE_MAGIC.
    2. `DELEGATION_DATA_SIZE` = 4 + crypto::Signer::PUBLIC_KEY_SIZE (= 2596). Note: cannot use crypto::Signer::PUBLIC_KEY_SIZE directly in constexpr since it's in a different header. Use inline constexpr size_t DELEGATION_PUBKEY_SIZE = 2592 and DELEGATION_DATA_SIZE = 4 + DELEGATION_PUBKEY_SIZE = 2596.
    3. `is_delegation(span<const uint8_t> data)` -- checks size == DELEGATION_DATA_SIZE and magic prefix matches.
    4. `extract_delegate_pubkey(span<const uint8_t> data)` -- returns vector<uint8_t> of the pubkey bytes (data[4..4+2592]). Use vector, not span, since it needs to be returned by value (same lifetime pattern as extract_tombstone_target but variable-length).
    5. `make_delegation_data(span<const uint8_t> delegate_pubkey)` -- creates 2596-byte delegation data (magic + pubkey).

    **Storage delegation index (db/storage/storage.h and storage.cpp):**

    1. Add `delegation_map` to Storage::Impl (4th sub-database). The max_maps is already 4, so this uses the spare slot.
    2. Create delegation_map in Impl constructor: `delegation_map = txn.create_map("delegation");`
    3. Key format for delegation_map: `[namespace:32][delegate_pubkey_hash:32]` (64 bytes). The delegate_pubkey_hash is SHA3-256(delegate_pubkey) -- using the hash instead of the full 2592-byte pubkey keeps keys compact and fixed-size. Value: `[delegation_blob_hash:32]` (the content hash of the delegation blob itself, needed for tombstone cross-check).
    4. In `store_blob()`: After storing in blobs_map and before commit, check if the blob data is a delegation (`wire::is_delegation(blob.data)`). If so, extract the delegate pubkey via `wire::extract_delegate_pubkey(blob.data)`, compute `delegate_pk_hash = crypto::sha3_256(delegate_pubkey)`, compute the delegation blob hash (already computed as `hash` variable), and upsert into delegation_map with key `[namespace:32][delegate_pk_hash:32]` and value `[delegation_blob_hash:32]`.
    5. In `delete_blob_data()`: Before commit, check if the blob being deleted is a delegation. The blob is already decoded (variable `blob`). If `wire::is_delegation(blob.data)`, extract the delegate pubkey, compute delegate_pk_hash, build the delegation_map key, and erase it.
    6. Add public method `has_valid_delegation(span<const uint8_t, 32> namespace_id, span<const uint8_t> delegate_pubkey) -> bool`. Implementation: compute delegate_pk_hash = SHA3-256(delegate_pubkey), build key [namespace:32][delegate_pk_hash:32], do a read txn get from delegation_map. If found, return true. This is O(1) btree lookup, not a scan.

    **Engine (db/engine/engine.cpp):**

    No changes needed to ingest() for delegation blob creation. The existing ingest pipeline already handles this correctly:
    1. Step 0: size check -- delegation data is 2596 bytes, well under MAX_BLOB_DATA_SIZE
    2. Step 1: structural checks pass (valid pubkey, non-empty sig)
    3. Step 2: namespace ownership -- owner signs with their key, SHA3(pubkey) == namespace_id
    4. Step 3: signature verification passes
    5. Step 3.5: is_tombstone check -- delegation blobs are NOT tombstones, so they pass through to regular path. The tombstone-before-blob check applies (if somehow someone tombstoned this delegation blob before it arrived, it would be rejected -- which is correct revocation behavior).
    6. Step 4: store_blob succeeds, and the storage layer automatically populates delegation_map

    The key insight is that delegation blob creation flows through the existing ingest() pipeline with zero engine changes. The storage layer handles index population transparently.
