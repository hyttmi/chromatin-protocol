  <files>db/engine/engine.h, db/engine/engine.cpp, tests/engine/test_engine.cpp</files>
  <behavior>
    - Delegate writes accepted: a blob signed by a delegate key, targeting the owner's namespace, is accepted when a valid delegation exists
    - Delegate writes rejected: a blob signed by a key with no delegation in that namespace is rejected with IngestError::no_delegation
    - Delegate writes rejected after revocation: if the delegation blob has been tombstoned, delegate writes are rejected
    - Owner writes still work: namespace owner can still write directly (ownership check passes before delegation check)
    - delete_blob remains owner-only: a delegate attempting to delete_blob gets namespace_mismatch (not delegation bypass)
    - Re-delegation: after revocation, creating a new delegation blob and writing again succeeds
    - Delegation blobs themselves are still owner-signed (owner creates delegations, not delegates)
    - Delegate-written blobs persist after delegation revocation (no cascade delete)
  </behavior>
  <action>
    **Engine header (db/engine/engine.h):**

    Add `no_delegation` to IngestError enum:
    ```cpp
    enum class IngestError {
        namespace_mismatch,
        invalid_signature,
        malformed_blob,
        oversized_blob,
        storage_error,
        tombstoned,
        no_delegation  // Delegate write rejected: no valid delegation exists
    };
    ```

    **Engine ingest modification (db/engine/engine.cpp):**

    Modify Step 2 in `ingest()` to check delegation as alternative to ownership:

    ```cpp
    // Step 2: Namespace ownership OR delegation check
    auto derived_ns = crypto::sha3_256(blob.pubkey);
    bool is_owner = (derived_ns == blob.namespace_id);
    bool is_delegate = false;

    if (!is_owner) {
        // Not the owner -- check if this pubkey has a valid delegation
        is_delegate = storage_.has_valid_delegation(blob.namespace_id, blob.pubkey);
        if (!is_delegate) {
            spdlog::warn("Ingest rejected: no ownership or delegation for namespace {:02x}{:02x}...",
                         blob.namespace_id[0], blob.namespace_id[1]);
            return IngestResult::rejection(IngestError::no_delegation,
                "pubkey has no ownership or valid delegation for this namespace");
        }
    }
    ```

    The rest of ingest() remains unchanged. Key observations:
    - Step 3 (signature verification) already works for delegates because build_signing_input uses the namespace_id (owner's namespace) + data + ttl + timestamp, and the signature is verified against blob.pubkey (the delegate's pubkey). The delegate signs the canonical form with their own key.
    - Step 3.5 (tombstone handling) works correctly -- if a delegate submits a blob and a tombstone exists for it, it's rejected. Delegates cannot submit tombstone data because is_tombstone check would try to delete the target blob, but the tombstone itself would be signed by the delegate key, not the owner -- and tombstone handling in delete_blob is owner-only.
    - Step 4 (store) works as before.

    IMPORTANT: Delegation blobs themselves (data=make_delegation_data) must be owner-signed. A delegate cannot create delegations. The existing Step 2 handles this naturally: delegation blobs are stored via ingest(), and the owner signs them with their key, so is_owner=true. A delegate trying to create a delegation blob would need to sign it and target the owner's namespace, but their derived_ns would not match, so they'd fall through to the delegation check -- and they do have a delegation, but that's fine because the delegation blob they're trying to create is just data. Actually, we need to prevent delegates from creating delegation blobs. Wait -- this is actually fine. A delegate CAN write any data to the namespace, including data that looks like delegation format. But that data is signed by the delegate's key, not the owner's. The delegation_map index only gets populated when the blob passes through store_blob, and the delegation blob's namespace_id is the owner's namespace. The index key is [namespace:32][delegate_pk_hash:32]. If a delegate writes a "delegation blob" for another key, it would be stored and indexed -- but the delegation it grants would be in the DELEGATE's namespace (since namespace_id in the blob == owner's namespace, but... wait).

    Actually, let me reconsider. The delegate writes to the OWNER's namespace. The delegation blob stored by a delegate would have namespace_id = owner's namespace. The storage layer would index it as a delegation from owner's namespace to whatever pubkey is in the data. This could be a security issue.

    To prevent this: In the ingest() pipeline, after Step 2 determines is_delegate=true, add a check: if the blob data is a delegation (`wire::is_delegation(blob.data)`), reject with a new error or reuse malformed_blob. Delegates are write-only for regular data, not for creating delegations.

    ```cpp
    // Delegates cannot create delegation blobs (only owners can)
    if (is_delegate && wire::is_delegation(blob.data)) {
        spdlog::warn("Ingest rejected: delegates cannot create delegation blobs");
        return IngestResult::rejection(IngestError::no_delegation,
            "delegates cannot create delegation blobs");
    }
    ```

    Similarly, prevent delegates from creating tombstone data:
    ```cpp
    // Delegates cannot create tombstone blobs (deletion is owner-privileged)
    if (is_delegate && wire::is_tombstone(blob.data)) {
        spdlog::warn("Ingest rejected: delegates cannot create tombstone blobs");
        return IngestResult::rejection(IngestError::no_delegation,
            "delegates cannot create tombstone blobs");
    }
    ```

    Place these checks right after the delegation verification, before Step 3 (signature check), to fail fast.

    **Engine delete_blob (db/engine/engine.cpp):**

    No changes needed. delete_blob() already has a strict ownership check at Step 2:
    ```cpp
    auto derived_ns = crypto::sha3_256(delete_request.pubkey);
    if (derived_ns != delete_request.namespace_id) {
        return IngestResult::rejection(IngestError::namespace_mismatch, ...);
    }
    ```
    This does NOT check delegation -- delegates cannot delete. This is by design per CONTEXT.md: "Delegates write-only (no delete). Deletion is owner-privileged."

    1. **Delegate write accepted**: Create owner identity, delegate identity. Owner creates delegation blob via ingest. Delegate writes a regular blob to owner's namespace (namespace_id = SHA3(owner_pubkey), signed by delegate key). Verify ingest succeeds.

    2. **Delegate write rejected (no delegation)**: Delegate writes to owner's namespace without delegation. Verify IngestError::no_delegation.

    7. **Re-delegation after revocation**: After revocation, owner creates new delegation blob for same delegate. Delegate writes again. Verify ingest succeeds.

    10. **Multiple delegates**: Owner delegates to two different keys. Both can write. Revoke one. First is rejected, second still works.
