    - `IngestResult delete_blob(const wire::BlobData& delete_request)` -- validates the delete request:
      1. Structural checks: pubkey size, signature non-empty (same as ingest Step 1)
      2. Namespace ownership: SHA3-256(pubkey) == namespace_id (same as ingest Step 2)
      3. Signature verification: sign(namespace_id || blob_hash_from_request || ttl=0 || timestamp). The `data` field of the request IS the 32-byte target blob hash.         d. Calls storage_.store_blob() with the tombstone BlobData (gets content-addressed hash, seq_num)
        e. Returns IngestResult::success with the tombstone's blob_hash

    **DELETE message wire format:**
    The DELETE message payload is a FlatBuffer-encoded BlobData where:
    - namespace_id: the target namespace (32 bytes)
    - pubkey: owner's ML-DSA-87 public key
    - data: the 32-byte target blob hash (raw, NOT tombstone-encoded -- engine handles that)
    - ttl: 0
    - timestamp: client's unix timestamp
    - signature: ML-DSA-87 signature over canonical form SHA3-256(namespace || data || 0 || timestamp)

    **DELETE_ACK message wire format:**
    The DeleteAck payload is a FlatBuffer-encoded BlobData containing the TOMBSTONE (not the original delete request). This mirrors WriteAck pattern -- the ack contains the stored artifact. Alternatively, for simplicity, encode just the 32-byte tombstone blob_hash + 8-byte seq_num + 1-byte status as raw bytes (same as how other ack-like messages work). Use the simpler raw format:
    - [blob_hash:32][seq_num_be:8][status:1] where status: 0=stored, 1=duplicate

    Add to PeerManager (or a new encode/decode helper):
