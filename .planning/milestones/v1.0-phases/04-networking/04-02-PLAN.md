  <action>
    **src/net/handshake.h:** Define the session key types and derivation function:

    ```cpp
    namespace chromatin::net {

    struct SessionKeys {
        crypto::SecureBytes send_key;   // 32 bytes, for encrypting outbound frames
        crypto::SecureBytes recv_key;   // 32 bytes, for decrypting inbound frames
        std::array<uint8_t, 32> session_fingerprint;  // for mutual auth signing
    };

    /// Derive directional session keys from ML-KEM shared secret.
    /// is_initiator determines which direction maps to send vs recv.
    /// Initiator's send_key = expand(prk, "chromatin-init-to-resp-v1", 32)
    /// Responder's send_key = expand(prk, "chromatin-resp-to-init-v1", 32)
    SessionKeys derive_session_keys(
        std::span<const uint8_t> shared_secret,
        std::span<const uint8_t> initiator_signing_pubkey,
        std::span<const uint8_t> responder_signing_pubkey,
        bool is_initiator);

    } // namespace chromatin::net
    ```

    **Implementation (handshake.cpp):**
    1. HKDF extract: `prk = KDF::extract(/*salt=empty*/{}, shared_secret)` -- empty salt per RFC 5869 when no salt available
    2. Derive init->resp key: `KDF::expand(prk, "chromatin-init-to-resp-v1", 32)`
    3. Derive resp->init key: `KDF::expand(prk, "chromatin-resp-to-init-v1", 32)`
    4. Compute fingerprint: concatenate shared_secret + initiator_signing_pubkey + responder_signing_pubkey, then SHA3-256 hash
    5. If is_initiator: send_key = init->resp, recv_key = resp->init
    6. If responder: send_key = resp->init, recv_key = init->resp

    Write TDD tests FIRST. Test directional symmetry by deriving keys for both sides and verifying cross-match.

  <files>

  <behavior>
    Handshake happy path (in-memory, no sockets):
    - Initiator generates ephemeral KEM keypair and produces KemPubkey message bytes
    - Responder receives KemPubkey, encapsulates, produces KemCiphertext message bytes + derives session keys
    - Initiator receives KemCiphertext, decapsulates, derives session keys (matching responder's)
    - Initiator signs session fingerprint, sends AuthSignature (encrypted with session key)
    - Responder receives AuthSignature, decrypts, verifies signature against initiator's signing pubkey
    - Responder signs session fingerprint, sends AuthSignature (encrypted with session key)
    - Initiator receives AuthSignature, decrypts, verifies signature against responder's signing pubkey
    - Both sides now have authenticated SessionKeys

    Auth failure cases:
    - Invalid ML-DSA-87 signature from initiator -> responder rejects (HandshakeError::AuthFailed)
    - Invalid ML-DSA-87 signature from responder -> initiator rejects (HandshakeError::AuthFailed)
    - Corrupt KEM ciphertext -> initiator gets different shared secret -> auth fails on fingerprint mismatch

    Edge cases:
    - Handshake with self (same identity on both sides) succeeds (valid use case for testing)
  </behavior>
  <action>
    **Add to src/net/handshake.h:**

    ```cpp
    namespace chromatin::net {

    enum class HandshakeError {
        Success,
        KemFailed,
        AuthFailed,
        DecryptFailed,
        InvalidMessage,
    };

    /// Handshake state machine for in-memory testing.
    /// In production, the transport layer drives this by sending/receiving bytes over TCP.
    /// This class handles the crypto logic; the caller handles IO.

    class HandshakeInitiator {
    public:
        explicit HandshakeInitiator(const identity::NodeIdentity& identity);

        /// Step 1: Generate ephemeral KEM keypair, return KemPubkey message bytes (unencrypted).
        std::vector<uint8_t> start();

        /// Step 2: Receive KemCiphertext from responder, derive session keys.
        /// Returns error if ciphertext is invalid.
        HandshakeError receive_kem_ciphertext(std::span<const uint8_t> kem_ct_msg);

        /// Step 3: Produce encrypted AuthSignature message (signing pubkey + signature over fingerprint).
        /// Must be called after receive_kem_ciphertext succeeds.
        std::vector<uint8_t> create_auth_message();

        /// Step 4: Receive and verify responder's encrypted AuthSignature.
        /// Returns the responder's signing public key on success.
        std::pair<HandshakeError, std::vector<uint8_t>> verify_peer_auth(std::span<const uint8_t> encrypted_auth_msg);

        /// Get the established session keys (valid after successful handshake).
        SessionKeys&& take_session_keys();

    private:
        const identity::NodeIdentity& identity_;
        crypto::KEM kem_;
        SessionKeys keys_;
        bool keys_derived_ = false;
        uint64_t send_counter_ = 0;
        uint64_t recv_counter_ = 0;
    };

    class HandshakeResponder {
    public:
        explicit HandshakeResponder(const identity::NodeIdentity& identity);

        /// Step 1: Receive KemPubkey from initiator, encapsulate, return KemCiphertext message bytes.
        std::pair<HandshakeError, std::vector<uint8_t>> receive_kem_pubkey(std::span<const uint8_t> kem_pk_msg);

        /// Step 2: Receive and verify initiator's encrypted AuthSignature.
        std::pair<HandshakeError, std::vector<uint8_t>> verify_peer_auth(std::span<const uint8_t> encrypted_auth_msg);

        /// Step 3: Produce encrypted AuthSignature message.
        std::vector<uint8_t> create_auth_message();

        /// Get the established session keys.
        SessionKeys&& take_session_keys();

    private:
        const identity::NodeIdentity& identity_;
        SessionKeys keys_;
        bool keys_derived_ = false;
        uint64_t send_counter_ = 0;
        uint64_t recv_counter_ = 0;
    };

    } // namespace chromatin::net
    ```

    **Implementation notes:**
    - KEM pubkey exchange happens UNENCRYPTED (no session key yet). Use TransportCodec to encode KemPubkey/KemCiphertext messages. Wrap in plain framing (no AEAD) OR send raw since it's the first exchange. Simplest: send raw FlatBuffer bytes with length prefix but no encryption for KEM exchange steps. After keys are derived, auth messages are encrypted.
    - For auth messages: encode signing pubkey + signature into a single payload. Format: [2592-byte signing pubkey][variable-length ML-DSA-87 signature]. Use TransportCodec with AuthSignature type.
    - Auth messages MUST be encrypted with the newly derived session keys. This proves both sides derived the same key.
    - The send_counter/recv_counter track nonces for the encrypted auth phase. Start at 0. Initiator sends first auth (counter 0), responder sends second (counter 0 on their send direction).
    - On auth failure: return HandshakeError, do NOT throw. The caller (connection manager) will close the TCP socket silently.

    **For testing without sockets:** Create a helper function or test that runs the full 4-step handshake by passing byte buffers between HandshakeInitiator and HandshakeResponder directly. This simulates the TCP exchange in memory.

    Write TDD tests FIRST. Tests should cover: happy path, auth failure with wrong identity, corrupt ciphertext.
