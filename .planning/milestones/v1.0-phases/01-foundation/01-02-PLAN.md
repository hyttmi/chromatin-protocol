  <action>
    **schemas/blob.fbs:** Create the FlatBuffers schema for the blob wire format:

    ```flatbuffers
    namespace chromatin.wire;

    table Blob {
      namespace_id:[ubyte];   // 32 bytes: SHA3-256(pubkey)
      pubkey:[ubyte];         // 2592 bytes: ML-DSA-87 public key
      data:[ubyte];           // variable length: application data
      ttl:uint32;             // seconds until expiry, 0 = permanent
      timestamp:uint64;       // writer's unix timestamp
      signature:[ubyte];      // up to 4627 bytes: ML-DSA-87 signature
    }

    root_type Blob;
    ```

    Use `[ubyte]` vectors (not fixed-size arrays) for all byte fields -- FlatBuffers fixed arrays are structs stored inline, vectors are more flexible and standard for byte data.

    **CMakeLists.txt updates:** Add a custom command to run `flatc --cpp --gen-object-api -o ${CMAKE_CURRENT_SOURCE_DIR}/src/wire/ schemas/blob.fbs` at build time to generate `blob_generated.h`. The `--gen-object-api` flag enables the NativeTable (Object API) for convenient C++ access. Ensure the flatc binary from the FetchContent build is used.

    Verify: `flatc --cpp schemas/blob.fbs` produces valid `blob_generated.h`.

  <behavior>
    Serialization round-trip:
    - encode_blob with all fields -> decode_blob recovers identical field values
    - namespace_id is exactly 32 bytes in encoded blob
    - pubkey field preserves all 2592 bytes
    - data field preserves arbitrary byte content (including empty)
    - ttl and timestamp round-trip exactly
    - signature field preserves all bytes

    Determinism/canonicality:
    - encode_blob called twice with same input produces byte-identical output
    - encode_blob -> decode_blob -> encode_blob produces byte-identical output (roundtrip canonicality)
    - ForceDefaults ensures zero-value ttl is still encoded

    Canonical signing input:
    - build_signing_input(namespace, data, ttl, timestamp) produces: namespace(32B) || data(varB) || ttl_le(4B) || timestamp_le(8B)
    - ttl is encoded as little-endian uint32
    - timestamp is encoded as little-endian uint64
    - Signing input is independent of FlatBuffer encoding (different from encoded blob bytes)
    - SHA3-256 of signing input is deterministic for same logical content

    Blob hashing:
    - blob_hash(encoded_blob) produces SHA3-256 of the full encoded FlatBuffer bytes
    - Different signatures on same data produce different blob hashes
    - Same encoded blob always produces same hash
  </behavior>
  <action>
    **codec.h:** Define in `chromatin::wire` namespace:
    ```cpp
    struct BlobData {
        std::array<uint8_t, 32> namespace_id;
        std::vector<uint8_t> pubkey;
        std::vector<uint8_t> data;
        uint32_t ttl;
        uint64_t timestamp;
        std::vector<uint8_t> signature;
    };

    // Serialize BlobData to FlatBuffer bytes
    std::vector<uint8_t> encode_blob(const BlobData& blob);

    // Deserialize FlatBuffer bytes to BlobData
    BlobData decode_blob(std::span<const uint8_t> buffer);

    // Build canonical signing input: namespace || data || ttl_le || timestamp_le
    // This is what gets hashed then signed -- independent of FlatBuffer format
    std::vector<uint8_t> build_signing_input(
        std::span<const uint8_t, 32> namespace_id,
        std::span<const uint8_t> data,
        uint32_t ttl,
        uint64_t timestamp);

    // SHA3-256 hash of the full encoded FlatBuffer blob (for content addressing)
    std::array<uint8_t, 32> blob_hash(std::span<const uint8_t> encoded_blob);
    ```

    **codec.cpp:** Implement using:
    - `flatbuffers::FlatBufferBuilder` with `ForceDefaults(true)` for deterministic encoding
    - Always use `CreateBlob()` generated function (not manual add_* calls) for field ordering consistency
    - `build_signing_input`: concatenate raw bytes in fixed order, use `std::memcpy` for integer little-endian encoding (C++20 `std::bit_cast` or manual byte extraction)
    - `blob_hash`: call `chromatin::crypto::sha3_256()` on the full encoded buffer

    **IMPORTANT:** The signing input does NOT include the signature field and does NOT include the pubkey field -- it is `namespace_id(32) || data(var) || ttl_le(4) || timestamp_le(8)`. The blob hash DOES cover the entire encoded blob including signature.

    Write TDD tests FIRST. Use Catch2 with `[codec]` tag.
