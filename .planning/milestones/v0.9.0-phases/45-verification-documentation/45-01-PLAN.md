  <action>
Create a standalone bash script at deploy/test-crash-recovery.sh following the established run-benchmark.sh pattern. The script validates libmdbx ACID crash recovery via Docker kill-9 scenarios.

**Script structure:**
1. Shebang, set -euo pipefail, SCRIPT_DIR/REPO_ROOT derivation (same as run-benchmark.sh)
2. Configuration variables: BLOB_COUNT (e.g., 50), SYNC_INTERVAL (10s for fast test), COMPOSE/NETWORK vars matching run-benchmark.sh
3. check_deps() for docker, docker compose, jq
4. build_image() with --skip-build flag support (reuse pattern from run-benchmark.sh)
5. wait_for_health() function polling docker inspect --format '{{.State.Health.Status}}'
6. cleanup() function with trap for compose down on exit

**Scenario A -- Kill during idle:**
1. Compose up (2-node topology from existing docker-compose.yml)
2. Wait for both nodes healthy
3. Ingest BLOB_COUNT blobs into node1 via loadgen (same pattern as run-benchmark.sh)
4. Wait for sync to complete (poll node2 logs for "sync complete" or similar, or wait sync_interval + buffer)
5. Send SIGUSR1 to node2 to dump metrics, capture blobs_ingested count as PRE_COUNT
6. `docker kill --signal=KILL chromatindb-node2` (kill-9 equivalent)
7. `docker start chromatindb-node2`
8. wait_for_health node2
9. Four integrity checks:
   a. Send SIGUSR1 again, verify blobs_ingested >= PRE_COUNT (data intact)
   b. Check startup logs for "integrity_scan" with no error-level messages (clean startup)
   c. Verify no "stale reader" warnings in startup logs
   d. Trigger another sync round, verify reconciliation finds zero or near-zero differences (cursor intact -- inferred from logs showing no blob transfers)
10. Print PASS/FAIL for Scenario A

**Scenario B -- Kill during active sync:**
1. Compose down + up fresh (clean state)
2. Wait for both nodes healthy
3. Start loadgen on node1 with a larger batch (e.g., 100 blobs)
4. While loadgen running, poll node2 logs for sync activity (grep for "ReconcileInit" or "sync_request")
5. As soon as sync activity detected, `docker kill --signal=KILL chromatindb-node2`
6. `docker start chromatindb-node2`
7. wait_for_health node2
8. Same 4 integrity checks as Scenario A (with the understanding that some in-flight blobs may not have committed)
9. Print PASS/FAIL for Scenario B

**Output:** Clear PASS/FAIL summary per scenario, exit code 0 on all pass, 1 on any failure.

**Important anti-patterns to avoid:**
- Do NOT use fixed `sleep` for sync timing -- use log polling or health checks
- Do NOT hardcode container names if docker-compose.yml uses different names -- derive from compose project name
- Set sync_interval_seconds to a short value (5-10s) via config for the test topology, or use the existing configs in deploy/configs/ if they already have appropriate values
