namespace chromatindb::sync {

using Hash32 = std::array<uint8_t, 32>;
constexpr uint8_t RECONCILE_VERSION = 0x01;
constexpr size_t SPLIT_THRESHOLD = 16;
constexpr size_t MAX_RECONCILE_ROUNDS = 64;

enum class RangeMode : uint8_t { Skip = 0, Fingerprint = 1, ItemList = 2 };

struct RangeEntry {
    Hash32 upper_bound;
    RangeMode mode;
    uint32_t count = 0;
    Hash32 fingerprint{};
    std::vector<Hash32> items;
};

struct ReconcileInit {
    uint8_t version;
    Hash32 namespace_id;
    uint32_t count;
    Hash32 fingerprint;
};

struct ReconcileResult {
    std::vector<RangeEntry> response_ranges;
    std::vector<Hash32> have_items;   // Items from received ItemList ranges
    bool complete = false;            // True if no Fingerprint ranges remain
};

Hash32 xor_fingerprint(const std::vector<Hash32>& sorted, size_t begin, size_t end);
std::pair<size_t, size_t> range_indices(const std::vector<Hash32>& sorted,
                                        const Hash32& lower, const Hash32& upper);
ReconcileResult process_ranges(const std::vector<Hash32>& our_sorted,
                                const std::vector<RangeEntry>& received_ranges);

std::vector<uint8_t> encode_reconcile_init(const ReconcileInit& init);
std::optional<ReconcileInit> decode_reconcile_init(std::span<const uint8_t> payload);
std::vector<uint8_t> encode_reconcile_ranges(std::span<const uint8_t, 32> ns,
                                              const std::vector<RangeEntry>& ranges);
struct DecodedRanges { Hash32 namespace_id; std::vector<RangeEntry> ranges; };
std::optional<DecodedRanges> decode_reconcile_ranges(std::span<const uint8_t> payload);
std::vector<uint8_t> encode_reconcile_items(std::span<const uint8_t, 32> ns,
                                             const std::vector<Hash32>& items);
struct DecodedItems { Hash32 namespace_id; std::vector<Hash32> items; };
std::optional<DecodedItems> decode_reconcile_items(std::span<const uint8_t> payload);
}

    **5. Key implementation details:**
    - Snapshot hash vectors ONCE per namespace at reconciliation start (Pitfall 6 from research). Do not re-read from storage mid-reconciliation.
    - The responder also needs to build its own diff (bidirectional sync). When the responder processes ItemList ranges from the initiator, those items are what the initiator has. The responder can diff against its own set to find what it's missing.
    - Similarly, the initiator collects ItemList ranges from the responder to find what it's missing.
    - After Phase B reconciliation completes for all namespaces, BOTH sides have a list of missing hashes per namespace. Phase C handles the actual blob transfer bidirectionally (existing interleaved BlobRequest/BlobTransfer pattern).
    - Remove all `encode_hash_list` / `decode_hash_list` calls that were part of the old Phase B hash list sending. Keep the `encode_blob_request` / `decode_blob_request` calls in Phase C.
    - Remove the old "collect peer hash lists into map" code and "diff_hashes" calls.

    **6. Handle edge case: namespace exists on peer but not on us:**
    - Our hash vector is empty. XOR fingerprint = all zeros, count = 0.
    - Reconciliation will correctly identify all of peer's items as missing.
    - No special handling needed -- the algorithm handles this naturally.

    **7. Handle edge case: namespace exists on us but not on peer:**
    - Peer won't send ReconcileInit for namespaces they don't have.
    - But we want the peer to get our blobs too. The initiator should also reconcile namespaces that WE have but the PEER doesn't (peer has no hashes -> all our items are the diff the peer needs).
    - Implementation: after processing peer's namespaces, also iterate our namespaces that aren't in peer's list. For those, the peer has nothing, so all our hashes are the diff. Skip reconciliation for these -- just send all hashes directly as ReconcileItems (or the peer can request via BlobRequest after we advertise in NamespaceList).
    - Actually: the current protocol already handles this -- both sides send NamespaceList in Phase A. The diff computation already considers both sides' namespaces. But with reconciliation replacing hash lists, we need to handle the case where a namespace exists on one side only.
    - Simplest approach: the initiator drives reconciliation for ALL namespaces that appear in EITHER side's namespace list (union). For namespaces the peer doesn't have, send ReconcileInit with our fingerprint. The peer responds with empty set (count=0, fingerprint=zeros). This naturally produces the full diff.
    - For namespaces WE don't have but the PEER has: we still send ReconcileInit with count=0, fingerprint=zeros. The peer responds with their items. We then request those blobs in Phase C.

    **8. Collect missing hashes per namespace during reconciliation, then run Phase C on the collected diffs:**
    Store results in `std::map<Hash32, std::vector<Hash32>> missing_from_peer` (what we need from them) and let the peer's reconciliation results tell them what they need from us. The peer sends BlobRequests for what they're missing during Phase C.
