  <action>
Create `tests/integration/fuzz/` directory with two files:

**Dockerfile** -- minimal `python:3-slim` image that copies fuzzer.py and sets it as entrypoint. No pip installs needed (only stdlib: socket, struct, random, sys, argparse, time).

**fuzzer.py** -- self-contained Python script using raw TCP sockets. Takes CLI args:
- `--target HOST:PORT` (required)
- `--mode protocol|handshake` (required)
- `--timeout SECONDS` (default 2, per-connection timeout)

**Mode: protocol** (for SAN-04)
Opens a TCP connection per payload category, sends the payload, reads any response (with timeout), closes. Payload categories:

Core protocol:
1. Truncated length prefix (1 byte instead of 4)
2. Oversized length prefix (0xFFFFFFFF followed by 16 bytes)
3. Zero-length frame (4 bytes of 0x00)
4. Pure garbage bytes (256 random bytes)
5. Valid 4-byte length prefix + garbage payload (length says 100, send 100 random bytes)
6. Truncated FlatBuffer mid-field (valid length prefix, 50 bytes of partial FlatBuffer-like data starting with valid root offset but truncated)

Crypto-specific:
7. Invalid AEAD tag (24-byte nonce + 64 bytes data + 16 bytes of 0xFF as tag)
8. Wrong nonce length (12 bytes instead of 24, followed by 64 bytes)
9. Truncated ciphertext (24-byte nonce + 8 bytes, then close)
10. Replay modified bytes (connect, capture first server response bytes if any, replay with 1 bit flipped)

Semantic:
11. Valid FlatBuffer-ish structure with impossible message type enum (type byte = 0xFF at expected offset)
12. Valid structure with negative-looking sizes (0xFFFFFFFF as a size field)
13. Future version byte (version = 0xFF at offset 0)

Each payload: open fresh TCP socket, send payload, wait up to `--timeout` for response/disconnect, close. Log per-payload result: "PAYLOAD N (name): sent X bytes, result: [disconnected|timeout|response(Y bytes)]".

Exit code 0 if all payloads sent successfully (regardless of server response). The test script checks that the node survived.

**Mode: handshake** (for SAN-05)
The PQ handshake has stages. For each stage, open a fresh connection and send a malformed message at that point:

Stage 1 -- Malformed ClientHello:
1a. Truncated hello (send 4 bytes length + 10 bytes, close)
1b. Oversized hello (length prefix claims 10MB, send 100 bytes, close)
1c. Wrong magic/version byte at offset 0

Stage 2 -- Malformed ServerHello/KEM ciphertext:
2a. Send a valid-length ClientHello-sized frame (craft 200 random bytes with valid length prefix), then when server responds, send garbage as the "KEM ciphertext response"
2b. Send valid-length ClientHello-sized frame, then truncated ciphertext (50 bytes)

Stage 3 -- Malformed session confirmation:
3a. Send two valid-length frames (simulating ClientHello and KEM response), then garbage as session confirmation
3b. Send two valid-length frames, then zero bytes (just close)

Each stage: open socket, send malformed data at that stage, verify server disconnects (recv returns 0 or connection reset). Log result.

Exit code 0 if all stages completed (sent payloads). The test script verifies the node is still alive.

Key implementation notes:
- Use `socket.settimeout(timeout)` for non-blocking reads
- Catch `ConnectionRefusedError`, `ConnectionResetError`, `BrokenPipeError` as expected outcomes
- Use `struct.pack('>I', length)` for 4-byte big-endian length prefixes (chromatindb uses 4-byte length-prefixed framing)
- Print clear pass/fail per payload to stdout for test script parsing
- The script does NOT need to speak valid chromatindb protocol -- the whole point is that it sends invalid data
