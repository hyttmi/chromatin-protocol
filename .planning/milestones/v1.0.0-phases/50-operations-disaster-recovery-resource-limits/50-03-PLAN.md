Config fields (from db/README.md):
```json
{
  "rate_limit_bytes_per_sec": 1048576,
  "rate_limit_burst": 5242880,
  "sync_cooldown_seconds": 30,
  "max_sync_sessions": 1
}
```

Protocol messages:
- SyncRejected (type 30): 1-byte reason code, sent when sync rate exceeded or session limit hit
- Write rate limiting: per-connection token bucket on Data(8) and Delete(18) messages
- Peers exceeding rate_limit_bytes_per_sec are disconnected immediately (no rejection message)

Metrics fields (from db/peer/peer_manager.cpp):
```
metrics: ... rate_limited={} ... sync_rejections={} ...
```

Log patterns:
- Rate limit disconnect: "rate limit" or "rate_limited" in logs
- Sync rejection: "SyncRejected" or "sync_rejections" in logs/metrics
</interfaces>

  <action>
Create test_dos01_write_rate_limiting.sh:
- Topology: 3-node standalone (docker run). Dedicated network (172.36.0.0/16, chromatindb-dos01-test-net). Named volumes.
- Node1 (target): rate_limit_bytes_per_sec=10240 (10KB/s), rate_limit_burst=20480 (20KB burst). Writable config.
- Node2 (good peer): bootstraps to Node1, sends data at acceptable rate.
- Node3 (flooding peer): bootstraps to Node1, will flood.
- Step 1: Start all 3 nodes. Wait healthy. Let connections establish (sleep 10).
- Step 2: Ingest 10 blobs (--size 1024 --rate 5) from loadgen to Node2 (within rate limit). Verify Node2 still connected after ingest (peer count on Node1 includes Node2).
- Step 3: Run flooding loadgen targeting Node1 from network position (via run_loadgen to Node1 IP). Use --size 10240 --rate 100 --count 200 (1MB/s, far exceeds 10KB/s limit). The flooding loadgen will get disconnected partway through -- that is expected (loadgen may error out).
- Step 4: Verify via Node1 SIGUSR1 metrics: rate_limited > 0. Verify Node1 logs contain rate limit disconnect evidence.
- Step 5: Verify Node2 is still connected (peer count >= 1 on Node1). Ingest 5 more blobs from Node2 side to prove Node2's connection survived the flood. This proves flooding peer was disconnected while good peer continued.

Create test_dos02_sync_rate_limiting.sh:
- Topology: 2-node standalone. Dedicated network (172.37.0.0/16, chromatindb-dos02-test-net). Named volumes.
- Node1 (target): sync_cooldown_seconds=30 (high cooldown to force rejections).
- Node2: bootstraps to Node1.
- Step 1: Start both nodes. Wait healthy. Let initial sync complete.
- Step 2: Ingest 20 blobs to Node1. Wait for sync to Node2. This consumes the first sync window.
- Step 3: Immediately ingest 10 more blobs to Node1. Node2 will try to sync again but should hit sync_cooldown_seconds=30. Check Node1 logs for "SyncRejected" or sync rejection evidence.
- Step 4: Verify via SIGUSR1 on Node1: sync_rejections > 0.
- Step 5: Wait 35s (past cooldown). Ingest 5 more blobs. Verify Node2 eventually receives all blobs (sync resumes after cooldown). Final blob count on Node2 should match Node1.
- Note: The sync_cooldown applies to INCOMING sync requests from a peer. Node2 initiates sync on its sync_interval timer. With cooldown=30 and sync_interval=5, Node2 will attempt sync every 5s but Node1 will reject attempts within the 30s cooldown window.
  </action>
  <verify>
    <automated>bash tests/integration/run-integration.sh --skip-build --filter dos01 && bash tests/integration/run-integration.sh --skip-build --filter dos02</automated>
  </verify>
  <done>DOS-01: Flooding peer disconnected (rate_limited > 0 in metrics), good peer continues normally. DOS-02: Excess sync initiations rejected (sync_rejections > 0), sync resumes after cooldown.</done>
</task>

Create test_dos03_concurrent_sessions.sh:
- Topology: 4-node standalone (docker run). Dedicated network (172.38.0.0/16, chromatindb-dos03-test-net). Named volumes.
- Node1 (target): max_sync_sessions=1 (only 1 concurrent sync session allowed). sync_cooldown_seconds=0 (disable cooldown so we test sessions, not cooldown).
- Node2, Node3, Node4: all bootstrap to Node1. All will try to sync simultaneously.
- Step 1: Start Node1, wait healthy.
- Step 2: Ingest 500 blobs to Node1 (--size 1024 --rate 100) to create substantial data for sync.
- Step 3: Start Node2, Node3, Node4 simultaneously (all empty, all bootstrap to Node1). They all initiate sync at once.
- Step 4: Wait 30s. With max_sync_sessions=1, only 1 sync session runs at a time. Excess sessions get SyncRejected.
- Step 5: Check Node1 SIGUSR1 metrics: sync_rejections > 0 (proves excess sessions were rejected).
- Step 6: Wait for eventual convergence (up to 180s). Since rejected sessions retry on next sync interval, all nodes should eventually sync. Check that at least one node (Node2 or Node3 or Node4) has all 500 blobs. Check that sync_rejections > 0 in Node1 metrics (proves sessions were limited, not just all succeeded immediately).
- Step 7: Verify the node that completed sync has exactly 500 blobs (existing sessions complete normally, not interrupted by rejection of others).
