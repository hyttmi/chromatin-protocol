<!-- Network partitioning approach (from research) -->
Use `docker network disconnect` / `docker network connect` for coarse partitioning.
This is simpler and more portable than iptables. For partition tests, the topology
is "group A cannot reach group B" which maps cleanly to network disconnect/connect.

After partition heal, nodes may take up to 60-90s to reconnect due to auto-reconnect
exponential backoff. Use SIGHUP to reset backoff timers if convergence is too slow:
docker kill -s HUP container_name  (calls clear_reconnect_state())

Key log patterns:
- "Connected to peer" -- confirms reconnection after heal
- "metrics: peers=N blobs=M" -- SIGUSR1 metrics for verification

Write test_net01_partition_healing.sh following the established test pattern. Uses docker-compose.mesh.yml (5-node mesh from Plan 01).

**Test flow:**

1. Setup: Override COMPOSE to use docker-compose.mesh.yml. Start all 5 nodes. Wait healthy for all 5. Wait 15s for mesh to fully establish connections and PEX discovery.

2. Ingest baseline: Run loadgen to node1 with `--count 100 --size 256 --rate 50 --ttl 3600`. Wait for sync to node5 (100 blobs, 120s timeout). This proves full mesh connectivity.

3. Create partition: Disconnect nodes 3, 4, 5 from the compose network using:
   ```bash
   docker network disconnect chromatindb-test_test-net chromatindb-test-node3
   docker network disconnect chromatindb-test_test-net chromatindb-test-node4
   docker network disconnect chromatindb-test_test-net chromatindb-test-node5
   ```
   This creates partition: {node1, node2} | {node3, node4, node5 -- isolated}

4. Write during partition: Ingest 50 more blobs to node1 (`--count 50 --size 256 --rate 50 --ttl 3600`). Wait for sync to node2 (150 blobs, 60s).

5. Verify partition works: Check node3/node4/node5 blob counts are still 100 (partition is effective). Use `docker exec` to get blob count since these nodes are disconnected from the compose network -- run `docker kill -s USR1` and `docker logs` which work regardless of network state.

6. Heal partition: Reconnect nodes to the network:
   ```bash
   docker network connect --ip 172.28.0.4 chromatindb-test_test-net chromatindb-test-node3
   docker network connect --ip 172.28.0.5 chromatindb-test_test-net chromatindb-test-node4
   docker network connect --ip 172.28.0.6 chromatindb-test_test-net chromatindb-test-node5
   ```
   IMPORTANT: Use --ip to restore original IPs so bootstrap_peers reconnect correctly.

7. Force reconnect: Send SIGHUP to all 5 nodes to clear reconnect backoff timers: `docker kill -s HUP container_name` for each. This ensures reconnection happens within the test timeout rather than waiting for exponential backoff.

8. Wait for convergence: Wait for sync on node5 (150 blobs, 120s timeout). Also verify nodes 3 and 4 have 150 blobs.

9. Verify all 5 nodes converged: Get blob count from all 5 nodes. All should have >= 150 blobs.

**Cleanup:** `$COMPOSE down -v --remove-orphans` in trap.

**IMPORTANT considerations:**
- The compose network name is derived from compose project name + network name: `chromatindb-test_test-net`
- docker kill -s USR1 and docker logs work regardless of network state (they use Docker daemon, not container network)
- SIGHUP after reconnect reloads config which calls clear_reconnect_state(), resetting backoff timers
- Use 120s timeout after healing to allow reconnect + sync convergence

**Topology (4 nodes, manual docker run):**
- Create network: `docker network create --driver bridge --subnet 172.30.0.0/16 chromatindb-split-test-net`
- Node1: 172.30.0.2, bootstrap: [] (seed for group A)
- Node2: 172.30.0.3, bootstrap: [172.30.0.2:4200] (group A)
- Node3: 172.30.0.4, bootstrap: [172.30.0.2:4200] (will be group B seed after partition)
- Node4: 172.30.0.5, bootstrap: [172.30.0.4:4200] (group B)
- Use 172.30.0.0/16 subnet to avoid conflicts with compose networks on 172.28.0.0/16
- All configs: sync_interval_seconds: 5, full_resync_interval: 0, inactivity_timeout_seconds: 0, log_level: debug
- Write temp config files (like ACL-01 pattern with mktemp)

**Test flow:**

1. Start all 4 nodes. Wait healthy. Wait 15s for mesh to form.

2. Verify full connectivity: Ingest 50 blobs to node1 (`--count 50 --size 256 --rate 50 --ttl 3600`). Wait for sync to node4 (50 blobs). Override NETWORK variable for run_loadgen: `NETWORK="chromatindb-split-test-net"`.

3. Create partition: Disconnect nodes 3 and 4 from the network, then reconnect them on a separate network:
   ```bash
   docker network create --driver bridge --subnet 172.31.0.0/16 chromatindb-split-b-net
   docker network disconnect chromatindb-split-test-net chromatindb-test-node3
   docker network disconnect chromatindb-split-test-net chromatindb-test-node4
   docker network connect --ip 172.31.0.4 chromatindb-split-b-net chromatindb-test-node3
   docker network connect --ip 172.31.0.5 chromatindb-split-b-net chromatindb-test-node4
   ```
   Now: Group A = {node1, node2} on 172.30.x, Group B = {node3, node4} on 172.31.x

4. Write to Group A: Ingest 30 blobs to node1 on group A network. Wait sync to node2 (80 blobs).

5. Write to Group B: Override NETWORK to group B network. Ingest 20 blobs to node3 (172.31.0.4). Wait sync to node4 (70 blobs). This uses a separate loadgen on the B network.

6. Verify partition isolation: node1/node2 should have 80 blobs. node3/node4 should have 70 blobs.

7. Heal partition: Disconnect nodes 3/4 from B network, reconnect to A network with original IPs:
   ```bash
   docker network disconnect chromatindb-split-b-net chromatindb-test-node3
   docker network disconnect chromatindb-split-b-net chromatindb-test-node4
   docker network connect --ip 172.30.0.4 chromatindb-split-test-net chromatindb-test-node3
   docker network connect --ip 172.30.0.5 chromatindb-split-test-net chromatindb-test-node4
   ```

8. Force reconnect via SIGHUP to all 4 nodes.

9. Wait for convergence: All 4 nodes should converge to 100 blobs (50 + 30 + 20 = union of all writes). Wait 120s.

10. Verify union: All 4 nodes have >= 100 blobs.

**Cleanup:** Remove all 4 containers, both networks, temp config files in trap.
