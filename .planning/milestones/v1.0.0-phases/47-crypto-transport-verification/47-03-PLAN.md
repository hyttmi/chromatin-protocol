Key transport protocol facts:
- PQ handshake: initiator sends ClientHello with ephemeral ML-KEM-1024 encapsulation key. Responder encapsulates, sends ServerHello with ciphertext. Both derive session key from KEM shared secret. Both sign session fingerprint with ML-DSA-87 identity key.
- Session fingerprint = hash of (KEM shared secret + both identity pubkeys + nonces). MITM substituting KEM keys produces different shared secrets on each side -> fingerprint mismatch.
- Trusted peers: initiator sends TrustedHello (no KEM). Responder checks IP against trusted_peers. If trusted, replies with signed identity proof. If NOT trusted, replies PQRequired (forcing full PQ handshake). Wrong identity key = rejected even if IP is trusted.
- All handshake events logged at debug level (TrustedHello, PQRequired, fingerprint verification, handshake success/failure).
- Forward secrecy: even with long-term identity keys compromised, past session traffic is undecryptable because session key derived from ephemeral KEM shared secret.

Docker networking:
- Bridge network allows tcpdump capture
- NET_ADMIN capability needed for tcpdump in containers
- Container names resolve on bridge network (node1, node2, node3)

**test_crypt04_forward_secrecy.sh** -- Verifies CRYPT-04: ephemeral ML-KEM-1024 provides forward secrecy.

Forward secrecy is a protocol property, not something directly testable by "trying to decrypt." The test verifies it by confirming: (1) ephemeral KEM was used (logs), (2) captured traffic is opaque (no plaintext leakage), (3) the handshake produced a functional session.

Test flow:
1. Source helpers.sh, check_deps, build_image, cleanup
2. Start only node1 using docker-compose.test.yml
3. Start tcpdump capture on the test network using nicolaka/netshoot (has tcpdump pre-installed). Do NOT use --rm so we can copy the capture before removing:
   ```bash
   docker run -d --name tcpdump-capture \
     --network chromatindb-test_test-net \
     --cap-add NET_ADMIN \
     nicolaka/netshoot \
     tcpdump -i any -w /tmp/capture.pcap -s 0 port 4200
   sleep 2  # let tcpdump start
   ```
4. Start node2 (triggers PQ handshake): `$COMPOSE up -d node2`, wait_healthy
5. Ingest a blob and verify it syncs (proving the encrypted session works for data transfer):
   ```bash
   run_loadgen node1 --count 1 --size 256 --ttl 3600
   wait_sync node2 1
   ```
6. Stop tcpdump: `docker kill --signal SIGINT tcpdump-capture`, sleep 1, copy capture file, remove container
7. Verify the capture file has content: `tcpdump -r handshake.pcap | wc -l` should be > 0
8. Forward secrecy verification checks:
   a. Handshake succeeded -- both nodes connected (verified via logs):
      ```bash
      docker logs chromatindb-test-node1-1 2>&1 | grep -q "handshake complete\|session established"
      docker logs chromatindb-test-node2-1 2>&1 | grep -q "handshake complete\|session established"
      ```
   b. Ephemeral KEM was used (from debug logs):
      ```bash
      docker logs chromatindb-test-node2-1 2>&1 | grep -qi "kem\|encapsulat"
      ```

      ```bash

      PLAINTEXT_COUNT=$(strings handshake.pcap | grep -ci "chromatindb\|namespace\|blob" || true)
      # Count should be 0 (or very low -- only from TCP metadata, not payload)
      ```
9. PASS if: handshake succeeded with KEM, captured traffic contains no plaintext, session is functional

**CRYPT-05 MITM Rejection Test**

A true MITM test (intercepting and re-encrypting traffic in-flight) is extremely complex in Docker. The protocol's MITM protection has two layers: (1) KEM shared secret mismatch causes fingerprint mismatch, and (2) identity verification rejects unknown keys. Layer (1) is an inherent protocol property from ephemeral KEM -- we verify its prerequisite (session fingerprint includes ephemeral material) by showing fingerprints differ across sessions. Layer (2) is directly testable via allowed_keys.

**test_crypt05_mitm_rejection.sh flow:**

Part A -- Identity-based rejection (wrong identity rejected via ACL):
1. Source helpers.sh, build_image, cleanup
2. Start node1 and node2 with NO allowed_keys (open mode), wait for connection
3. Extract node2's pubkey from node1's debug logs (logged during handshake)
4. Tear down
5. Create a temporary node1 config with `allowed_keys = [node2_pubkey]`
6. Start node1 + node3-mitm (different identity, connecting to node1 via docker-compose.mitm.yml)
7. Verify node3-mitm is rejected -- check node1 logs for identity rejection:
   ```bash
   docker logs chromatindb-test-node1-1 2>&1 | grep -qi "not.*allowed\|reject\|unauthorized\|denied"
   ```
8. Verify node3-mitm did NOT successfully sync any data

Part B -- Session fingerprint uniqueness (proves ephemeral KEM contribution):
1. Start node1 + node2 (open mode), wait for connection, capture session fingerprint from debug logs
2. Restart node2 (forces new session/handshake), capture new session fingerprint
3. Verify the two fingerprints differ -- this proves the fingerprint includes ephemeral KEM material (not just static identity keys), which is the prerequisite for MITM detection
4. PASS if: wrong identity rejected AND fingerprints differ across sessions
