  <name>Task 1: Create chromatindb_verify CLI tool and update Docker build</name>
  <files>tools/verify_main.cpp, CMakeLists.txt, Dockerfile</files>
  <action>
Create `tools/verify_main.cpp` -- a standalone CLI tool linking against chromatindb_lib (same crypto code paths as the node). Two subcommands:

**`chromatindb_verify hash`**: Reads a FlatBuffer-encoded blob from stdin (or --file path), decodes it via `decode_blob()`, recomputes `build_signing_input(namespace_id, data, ttl, timestamp)` to get the 32-byte SHA3-256 digest, and prints the hex digest to stdout. Also computes and prints the full blob hash via `blob_hash()`. Exit 0 on success, 1 on decode failure.

Output format (JSON to stdout, one line per field for grep-ability):
```json
{"signing_digest":"<64 hex chars>","blob_hash":"<64 hex chars>","namespace_id":"<64 hex>","pubkey_size":<int>,"data_size":<int>,"ttl":<uint32>,"timestamp":<uint64>}
```

**`chromatindb_verify sig`**: Reads a FlatBuffer-encoded blob from stdin (or --file path), decodes it, recomputes the signing digest via `build_signing_input()`, then calls `Signer::verify(digest, signature, pubkey)`. Prints `{"valid":true}` or `{"valid":false}` to stdout. Exit 0 if valid, exit 1 if invalid.

CLI usage:
```
chromatindb_verify hash [--file PATH]   # reads from stdin if no --file
chromatindb_verify sig  [--file PATH]   # reads from stdin if no --file
```

The tool reads binary data from stdin or file, so test scripts pipe raw blob bytes to it.

Include spdlog for error messages to stderr (same pattern as loadgen). Use nlohmann/json for output formatting.

In `CMakeLists.txt` (top-level), add:
```cmake
add_executable(chromatindb_verify tools/verify_main.cpp)
target_link_libraries(chromatindb_verify PRIVATE chromatindb_lib)
```

In `Dockerfile`, add chromatindb_verify to the build targets and COPY it to runtime stage:
- Build line: `cmake --build build --target chromatindb chromatindb_loadgen chromatindb_verify`
- Strip line: `strip build/chromatindb build/chromatindb_loadgen build/chromatindb_verify`
- COPY line: `COPY --from=builder /src/build/chromatindb_verify /usr/local/bin/`

Keep the tool minimal -- no networking, no storage, just decode + crypto verify.

**`tests/integration/helpers.sh`**: Sourced by all test scripts. Utility functions (keep under 150 lines):

- `COMPOSE="docker compose -f $COMPOSE_FILE -p chromatindb-test"` (project name chromatindb-test to avoid conflict with deploy/)
- `log()` -- timestamped stderr logging: `echo "[$(date +%H:%M:%S)] $*" >&2`
- `build_image()` -- `docker build -t chromatindb:test "$REPO_ROOT"` (with --skip-build flag support)
- `cleanup()` -- `$COMPOSE down -v --remove-orphans 2>/dev/null || true`
- `start_nodes()` -- `$COMPOSE up -d` with optional profile arg, calls `wait_healthy` for each node
- `wait_healthy NODE TIMEOUT` -- polls `docker inspect --format '{{.State.Health.Status}}' $NODE` until "healthy" or timeout (default 60s)
- `get_blob_count NODE_CONTAINER` -- sends SIGUSR1 to get metrics, greps docker logs for the latest metrics line, extracts blob count via jq or grep.
- `wait_sync NODE_CONTAINER EXPECTED_COUNT TIMEOUT` -- polls get_blob_count until it matches expected or timeout.
- `run_loadgen()` -- wrapper around docker run for loadgen on the test network
- `run_verify()` -- wrapper around docker exec for chromatindb_verify
- `check_deps()` -- verifies docker, docker compose, jq are available
- `fail()` -- log message + exit 1
- `pass()` -- log PASS message

Each test script sources helpers.sh and uses these primitives. The per-test scripts handle specific test logic.

**`tests/integration/run-integration.sh`**: Discovers all `test_*.sh` files in `tests/integration/`, runs each sequentially, collects pass/fail, prints summary. Usage: `bash tests/integration/run-integration.sh [--skip-build] [--filter PATTERN]`. The --skip-build flag is forwarded to helpers.sh. The --filter flag runs only tests matching the pattern.
