  <name>Task 1: Fix PEX SIGSEGV -- coroutine lifetime during teardown</name>
  <files>db/peer/peer_manager.cpp, db/peer/peer_manager.h, db/tests/test_daemon.cpp</files>
  <action>
The PEX test SIGSEGV (test_daemon.cpp:296) is a coroutine lifetime issue during teardown. Three PeerManagers share one io_context; when stop() is called, coroutines spawned by PeerManager (sync_timer_loop, pex_timer_loop, etc.) may still be in-flight and reference destroyed PeerManager members.

Investigation approach:
1. Build with ASAN (from Plan 01) and run just the PEX test to get a stack trace:
   ```bash
   cd build-asan
   ASAN_OPTIONS="suppressions=../sanitizers/asan.supp:detect_leaks=1:halt_on_error=1" ./chromatindb_tests "three nodes: peer discovery via PEX"
   ```
   Run it several times -- the SIGSEGV may be intermittent. The ASAN stack trace will show the exact use-after-free/stack-use-after-return.

2. Build with TSAN to identify any data races in the teardown path:
   ```bash
   cmake -B build-tsan -DSANITIZER=tsan
   cmake --build build-tsan
   cd build-tsan
   TSAN_OPTIONS="suppressions=../sanitizers/tsan.supp:halt_on_error=1" ./chromatindb_tests "three nodes: peer discovery via PEX"
   ```

3. Analyze the root cause. Known patterns from project history (MEMORY.md):
   - "Every co_await is a potential container invalidation point"
   - "SIGHUP via dedicated coroutine member function (not lambda -- stack-use-after-return)"
   - "Snapshot iteration: copy connection pointers before iterating across co_await points"
   - "Deque for coroutine-accessed containers (not vector)"

   Likely root causes:
   a) PeerManager::stop() cancels timers but coroutines still hold `this` pointer. After PeerManager destructor runs (when test's local variables go out of scope), coroutine resumes on dangling pointer.
   b) The 2-second ioc.run_for(2s) after stop() isn't enough for all three nodes' coroutines to complete.
   c) co_spawn'd coroutines capture `this` and outlive the PeerManager.

4. Fix the root cause. The fix must ensure PeerManager's coroutines are fully completed before destruction. Approaches (choose the one that fits the actual root cause):

   a) **Ensure stop() is synchronous with respect to coroutines**: stop() sets stopping_=true and cancels all timers. The coroutines (sync_timer_loop, pex_timer_loop, etc.) check stopping_ on timer cancellation and exit. The issue is likely that the coroutine hasn't been polled yet after cancel when PeerManager is destroyed.

   b) **Extend teardown time in test**: If the fix is insufficient run_for time, increase the post-stop run_for. But this is fragile -- prefer a proper solution.

   c) **Use shared_from_this pattern**: Make PeerManager inherit from enable_shared_from_this and capture shared_ptr in coroutines. This ensures PeerManager outlives its coroutines. However, this is a large refactor -- only do this if simpler approaches fail.

   d) **Guard coroutine bodies**: Wrap coroutine bodies so that after stop/cancel, the resumed coroutine immediately returns without accessing any member state. Check if all coroutines properly handle asio::error::operation_aborted after co_await on timers.

5. If the test teardown pattern needs fixing (e.g., shared io_context with multiple PeerManagers):
   - The test creates 3 PeerManagers on 1 io_context
   - stop() on all 3, then run_for(2s) to drain
   - If coroutines reference PeerManager members after stop, the 2s drain may execute coroutine continuations on destroyed objects
   - Fix: ensure the io_context fully drains all PeerManager-related work. May need to run_for longer, or restructure stop() to post a completion handler that signals "all coroutines done."

6. After fixing, verify the PEX test passes 10 consecutive times under the normal build:
   ```bash
   for i in $(seq 1 10); do ./chromatindb_tests "three nodes: peer discovery via PEX" || exit 1; done
   ```

Do NOT apply band-aid fixes (disabling the test, adding sleeps as the primary fix, catching SIGSEGV). Fix the root cause even if it requires refactoring PeerManager teardown.

Create a reusable bash script that runs all [daemon][e2e] tagged tests N times and reports pass/fail statistics.

1. Create `scripts/run-e2e-reliability.sh`:
```bash
#!/usr/bin/env bash
# Run [daemon][e2e] tests N times and report reliability.
# Usage: ./scripts/run-e2e-reliability.sh [build_dir] [runs] [sanitizer_opts]
# Example: ./scripts/run-e2e-reliability.sh build-tsan 50 "TSAN_OPTIONS=suppressions=sanitizers/tsan.supp:halt_on_error=1"
set -euo pipefail

BUILD_DIR="${1:?Usage: $0 <build_dir> [runs] [env_opts]}"
RUNS="${2:-50}"
ENV_OPTS="${3:-}"

BINARY="$BUILD_DIR/chromatindb_tests"
if [[ ! -x "$BINARY" ]]; then
    echo "ERROR: $BINARY not found or not executable"
    exit 1
fi

TAG="[daemon][e2e]"
PASS=0
FAIL=0
FAILURES=()

echo "=== E2E Reliability Test ==="
echo "Binary: $BINARY"
echo "Tag filter: $TAG"
echo "Runs: $RUNS"
echo "Env: ${ENV_OPTS:-<none>}"
echo ""

for i in $(seq 1 "$RUNS"); do
    printf "Run %3d/%d ... " "$i" "$RUNS"
    if env $ENV_OPTS "$BINARY" "$TAG" --colour-mode none > /dev/null 2>&1; then
        echo "PASS"
        PASS=$((PASS + 1))
    else
        echo "FAIL"
        FAIL=$((FAIL + 1))
        FAILURES+=("$i")
    fi
done

echo ""
echo "=== Results ==="
echo "Passed: $PASS/$RUNS"
echo "Failed: $FAIL/$RUNS"
if [[ ${#FAILURES[@]} -gt 0 ]]; then
    echo "Failed runs: ${FAILURES[*]}"
    exit 1
fi
echo "All runs passed."
exit 0
```
