**test_ttl02_tombstone_ttl_expiry.sh** -- TTL-02: Tombstone TTL Expiry and GC

Topology: 2-node standalone topology. Subnet 172.44.0.0/16, containers chromatindb-ttl02-node{1,2}. Named volumes chromatindb-ttl02-node{N}-data. Config: sync_interval_seconds=5, full_resync_interval=9999, inactivity_timeout_seconds=0.

This test requires patience: TTL=60 means blobs expire after 60s, and the expiry_scan_loop runs every 60s. Total wait for GC: ~120-140s (TTL + one scan cycle + margin).

Flow:
1. Start node1 + node2. Wait healthy + mesh formation (sleep 10).
2. Generate owner identity on node1, ingest 20 blobs with --ttl 3600 (long-lived). Wait for sync to node2.
3. Owner tombstones all 20 blobs with **TTL=60** (finite-TTL tombstones): pipe all 20 hashes to loadgen --delete --hashes-from stdin --identity-file --ttl 60.
4. Wait for tombstones to sync to node2 (wait_sync to 40 on node2: 20 original + 20 tombstones).
5. Record pre-GC state:
   - Blob count on node1 (should be 40: 20 blobs + 20 tombstones).
   - Restart node1 to trigger integrity scan. Grep logs for "tombstone=" count (should be >= 20 tombstone_map entries). Record this value.
   - Record storage size: use `docker exec chromatindb-ttl02-node1 du -b /data/mdbx.dat` to get database file size.
6. Wait for tombstone TTL expiry + GC scan: sleep 140 (60s TTL + 60s scan cycle + 20s margin). The tombstones were created ~20s ago at this point, so effective wait from tombstone creation is ~160s, which is well past TTL=60 + one 60s scan cycle.
7. Post-GC verification on node1:
   - Check 1 (blob count decreased): get_blob_count on node1. After GC, expired tombstones AND their original blobs (already tombstoned) should be purged. The expiry_scan purges anything in the expiry_map whose timestamp has passed. Both the original 20 blobs (TTL=3600, not yet expired) and the tombstones (TTL=60, expired) are in play. Only the tombstones with TTL=60 should expire. Blob count should decrease from 40 to 20 (the 20 tombstones are removed by GC, leaving the 20 original blobs which still have TTL=3600 remaining). Wait -- the originals were already tombstoned. The original blobs still exist in storage (tombstoning adds a tombstone blob but doesn't remove the original from blobs_map). GC removes blobs whose TTL has expired from the expiry_map. The original 20 blobs have TTL=3600 (not expired yet). The 20 tombstones have TTL=60 (expired after 60s). So GC removes the 20 tombstones from blobs_map and their corresponding tombstone_map entries. Remaining: 20 original blobs. Blob count: 20.
   - Check 2 (tombstone_map entries removed): Restart node1 again (or send SIGUSR1). Grep integrity scan for "tombstone=" -- should now be 0 (all tombstone_map entries cleaned up when tombstone blobs were GC'd).
   - Check 3 (storage decreased): Re-measure database file size. It should be less than or equal to pre-GC size. Note from MEMORY.md: "mmap file size doesn't shrink (freed pages reused internally)" -- so du -b on the mdbx.dat file may not show decrease. Instead, use used_data_bytes() which is exposed via... the integrity scan log. Alternatively, verify via the blob count and tombstone_map entry count (more reliable than file size). Mark file size check as supplementary. The primary verification is blob count decrease (40 -> 20) and tombstone_map count decrease (20 -> 0).

8. Repeat checks on node2 for completeness: node2 also runs its own GC independently. After the same wait period, node2 should also show 20 blobs and 0 tombstone_map entries.

Note about storage decrease verification: Per project MEMORY.md, mmap file size doesn't shrink. The requirement says "storage decreases" which refers to logical storage (blob count, used B-tree pages) not physical file size. Verify via blob count and tombstone_map entries. If the integrity scan output on restart shows the blobs= count decreased, that satisfies the requirement.

**test_e2e02_history_backfill.sh** -- E2E-02: History Backfill (1000 messages)

Topology: 3-node standalone topology. Subnet 172.45.0.0/16, containers chromatindb-e2e02-node{1,2,3}. Named volumes chromatindb-e2e02-node{N}-data. Config: sync_interval_seconds=5, full_resync_interval=9999, inactivity_timeout_seconds=0.

Flow:
1. Start node1 (seed). Wait healthy.
2. Generate owner identity, ingest 1000 blobs to node1 (--count 1000 --size 256 --rate 2000 --ttl 3600 --identity-save). The rate 2000 prevents sync timer disconnect (per Phase 49 decision: rate 2000/sec for bulk ingest).
3. Verify node1 has 1000 blobs via get_blob_count.
4. Start node2 bootstrapping to node1. Wait for full sync (wait_sync to 1000, timeout 180s).
5. Verify node2 has 1000 blobs.
6. Start node3 (the "relay joining late") bootstrapping to node1. This is the late-joiner that needs to backfill all 1000 messages.
7. Wait for node3 to catch up (wait_sync to 1000, timeout 180s).
8. Check 1: Node3 has exactly 1000 blobs (complete backfill, no gaps).
9. Check 2: Blob counts match across all 3 nodes -- node1 = node2 = node3 = 1000.
10. Check 3: Monotonic sequence numbers -- on node3, the sequence numbers should be monotonically increasing. Verify via chromatindb_verify or by checking cursor_misses in SIGUSR1 metrics: cursor_misses >= 1 means the late-joiner had to do a full namespace sync (no prior cursor), which confirms reconciliation was used for backfill.
11. Check 4: No gaps -- ingest 10 more blobs to node1 (same identity). Wait for sync to all 3 nodes (wait_sync to 1010). If node3 syncs to 1010, it means cursor state is correct and incremental sync works after the initial backfill. This proves no gaps: the backfill left valid cursor state for subsequent syncs.
12. Check 5: Tombstone propagation within the namespace -- owner tombstones one blob. Wait for propagation (15s, one sync interval). Verify all 3 nodes have 1011 blobs (1010 + 1 tombstone). This satisfies the "tombstones propagate to all nodes within one sync interval" part of success criteria 4.
