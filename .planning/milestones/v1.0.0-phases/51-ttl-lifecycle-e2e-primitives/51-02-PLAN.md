**test_e2e01_async_delivery.sh** -- E2E-01: Async Message Delivery

Topology: 3-node standalone topology. Subnet 172.41.0.0/16, containers chromatindb-e2e01-node{1,2,3}. Named volumes chromatindb-e2e01-node{N}-data. Config: sync_interval_seconds=5, full_resync_interval=9999, inactivity_timeout_seconds=0.

Flow:
1. Start all 3 nodes (node1 as seed, node2 + node3 bootstrap to node1). Wait healthy.
2. Wait for peer mesh to form (sleep 10).
3. Generate "Alice" identity on node1, ingest 5 baseline blobs (--identity-save). Record blob count baseline.
4. Wait for sync to node2 and node3 (wait_sync to 5).
5. Disconnect node3: `docker stop chromatindb-e2e01-node3`. Node3 is now "offline recipient."
6. Alice writes 10 new "message" blobs to node1 (--identity-file, --count 10 --size 256 --ttl 3600). These are written while node3 is disconnected.
7. Wait for sync between node1 and node2 (wait_sync to 15 on node2).
8. Reconnect node3: `docker start chromatindb-e2e01-node3`. Wait healthy.
9. Wait for node3 to sync up (wait_sync to 15 on node3, timeout 60s).
10. Check 1: Node3 has all 15 blobs (5 baseline + 10 while offline). This proves async delivery via sync.
11. Check 2: Node3 logs show pub/sub notification activity -- grep for "Notifying" or "pub_sub" or "Notify" in node3 logs after reconnection. This proves pub/sub notification was received. (Note: the exact log message depends on implementation -- grep for "notify" case-insensitive.)
12. Check 3: All 3 nodes have matching blob counts (node1=node2=node3=15).

**test_e2e03_delete_for_everyone.sh** -- E2E-03: Delete for Everyone

Topology: 3-node standalone topology. Subnet 172.42.0.0/16, containers chromatindb-e2e03-node{1,2,3}. Same config (sync_interval_seconds=5).

Flow:
1. Start 3 nodes (mesh). Wait healthy + mesh formation (sleep 10).
2. Generate owner identity, ingest 10 blobs to node1. Wait for sync to all 3 nodes.
3. Record timestamp_before = $(date +%s).

6. Record timestamp_after = $(date +%s). Compute propagation_time = timestamp_after - timestamp_before.
7. Check 1: Node2 has the tombstone -- get_blob_count shows 11 (10 original + 1 tombstone).
8. Check 2: Node3 has the tombstone -- get_blob_count shows 11.
9. Check 3: Propagation completed within one sync interval -- propagation_time <= 15s (5s sync interval + 10s margin for test overhead).
10. Check 4: Tombstone is tracked in tombstone_map -- restart node2 (docker restart), wait healthy, grep integrity scan logs for "tombstone=1" (at least 1 tombstone_map entry).

**test_e2e04_namespace_isolation.sh** -- E2E-04: Multi-Namespace Isolation

Topology: 3-node standalone topology. Subnet 172.43.0.0/16, containers chromatindb-e2e04-node{1,2,3}. Config: sync_interval_seconds=5, full_resync_interval=9999, inactivity_timeout_seconds=0.

Flow:
1. Start 3 nodes. Wait healthy + mesh formation.
2. Generate Identity A (namespace A) and Identity B (namespace B) using two loadgen runs with --identity-save to separate temp dirs.
3. Ingest 20 blobs to namespace A via node1 (--identity-file /identityA --count 20 --size 256 --ttl 3600).
4. Ingest 15 blobs to namespace B via node2 (--identity-file /identityB --count 15 --size 256 --ttl 3600).
5. Wait for full sync convergence across all 3 nodes: wait_sync to 35 on all nodes (20 + 15 = 35 total).
6. Check 1: Total blob count matches on all 3 nodes (35 each). This confirms sync works correctly.
7. Check 2: Namespace fingerprint isolation -- use chromatindb_verify to inspect per-namespace state. Since we don't have a direct fingerprint query tool, verify isolation through the following approach:
   - On node3 (which received all blobs via sync), run integrity scan by restarting: `docker restart chromatindb-e2e04-node3`, wait healthy.
   - Grep node3 startup logs for integrity scan output. It should show blobs distributed across exactly 2 namespaces.
   - The integrity scan logs "integrity scan: blobs=N seq=N ..." which gives total counts.
8. Check 3: Namespace-specific blob counts via SIGUSR1 metrics -- metrics dump on each node. The blobs= count should be 35 on every node (sum of both namespaces). Since sync is per-namespace, the fact that all nodes have exactly 35 proves both namespaces synced independently and no blobs leaked between them.
9. Check 4: Verify distinct namespace existence -- the loadgen stderr output for each identity includes "namespace: <hex>". Verify the two namespace hex values are different. Then verify that node3 (which only received blobs via sync) has the correct total -- if namespaces leaked, counts would be wrong.
10. Check 5 (fingerprint verification): Ingest 1 more blob to namespace A on node1. Wait for sync. Node2 and node3 should now have 36 blobs. If there was cross-namespace contamination, the count would be anomalous. This is the definitive isolation check -- a change in namespace A only affects namespace A's sync, and the total increases by exactly 1 on all nodes.
