Existing run-benchmark.sh patterns:
- `run_loadgen NODE [args...]` -- docker run with --network
- `reset_topology` -- down -v, up -d
- `capture_stats PHASE LABEL` -- docker stats snapshot
- `wait_for_convergence CONTAINER EXPECTED_COUNT TIMEOUT` -- polls get_blob_count (seq_num high-water)
- `get_blob_count CONTAINER` -- SIGUSR1 + grep blobs=N (NOTE: this is seq_num, not live count)
- `format_ingest_row LABEL DISPLAY` -- builds markdown table row
- `format_resource_table SCENARIO` -- builds resource usage table
- `generate_report()` -- assembles REPORT.md + benchmark-summary.json

SIGUSR1 metrics line format:
```
metrics: peers=N connected_total=N ... blobs=N storage=X.XMiB ...
```
Note: `blobs=N` is latest_seq_num (monotonically increasing, NEVER decreases).
`storage=X.XMiB` tracks actual disk usage and DOES decrease after GC.

  <action>
Add three new scenario functions and supporting helpers to run-benchmark.sh. These go after the existing scenario functions (after `run_scenario_trusted_vs_pq`), before `main()`.

**CRITICAL NOTE from RESEARCH:** `get_blob_count()` returns `latest_seq_num` which is a high-water mark and NEVER decreases. GC measurement CANNOT use `get_blob_count()`. Must use `storage=X.XMiB` from SIGUSR1 output for GC convergence detection.

1. **Helper: `get_storage_mib()`**: New function that sends SIGUSR1 and parses `storage=X.XMiB` from the metrics line. Returns the floating-point MiB value. Pattern:
   ```bash
   get_storage_mib() {
       local container="$1"
       docker kill -s USR1 "$container" >/dev/null 2>&1 || true
       sleep 1
       docker logs "$container" 2>&1 | grep "metrics:" | tail -1 \
           | grep -oP 'storage=\K[0-9.]+' || echo "999"
   }
   ```

2. **Helper: `wait_for_gc_completion()`**: Polls `get_storage_mib()` until storage drops to or below a threshold. Parameters: container, max_storage_mib, timeout_seconds. Returns elapsed_ms. Uses `bc -l` for float comparison. Poll every 3 seconds. Pattern from RESEARCH.md `wait_for_gc_completion()`.

3. **Helper: `wait_for_count_decrease()`**: Polls `get_blob_count()` until count drops to or below expected_count. Parameters: container, expected_count, timeout_seconds. Returns elapsed_ms. Used for tombstone sync measurement where seq_num IS useful (after tombstones sync, remote seq_num increases to include tombstone entries -- so wait_for_convergence with expected = original_blobs + tombstone_count works).

   Actually, CORRECTION: For sync propagation of tombstones, we should wait for the blob count (seq_num) to INCREASE on remote nodes to `original_count + tombstone_count`. Tombstones get their own seq_nums. So `wait_for_convergence()` (existing, waits for count >= expected) works for sync measurement. We only need the storage-based polling for GC.

4. **`run_scenario_tombstone_creation()`**: Measures tombstone creation throughput (BENCH-01).
   - Iterate over sizes=(1024 102400 1048576) with labels=("1k" "100k" "1m")
   - For each size:
     a. `reset_topology`
     b. `capture_stats "pre" "tombstone-creation-${label}"`
     c. Create temp identity dir: `identity_dir=$(mktemp -d)`
     d. Write phase: `write_output=$(run_loadgen node1 --count "$BLOB_COUNT" --rate "$RATE" --size "$size" --ttl 3600 --drain-timeout 10 --identity-save "$identity_dir")`
       - Note: run_loadgen uses `docker run --rm`. The --identity-save path must be INSIDE the container. Mount the temp dir: add `-v "$identity_dir:/tmp/bench-id"` to the docker run call. Use `--identity-save /tmp/bench-id`.
       - IMPORTANT: Need a variant of run_loadgen or modify it to accept volume mounts. Create `run_loadgen_with_identity()` that adds the `-v` mount. OR simpler: modify `run_loadgen()` to accept an optional identity dir parameter.
       - Cleanest approach: Create `run_loadgen_v()` that takes an extra volume mount arg before the target host:
         ```bash
         run_loadgen_v() {
             local volume_mount="$1"
             local target_host="$2"
             shift 2
             docker run --rm --network "$NETWORK" \
                 -v "$volume_mount" \
                 --entrypoint chromatindb_loadgen \
                 chromatindb:latest \
                 --target "${target_host}:4200" "$@" \
                 2>/dev/null
         }
         ```
     e. Extract hashes: `hashes=$(echo "$write_output" | jq -r '.blob_hashes[]')`
     f. Delete phase: `delete_output=$(echo "$hashes" | run_loadgen_v "$identity_dir:/tmp/bench-id" node1 --delete --hashes-from stdin --identity-file /tmp/bench-id --drain-timeout 10)`
       - Note: need `-i` flag on docker run to read stdin. Modify `run_loadgen_v` to also include `-i`.
     g. `capture_stats "post" "tombstone-creation-${label}"`
     h. `echo "$delete_output" > "$RESULTS_DIR/scenario-tombstone-creation-${label}.json"`
     i. Log summary (blobs_per_sec, p50, p95, p99 from delete_output)
     j. `rm -rf "$identity_dir"`

5. **`run_scenario_tombstone_sync()`**: Measures tombstone sync propagation (BENCH-02).
   - `reset_topology`
   - `capture_stats "pre" "tombstone-sync"`
   - Create identity dir, write 200 blobs (1 KiB, TTL=3600) to node1 via `run_loadgen_v`
   - Wait for all 3 nodes to converge at 200 blobs using `wait_for_convergence`
   - Extract hashes from write output
   - Delete all 200 blobs on node1 via `run_loadgen_v` with `--delete`
   - After delete completes on node1, measure time until node2 and node3 have the tombstones:
     - Tombstones get new seq_nums, so after writing 200 blobs + 200 tombstones, seq_num high-water = 400
     - Use `wait_for_convergence chromatindb-node2 400 120` for 1-hop
     - Use `wait_for_convergence chromatindb-node3 400 120` for 2-hop
   - `capture_stats "post" "tombstone-sync"`
   - Build result JSON: `{ scenario: "tombstone-sync", blob_count: 200, sync_1hop_ms: N, sync_2hop_ms: N }`
   - Write to `$RESULTS_DIR/scenario-tombstone-sync.json`
   - Cleanup identity dir

6. **`run_scenario_tombstone_gc()`**: Measures GC/expiry performance (BENCH-03).
   - `reset_topology`
   - `capture_stats "pre" "tombstone-gc"`
   - Capture baseline storage: `baseline_storage=$(get_storage_mib chromatindb-node1)`
   - Create identity dir, write 200 blobs (1 KiB, TTL=30) to node1 via `run_loadgen_v`
   - Wait for all 3 nodes to converge at 200 (blobs synced)
   - Create 200 tombstones with TTL=30: `run_loadgen_v ... --delete --hashes-from stdin --identity-file ... --ttl 30 --drain-timeout 10`
   - Wait for tombstone sync: `wait_for_convergence` on all 3 nodes at 400 (200 blobs + 200 tombstones)
   - Now wait for GC. Both original blobs (TTL=30) and tombstones (TTL=30) should expire.
   - Expiry scan runs every 60 seconds. With TTL=30, worst case: written at t=0, expires t=30, next scan at up to t=120. Allow 180 seconds timeout.
   - Set GC threshold: `gc_threshold=$(echo "$baseline_storage + 0.5" | bc -l)` (allow small overhead for metadata)
   - For each node, poll: `gc_ms_nodeN=$(wait_for_gc_completion chromatindb-nodeN "$gc_threshold" 180)`
   - `capture_stats "post" "tombstone-gc"`
   - Build result JSON: `{ scenario: "tombstone-gc", blob_count: 200, ttl: 30, gc_node1_ms: N, gc_node2_ms: N, gc_node3_ms: N, gc_worst_ms: max(all) }`
   - Write to `$RESULTS_DIR/scenario-tombstone-gc.json`
   - Cleanup identity dir
