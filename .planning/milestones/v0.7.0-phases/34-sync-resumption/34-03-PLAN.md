## Most Likely Root Cause

The most probable bug is a **stale cursor comparison**. Here is the scenario:

1. First sync: node2 initiates, node1 responds
2. Node2 stores cursor for node1's namespace with seq_num=1 (node1 had 1 blob)
3. Tombstone added on node1 (seq_num becomes 2)
4. SIMULTANEOUSLY: Both nodes try to sync. Node1 initiates (timer fires), node2 initiates (timer fires).
5. One sync succeeds, one gets "no SyncAccept" and aborts
6. The successful sync might be node1-initiated. In that case:
   - Node1 (initiator) sends NamespaceList with seq_num=2
   - Node2 (responder) receives it, cursor check: cursor.seq_num=1, peer_seq=2, MISS. Good.
   - But node2 as responder also sends its own NamespaceList with seq_num=1
   - Node1 receives node2's NamespaceList, no cursor exists for node2 -> MISS. Good.
   - Phase C: node2 requests tombstone from node1. Node1 sends it. Tombstone propagates.
   - Post-sync: node2 updates cursor for node1 to seq_num=2.

This still seems correct. But consider a more subtle scenario:

**What if the first sync takes TWO rounds to fully propagate?** In round 1, node2 gets the blob. In round 2, the blob might still need to "settle" (e.g., node1 didn't see node2's blob in the first round because node2's NamespaceList was sent before receiving the blob).

After round 1: node2 has blob, cursor for node1 = seq_num=1.
After round 2 (no new data): node2's cursor for node1 stays at seq_num=1, round_count=2.
Then tombstone added. Round 3: node1 seq=2, cursor.seq=1, MISS. Phase C runs. Should work.

**The real danger might be in the `now_ts` variable.** The `now_ts = std::time(nullptr)` is captured once at the start of the cursor decision block and used for ALL cursor updates at the end. If the sync takes a while, the `last_sync_timestamp` written might be slightly stale, but that shouldn't cause a cursor hit/miss problem.

**Alternative theory: The cursor update at the end of sync uses `pns.latest_seq_num` from the NamespaceList received at the START of sync.** If node1 adds the tombstone DURING the sync (between Phase A and the post-sync cursor update), the cursor would be set to the OLD seq_num (before tombstone), and the NEXT sync round would see cursor.seq == peer.seq (if peer hasn't changed since) -- CURSOR HIT, skipping the namespace entirely. This is the "cursor update before completion" pitfall in reverse: the seq_num in `peer_namespaces` is a point-in-time snapshot from Phase B, but by the time Phase C completes, the peer may have new data.

This is the most likely cause given the test flow:
- ioc.run_for(8s): multiple sync rounds happen, blob propagates
- Tombstone added on node1 AFTER run_for(8s) returns
- ioc.run_for(15s): more sync rounds

The tombstone is added BETWEEN run_for calls. So during the run_for(15s), the first sync round should see node1's updated seq_num. Unless the cursor from the LAST sync round during run_for(8s) already matches node1's seq_num at that time, and node1's seq_num hasn't changed when the tombstone is added after the ioc stops.

Wait, that can't be right. The tombstone is added AFTER `ioc.run_for(8s)` returns, meaning node1's seq_num increases after the io_context stops. When `ioc.run_for(15s)` starts, node1's NamespaceList will include the higher seq_num.

But actually... `ioc.run_for(8s)` may not drain all pending completions cleanly. There could be a sync round IN PROGRESS when the 8s expires. The run_for returns, but coroutines may be suspended mid-sync. Then the tombstone is added. Then `ioc.run_for(15s)` resumes those suspended coroutines. The resumed sync might complete its post-sync cursor update with the OLD seq_num (from before the tombstone), setting cursor.seq_num = node1's seq_num (1, before tombstone). Then node1 adds tombstone (seq becomes 2). But the cursor was JUST set to 1.

No wait, that's the normal case. The cursor is set to 1, then node1's seq becomes 2. Next sync round: cursor=1, peer_seq=2, MISS. Tombstone transfers.

I'm going in circles. Let me just write a plan that has the executor diagnose and fix it.
