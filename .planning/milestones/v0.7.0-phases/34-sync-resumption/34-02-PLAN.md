<objective>
Integrate cursor-based sync decisions into the sync orchestration layer, making sync rounds skip unchanged namespaces while preserving correctness via periodic full resync.

Purpose: SYNC-02 requires skipping hash exchange for unchanged namespaces. SYNC-04 requires periodic full resync fallback. This plan layers cursor logic onto run_sync_with_peer and handle_sync_as_responder, adds metrics (cursor_hits, cursor_misses, full_resyncs), extends SIGHUP to reset round counters, and adds startup cursor cleanup.
Output: Cursor-aware sync that skips unchanged namespaces, falls back to full resync periodically, and reports metrics via SIGUSR1 and periodic log.
</objective>

    **Cursor-aware sync logic (peer_manager.cpp):**

    The core change is in Phase A of BOTH `run_sync_with_peer` AND `handle_sync_as_responder`. The cursor decision is about which namespaces to send hash lists for (our outbound side), and does NOT change the receiving side.

    **IMPORTANT DESIGN NOTE:** The cursor tracks what we know about the PEER's state. When we see the peer's NamespaceList with their seq_nums, we compare against our stored cursor. If cursor.seq_num == peer_seq_num for a namespace, the peer has nothing new for us -- so we can skip requesting hashes for that namespace. But we STILL need to send our own hash lists for all our namespaces so the PEER can decide what to request from us. The cursor skip applies only to Phase B (receiving peer's hash lists) and Phase C (requesting blobs from peer).

    **REVISED FLOW for run_sync_with_peer:**
    1. After handshake (SyncRequest/SyncAccept), compute `peer_hash = crypto::sha3_256(conn->peer_pubkey())`
    2. Phase A (UNCHANGED): Send our NamespaceList + all our HashLists + SyncComplete. We always send our full state so the peer can request what they need.
    3. Phase B: Receive peer's NamespaceList. For each peer namespace:
       - Look up cursor: `storage_.get_sync_cursor(peer_hash, ns_id)`
       - Determine if full resync round via `check_full_resync(cursor, now)`
       - If cursor exists AND cursor.seq_num == peer_seq_num AND NOT full resync:
         CURSOR HIT -- mark namespace as "skip" (don't need peer's hashes for this ns)
       - If cursor exists AND peer_seq_num < cursor.seq_num:
         MISMATCH -- log warn, reset cursor for this namespace (delete_sync_cursor), fall through to full diff
       - Otherwise: CURSOR MISS -- need full hash diff for this namespace
    4. Receive peer's HashLists + SyncComplete. For cursor-hit namespaces, ignore their HashList (or they may not send one -- either way, skip Phase C for that namespace).

    **CRITICAL INSIGHT:** The existing protocol has BOTH sides independently sending NamespaceList + HashLists. The cursor skip means we still RECEIVE the peer's HashLists for all their namespaces (we can't tell them to skip), but we simply don't process the diff for cursor-hit namespaces. The outbound optimization (not sending our HashLists for namespaces the peer doesn't need) would require a wire protocol change -- out of scope per research.

    **SIMPLIFIED APPROACH:** Keep the wire protocol identical. Both sides send all their HashLists as before. The optimization is purely in Phase C: for cursor-hit namespaces, skip computing the diff and skip BlobRequest/BlobTransfer. This is simpler, correct, and still achieves O(new) for the expensive Phase C operations (blob transfer is the dominant cost, not hash exchange).

    In Phase C, for each namespace in peer_hashes:
    - If namespace was marked as cursor-hit: skip diff computation + blob requests, `++metrics_.cursor_hits`
    - Otherwise: do full diff as before, `++metrics_.cursor_misses`

    After Phase C completes successfully (before PEX):
    - For each peer namespace, update cursor:
      ```cpp
      storage::SyncCursor updated;
      updated.seq_num = peer_ns_info.latest_seq_num;
      updated.round_count = old_round_count + 1;
      updated.last_sync_timestamp = now;
      storage_.set_sync_cursor(peer_hash, peer_ns_info.namespace_id, updated);
      ```
    - ONLY update cursors after successful sync completion (Pitfall 1 from research)

    If full resync round: `++metrics_.full_resyncs`, log at info (periodic) or warn (time gap, mismatch)

    **Same changes in handle_sync_as_responder:** Mirror the same cursor logic. Both sides independently decide cursor skip.

    **check_full_resync helper:**
    ```cpp
    FullResyncReason PeerManager::check_full_resync(
        const storage::SyncCursor& cursor, uint64_t now) const {
        // Periodic: every Nth round (round_count 0 always triggers -- covers SIGHUP reset)
        if (config_.full_resync_interval > 0 &&
            cursor.round_count % config_.full_resync_interval == 0)
            return FullResyncReason::Periodic;
        // Time gap
        if (config_.cursor_stale_seconds > 0 &&
            cursor.last_sync_timestamp > 0 &&
            now - cursor.last_sync_timestamp > config_.cursor_stale_seconds)
            return FullResyncReason::TimeGap;
        return FullResyncReason::None;
    }
    ```
    Note: round_count=0 (from SIGHUP reset or new cursor) naturally triggers Periodic since 0 % N == 0. The `SighupReset` reason is implicit.
