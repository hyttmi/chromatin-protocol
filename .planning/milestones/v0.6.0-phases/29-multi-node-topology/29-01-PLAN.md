**deploy/docker-compose.yml:**
- 3 services (node1, node2, node3) using `image: chromatindb:latest`
- Chain topology: node1 is seed, node2 depends on node1 healthy, node3 depends on node2 healthy
- Each service: `command: ["run", "--config", "/config/nodeN.json", "--data-dir", "/data", "--log-level", "debug"]`
- Each service: own named volume for /data (node1-data, node2-data, node3-data, node4-data)
- Each service: config file bind-mounted read-only from `./configs/nodeN.json:/config/nodeN.json:ro`
- All services on shared `chromatindb-net` bridge network
- Host port mapping: node1=4201, node2=4202, node3=4203, node4=4204 (all mapping to container port 4200)
- Container names: chromatindb-node1, chromatindb-node2, chromatindb-node3, chromatindb-node4
- Healthcheck override per service: `exec 3<>/dev/tcp/127.0.0.1/4200 && exec 3>&-` with interval=5s, timeout=3s, start_period=10s, retries=3
- `depends_on` with `condition: service_healthy` for ordered startup
- node4 service: same pattern but with `profiles: ["latejoin"]` so it does not start with `docker compose up`
- node4 depends_on node3 service_healthy
- 4 named volumes declared (node1-data through node4-data)
- No `version:` key (Compose v2 does not need it)

Key anti-patterns to avoid:
- Do NOT use full mesh bootstrap (defeats multi-hop testing)
- Do NOT use host bind-mounts for /data (named volumes avoid permission issues)
- Do NOT hardcode container IPs (use service name DNS)
- Do NOT share data volumes between nodes (libmdbx file-level locking)
- sync_interval_seconds MUST be 10 (not default 60) for fast validation

<task type="checkpoint:human-verify" gate="blocking">
  <name>Task 2: Verify multi-node topology runs and replicates</name>
  <what-built>Docker Compose 3-node chain topology with late-joiner support. Node1 (seed) -> Node2 -> Node3 chain, with node4 available via latejoin profile.</what-built>
  <how-to-verify>
1. Build the Docker image (if not already built):
   ```
   docker build -t chromatindb:latest .
   ```

2. Start the 3-node topology:
   ```
   docker compose -f deploy/docker-compose.yml up -d
   ```

3. Verify all 3 nodes are healthy:
   ```
   docker compose -f deploy/docker-compose.yml ps
   ```
   Expected: node1, node2, node3 all show "healthy" status.

4. Watch logs to confirm handshakes and peer connections:
   ```
   docker compose -f deploy/docker-compose.yml logs --tail=20
   ```
   Expected: Each node logs successful peer connections. Node2 connects to node1, node3 connects to node2.

5. Write blobs to node1 via loadgen (from host, port 4201):
   ```
   docker compose -f deploy/docker-compose.yml exec node1 chromatindb_loadgen --target 127.0.0.1:4200 --count 5 --rate 1
   ```

6. Wait ~20 seconds for sync propagation through chain (10s sync interval x 2 hops).

7. Verify node3 received the blobs (multi-hop chain replication):
   ```
   docker kill -s USR1 chromatindb-node3
   docker compose -f deploy/docker-compose.yml logs node3 | tail -10
   ```
   Expected: Metrics show ingested blobs (count > 0).

8. Start the late-joiner node4:
   ```
   docker compose -f deploy/docker-compose.yml --profile latejoin up -d node4
   ```

9. Wait ~10 seconds for node4 to sync from node3.

10. Verify node4 caught up:
    ```
    docker kill -s USR1 chromatindb-node4
    docker compose -f deploy/docker-compose.yml logs node4 | tail -10
    ```
    Expected: node4 shows same blob count as other nodes.

11. Clean up:
    ```
    docker compose -f deploy/docker-compose.yml --profile latejoin down -v
    ```
  </how-to-verify>
  <resume-signal>Type "approved" if all nodes connected, blobs replicated through chain, and late-joiner caught up. Otherwise describe issues.</resume-signal>
