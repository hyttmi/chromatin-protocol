  <name>Task 1: Create run-benchmark.sh with utility functions and ingest + sync scenarios</name>
  <files>deploy/run-benchmark.sh</files>
  <action>
Create deploy/run-benchmark.sh as a bash script with the following structure. Use `set -euo pipefail`. The script uses `-p chromatindb` for predictable Docker Compose project naming, so the network is `chromatindb_chromatindb-net`.

CONFIGURATION SECTION at top:
- COMPOSE_FILE="deploy/docker-compose.yml" (relative to repo root)
- COMPOSE="docker compose -f $COMPOSE_FILE -p chromatindb"
- NETWORK="chromatindb_chromatindb-net"
- RESULTS_DIR="deploy/results"
- BLOB_COUNT=200 (enough to measure, fast enough to iterate)
- RATE=50 (blobs/sec)

UTILITY FUNCTIONS:

1. `log()` - Timestamped log to stderr: `echo "[$(date +%H:%M:%S)] $*" >&2`

2. `check_deps()` - Verify docker, docker compose, jq are available. Exit 1 if missing.

3. `build_image()` - Run `docker build -t chromatindb:latest .` from repo root. Skip if `--skip-build` flag was passed.

4. `reset_topology()` - `$COMPOSE down -v 2>/dev/null; $COMPOSE up -d --wait`. This removes volumes for clean state and waits for healthchecks. Add a fallback: if `--wait` fails (older docker compose), poll `docker inspect --format '{{.State.Health.Status}}'` for each node in a loop with 30s timeout.

5. `capture_stats()` - Takes phase ("pre"|"post"|"during") and label args. Runs `docker stats --no-stream --format '{"container":"{{.Name}}","cpu":"{{.CPUPerc}}","mem_usage":"{{.MemUsage}}","mem_perc":"{{.MemPerc}}","net_io":"{{.NetIO}}","block_io":"{{.BlockIO}}","pids":"{{.PIDs}}"}'` for chromatindb-node1, node2, node3, pipes through `jq -s '.'` to get a JSON array, writes to `$RESULTS_DIR/docker-stats-${label}-${phase}.json`.

6. `get_blob_count()` - Takes container name. Sends SIGUSR1 via `docker kill -s USR1 "$container"`, sleeps 1s (spdlog buffer flush + docker log latency), then parses `docker logs "$container" 2>&1 | grep "metrics:" | tail -1 | grep -oP 'blobs=\K[0-9]+'`. Returns the count (echo). Returns 0 if no match.

7. `wait_for_convergence()` - Takes container, expected_count, timeout_seconds. Polls get_blob_count in a loop with 2s sleep between polls. Measures wall-clock time from function entry using `date +%s%N`. Returns elapsed milliseconds on success (echo), returns 1 on timeout. Log progress every poll: "Waiting for $container: $count/$expected blobs..."

8. `run_loadgen()` - Takes target_host, extra args. Runs `docker run --rm --network "$NETWORK" chromatindb:latest chromatindb_loadgen --target "${target_host}:4200" "$@" 2>/dev/null`. Captures and echoes stdout (the JSON).

SCENARIO FUNCTIONS:

9. `run_scenario_ingest()` - Implements PERF-01. Loops over 3 sizes: 1024 102400 1048576. For each size:
   a. Compute label ("1k", "100k", "1m")
   b. reset_topology
   c. capture_stats "pre" "ingest-${label}"
   d. Run loadgen: `run_loadgen node1 --count $BLOB_COUNT --rate $RATE --size $size --drain-timeout 10`
   e. capture_stats "post" "ingest-${label}"
   f. Write loadgen JSON output to `$RESULTS_DIR/scenario-ingest-${label}.json`
   g. Log summary: blobs/sec and p50/p95/p99 from the JSON (parse with jq)

10. `run_scenario_sync()` - Implements PERF-02 + PERF-03. Measures sync latency AND multi-hop in one scenario:
    a. reset_topology
    b. capture_stats "pre" "sync"
    c. Run loadgen against node1: `run_loadgen node1 --count $BLOB_COUNT --rate $RATE --size 1024 --drain-timeout 10` and save output
    d. Record loadgen completion time: `load_done_ns=$(date +%s%N)`
    e. Wait for node2 convergence (1-hop): `sync_1hop_ms=$(wait_for_convergence chromatindb-node2 $BLOB_COUNT 120)`
    f. Wait for node3 convergence (2-hop): `sync_2hop_ms=$(wait_for_convergence chromatindb-node3 $BLOB_COUNT 120)`
    g. capture_stats "post" "sync"
    h. Build result JSON with jq: `jq -n --arg blobs "$BLOB_COUNT" --arg sync_1hop "$sync_1hop_ms" --arg sync_2hop "$sync_2hop_ms" --argjson loadgen "$(cat loadgen output)" '{scenario:"sync-latency", blob_count:($blobs|tonumber), sync_1hop_ms:($sync_1hop|tonumber), sync_2hop_ms:($sync_2hop|tonumber), sync_interval_seconds:10, loadgen:$loadgen}'`
    i. Write to `$RESULTS_DIR/scenario-sync-latency.json`
    j. Log summary: 1-hop and 2-hop times

MAIN function:
- Parse `--skip-build` flag from args
- `mkdir -p "$RESULTS_DIR"`
- check_deps
- build_image (unless --skip-build)
- Run `run_scenario_ingest`
- Run `run_scenario_sync`
- (Phase 30-02 will add late-joiner and trusted-vs-pq scenarios here)
- Log completion message with results directory path

Make the script executable: chmod +x.

IMPORTANT implementation notes:
- Use `2>/dev/null` on docker run for loadgen to suppress stderr (spdlog output) -- stdout is the clean JSON
- For wait_for_convergence timing, use `date +%s%N` for nanosecond precision, convert to ms via arithmetic
- The sync scenario starts timing from loadgen completion (not from function start), because we want to measure how long after writes complete until replication finishes
- Note in a comment that sync latency includes the sync_interval_seconds (10s) scheduling delay -- this is expected real-world behavior
- `jq` is used for both parsing loadgen output and constructing result JSON -- do NOT hand-roll JSON in bash
