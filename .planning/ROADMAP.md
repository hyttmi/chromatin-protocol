### v1.6.0 Python SDK (In Progress)

**Milestone Goal:** Pip-installable Python SDK that connects to a chromatindb relay via PQ-authenticated transport and exposes all 38 client message types through a clean async API with tutorial and documentation.

**Phase Numbering:**
- Integer phases (70, 71, ...): Planned milestone work
- Decimal phases (71.1, 71.2): Urgent insertions (marked with INSERTED)

- [x] **Phase 70: Crypto Foundation & Identity** - Verified PQ crypto primitives and client identity management (completed 2026-03-29)
- [x] **Phase 71: Transport & PQ Handshake** - Encrypted connection to relay with mutual PQ authentication (completed 2026-03-29)
- [ ] **Phase 72: Core Data Operations** - Write, read, delete, list, and exists against a live node
- [ ] **Phase 73: Extended Queries & Pub/Sub** - All remaining message types and real-time notifications
- [ ] **Phase 74: Packaging & Documentation** - Pip-installable SDK with tutorial and docs refresh

## Phase Details

### Phase 70: Crypto Foundation & Identity
**Goal**: SDK crypto primitives produce identical output to the C++ node, and client identities are interoperable with the existing key format
**Depends on**: Nothing (first phase)
**Requirements**: XPORT-01, XPORT-06, PKG-01, PKG-03
**Success Criteria** (what must be TRUE):
  1. ML-DSA-87 keypair generated by SDK can be loaded by C++ node (and vice versa) -- key format is interoperable
  2. SHA3-256, HKDF-SHA256, and ChaCha20-Poly1305 produce byte-identical output to C++ for known test vectors
  3. SDK derives namespace from pubkey identically to C++ (SHA3-256(pubkey) == namespace)
  4. Project is pip-installable in editable mode with pyproject.toml, correct dependencies, exception hierarchy, and FlatBuffers generated code
**Plans:** 3/3 plans complete
Plans:
- [x] 70-01-PLAN.md — SDK package skeleton, pyproject.toml, exception hierarchy, FlatBuffers codegen
- [x] 70-02-PLAN.md — C++ test vector generator binary, JSON crypto vectors
- [x] 70-03-PLAN.md — Crypto primitives, identity management, wire format, tested against C++ vectors

### Phase 71: Transport & PQ Handshake
**Goal**: SDK establishes a PQ-authenticated, AEAD-encrypted session with a live relay
**Depends on**: Phase 70
**Requirements**: XPORT-02, XPORT-03, XPORT-04, XPORT-05, XPORT-07
**Success Criteria** (what must be TRUE):
  1. SDK completes ML-KEM-1024 key exchange and ML-DSA-87 mutual authentication with a running relay
  2. SDK sends and receives AEAD-encrypted frames with correct per-direction nonce counters (no desync)
  3. SDK connects and disconnects cleanly (context manager, Goodbye message)
  4. Handshake follows C++ implementation (empty HKDF salt, correct endianness per field, nonce counters start at 1 post-handshake)
**Plans:** 3/3 plans complete
Plans:
- [x] 71-01-PLAN.md — Encrypted frame IO layer (nonce, raw/AEAD send/recv, exceptions)
- [x] 71-02-PLAN.md — PQ handshake, background reader transport, ChromatinClient context manager
- [ ] 71-03-PLAN.md — Integration tests against live KVM relay + human verification

### Phase 72: Core Data Operations
**Goal**: SDK performs complete blob lifecycle (write, read, delete, list, exists) against a live node through the relay
**Depends on**: Phase 71
**Requirements**: DATA-01, DATA-02, DATA-03, DATA-04, DATA-05, DATA-06
**Success Criteria** (what must be TRUE):
  1. SDK writes a signed blob and receives WriteAck with correct hash and sequence number
  2. SDK reads back a written blob by namespace + hash, with payload matching what was written
  3. SDK deletes a blob via tombstone and confirms deletion (subsequent read returns not-found)
  4. SDK lists blobs in a namespace with pagination and receives correct sequence-ordered results
  5. SDK checks blob existence without data transfer, returning correct boolean result
**Plans:** 2/3 plans executed
Plans:
- [x] 72-01-PLAN.md — Types module (result dataclasses) and codec module (binary payload encode/decode)
- [x] 72-02-PLAN.md — ChromatinClient data operation methods and unit tests
- [ ] 72-03-PLAN.md — Integration tests against live KVM relay + human verification

### Phase 73: Extended Queries & Pub/Sub
**Goal**: SDK exposes all 38 client message types including batch operations, node introspection, and real-time namespace notifications
**Depends on**: Phase 72
**Requirements**: QUERY-01, QUERY-02, QUERY-03, QUERY-04, QUERY-05, QUERY-06, QUERY-07, QUERY-08, QUERY-09, QUERY-10, PUBSUB-01, PUBSUB-02, PUBSUB-03
**Success Criteria** (what must be TRUE):
  1. SDK queries blob metadata, batch-checks existence, and batch-reads multiple blobs in single round-trips
  2. SDK queries node info, storage status, namespace list, namespace stats, peer info, delegation list, and time range -- all returning typed results
  3. SDK subscribes to a namespace, receives real-time notification when a blob is written, and unsubscribes cleanly
  4. Notification dispatch routes server-pushed messages (request_id=0) to an async iterator, separate from request-response correlation
**Plans**: TBD
