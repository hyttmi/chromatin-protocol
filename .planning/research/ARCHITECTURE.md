# Architecture Patterns: v0.9.0 Connection Resilience & Hardening

**Domain:** Connection lifecycle management, logging infrastructure, config validation, storage integrity for chromatindb
**Researched:** 2026-03-19
**Confidence:** HIGH (source code analysis of all affected components)

## Existing Architecture Summary

Three-tier component model, single-threaded event loop with async coroutines:

```
main.cpp
  |-- Config (load JSON, parse CLI)
  |-- Logging (spdlog, console-only)
  |-- Storage (libmdbx, 7 sub-databases, DARE encryption)
  |-- BlobEngine (validation pipeline, crypto offload)
  |-- PeerManager (orchestrator)
        |-- Server (TCP accept/connect, reconnect_loop)
        |-- SyncProtocol (reconciliation, blob transfer)
        |-- AccessControl (ACL gating)
        |-- 5 timer loops: expiry, sync, PEX, flush, metrics
        |-- 2 signal handlers: SIGHUP (config reload), SIGUSR1 (metrics dump)
```

**Critical constraint:** Single io_context thread. All PeerManager state is thread-unsafe by design. The thread_pool is only for crypto offload (returns to event loop via co_await).

## Feature Integration Analysis

### 1. Connection Retry with ACL-Aware Suppression

**What exists:**
- `Server::reconnect_loop()` (server.cpp:204-263) implements exponential backoff (1s to 60s) for bootstrap peers. After `conn->run()` returns, it loops to reconnect.
- `Server::connect_once()` is fire-and-forget for PEX-discovered peers. No reconnect.
- The critical bug: `handshake_ok` is set in the `on_ready` callback (server.cpp:186,248). ACL rejection happens in `PeerManager::on_peer_connected` AFTER `on_ready` fires (because on_ready fires after handshake, before message loop -- ACL check is the first thing in on_peer_connected). So `handshake_ok` is true even for ACL-rejected peers. The reconnect loop sees `ok || handshake_ok` (server.cpp:254) and resets delay to 1s, creating a tight reconnect cycle against ACL-blocked peers.

**Integration approach:**
- Add `bool acl_rejected_` flag to Connection. Set by PeerManager's `on_peer_connected` before `close()` call. Checked by Server's reconnect_loop after `conn->run()` returns.
- If `acl_rejected_`: enter extended backoff (e.g., 600s) or stop until SIGHUP reload.
- This avoids changing the callback API. Connection already has flags like `received_goodbye_`.

**What is new:**
- `Connection::acl_rejected_` flag + `set_acl_rejected()` setter.
- Server::reconnect_loop checks the flag after connection ends.

### 2. Keepalive Timeout for Dead Peer Detection

**What exists:**
- `Connection::message_loop()` handles Ping/Pong (connection.cpp:682-690). Pong is sent on received Ping.
- No code ever sends Ping. No timeout on Pong. Ping/Pong are effectively dead code paths.
- Dead peers detected only on TCP read failure (can take minutes with silent death).
- Connection.h already includes `<asio/experimental/awaitable_operators.hpp>` -- the `operator||` for racing coroutines is available.

**Integration approach (recommended: in Connection):**
- Add `heartbeat_loop()` coroutine to Connection. Self-contained, connection-scoped.
- `Connection::run()` becomes: `co_await (message_loop() || heartbeat_loop())` using the already-included awaitable_operators.
- heartbeat_loop: every `keepalive_interval` seconds, check `last_activity_`. If gap exceeds `keepalive_timeout`, call `close()` (cancels message_loop's pending read). Otherwise, send Ping.
- `last_activity_` updated in `recv_encrypted()` on every successful receive (Pong or any message).
- Config: `keepalive_interval_seconds` (default 30), `keepalive_timeout_seconds` (default 90). Passed to Connection via setters (same pattern as `set_pool()`, `set_trust_check()`).

**Why in Connection (not PeerManager):**
- Ping/Pong already handled in Connection.
- Connection-scoped lifetime (no leak on disconnect).
- `operator||` header already included in connection.h.
- Avoids N per-peer timers in PeerManager (PeerManager already manages 5 timer loops).

**Alternative considered (sweep timer in PeerManager):** Single timer sweeps all peers every 30s, checks `last_activity`. Simpler but less precise (up to 30s detection lag) and requires adding `last_activity` to PeerInfo instead of Connection.

### 3. Cursor Compaction (Age-Based Pruning)

**What exists:**
- Storage has: `list_cursor_peers()`, `cleanup_stale_cursors(known_peer_hashes)`, `delete_peer_cursors(peer_hash)`.
- `SyncCursor.last_sync_timestamp` provides age information.
- Startup cleanup: `PeerManager::start()` calls `cleanup_stale_cursors()` to remove cursors for unknown peers.

**Integration approach:**
- New `Storage::compact_stale_cursors(uint64_t max_age_seconds)`: iterate cursor sub-database, delete entries where `now - last_sync_timestamp > max_age`.
- Call from PeerManager at startup (alongside existing `cleanup_stale_cursors`).
- Optionally call periodically (piggyback on expiry_scan_loop, which runs every 60s).
- Config: `cursor_max_age_seconds` (default 604800 = 7 days).

### 4. Tombstone GC Investigation

**Root cause analysis from code reading:**

1. **TTL=0 tombstones are permanent by design.** `run_expiry_scan()` only visits `expiry_map` entries. TTL=0 blobs have no expiry entry. If benchmarks create TTL=0 tombstones, they accumulate forever. This is correct behavior per MEMORY.md: "TTL=0 = permanent."

2. **seq_map sentinels accumulate.** `delete_blob_data()` replaces seq_map values with zero-hash sentinels (storage.cpp:768-790) to preserve seq_num monotonicity for cursor change detection. These 72-byte entries (40B key + 32B value) never get cleaned up. For heavy delete workloads, this grows the seq_map indefinitely.

3. **used_bytes() reports file size, not actual usage.** `Storage::used_bytes()` returns `env.get_info().mi_geo.current` (mapped file size). libmdbx reuses freed pages internally but may not shrink the file (shrink_threshold = 4 MiB). After bulk deletes, the reported size stays high even though internal free pages exist.

**Conclusion:** This is likely a measurement/documentation issue, not a code bug. Freed storage IS reused by subsequent writes, but the file size metric does not reflect it. Fix: add a `used_pages()` or `free_pages()` metric using mdbx env stats, and document the behavior.

### 5. Startup Integrity Scan

**What exists:**
- `validate_no_unencrypted_data()` runs on Storage construction (checks all blobs are DARE-encrypted).
- `rebuild_quota_aggregates()` exists in API but is NOT called at startup. Quotas are maintained incrementally.

**Integration approach:**
- New `Storage::run_integrity_scan()` method: cross-references sub-databases.
  - Verify seq_map entries point to existing blobs or are zero-hash sentinels.
  - Verify tombstone_map entries correspond to actual tombstone blobs.
  - Verify delegation_map entries correspond to actual delegation blobs.
  - Verify expiry_map entries point to existing blobs.
  - Log warnings for orphaned entries; optionally repair (delete orphans).
- Call `rebuild_quota_aggregates()` at startup for consistency insurance.
- Both called from main.cpp after Storage construction, before PeerManager start.

### 6. libmdbx Crash Recovery Audit

**What exists:**
- `durability::robust_synchronous` mode (storage.cpp:179) -- highest durability.
- `write_mapped_io` mode for writes.
- libmdbx uses copy-on-write / shadow paging (no WAL). After unclean shutdown, last committed transaction is intact; in-progress transaction is rolled back automatically.

**What needs verification (tests only):**
- kill -9 during `store_blob` (mid-transaction) -> restart -> verify consistency.
- kill -9 during `run_expiry_scan` (mid-batch-delete in single transaction) -> restart -> verify no partial deletes.
- These should pass due to MDBX ACID guarantees but need empirical confirmation.

### 7. Delegation Quota Verification

**Code analysis confirms correct behavior:**
- `BlobEngine::ingest()` Step 2: checks `is_owner` or `has_valid_delegation`. Step 2a: checks quota using `blob.namespace_id` (the OWNER's namespace). Delegate writes correctly count against the namespace quota.
- Owner can always delete (tombstones bypass quota check at Step 0b and Step 2a).
- Edge case to document: a delegate can fill an owner's quota, preventing owner writes (but owner can still tombstone).
- Needed: tests confirming this behavior.

### 8. Version Injection via CMake

**What exists:** Stale `db/version.h` with hardcoded `"0.6.0"`.

**Integration approach:**
- Add `project(chromatindb-core VERSION 0.9.0 LANGUAGES C CXX)` to CMakeLists.txt.
- Create `db/version.h.in`: `static constexpr const char* VERSION = "@PROJECT_VERSION@";`
- Add `configure_file(version.h.in ${CMAKE_CURRENT_BINARY_DIR}/db/version.h @ONLY)`.
- Add `$<BUILD_INTERFACE:${CMAKE_CURRENT_BINARY_DIR}>` to target_include_directories.
- Delete `db/version.h`, add to .gitignore.
- main.cpp unchanged (`#include "db/version.h"` resolves to generated file).

### 9. Config Validation Fail-Fast

**What exists:** `load_config()` validates JSON parse, hex keys, IP addresses. No numeric range validation.

**Integration approach:**
- New `validate_config(const Config&)` function in config.h/config.cpp.
- Validates: `max_peers > 0`, `sync_interval_seconds > 0`, `rate_limit_burst >= rate_limit_bytes_per_sec` (when both > 0), `max_sync_sessions > 0`, `full_resync_interval > 0`, `cursor_stale_seconds > 0`, `bind_address` parseable.
- Called in main.cpp after `parse_args()`, before any component construction.
- Throws `std::runtime_error` with field name, invalid value, and expected range.

### 10. Complete Metrics

**What is missing from periodic `log_metrics_line()`:**
- `quota_rejections` and `sync_rejections` (only in SIGUSR1 dump, not periodic line).
- No byte counters (bytes_received, bytes_sent).
- No tombstone counters.
- No cursor_compactions counter (new for v0.9.0).

**Integration:** Add missing fields to `NodeMetrics` and include all in `log_metrics_line()`.

### 11. Structured Logging + File Sink

**What exists:** spdlog with `[%Y-%m-%d %H:%M:%S.%e] [%l] [%n] %v` pattern, stderr only.

**Integration approach:**

**Structured format:** JSON-lines via custom formatter. The `%v` (message) field may contain quotes that break JSON. Recommend subclassing `spdlog::custom_flag_formatter` for JSON-safe message escaping. One-time cost, prevents broken JSON in monitoring pipelines.

**File logging:** `spdlog::sinks::rotating_file_sink_mt` with configurable max size and max files. Multi-sink logger: both console and file get the same formatter.

**Config fields:** `log_format` ("text"|"json", default "text"), `log_file` (path, empty = disabled), `log_file_max_bytes` (default 50 MiB), `log_file_max_files` (default 5).

### 12. Timer Cleanup

**The problem:** `PeerManager::stop()` and the `server_.set_on_shutdown()` lambda both cancel the same 5 timers + 2 signals. Adding a timer requires updating both. Can't make `on_shutdown` call `stop()` (recursive: `stop()` calls `server_.stop()` which calls `on_shutdown`).

**Integration approach:** Extract `cancel_all_timers()` private method. Both `stop()` and the `on_shutdown` lambda call it. Idempotent. New timers added in one place.

## Data Flow Changes

### Current Connection Lifecycle
```
bootstrap_peers -> Server::connect_to_peer() -> handshake
  -> on_ready (PeerManager::on_peer_connected: ACL check)
  -> message_loop() until disconnect or error
  -> on_close -> PeerManager::on_peer_disconnected
  -> if bootstrap: enter reconnect_loop (backoff 1s-60s, forever)
  -> if discovered: gone forever
```

### v0.9.0 Connection Lifecycle
```
bootstrap_peers -> Server::connect_to_peer() -> handshake
  -> on_ready (PeerManager::on_peer_connected: ACL check)
  |    -> if ACL rejected: conn->set_acl_rejected(), conn->close()
  -> message_loop() || heartbeat_loop()    <-- concurrent coroutines
  |    heartbeat: check last_activity_, send Ping, close on timeout
  -> on_close -> PeerManager::on_peer_disconnected
  -> reconnect_loop:
  |    -> if acl_rejected_: delay = 600s (extended suppression)
  |    -> else: delay = 1s (reset on success)
  -> if discovered: gone forever (unchanged)
```

### Startup Flow (Updated)
```
main.cpp::cmd_run()
  |-- parse_args + load_config
  |-- validate_config(config)              <-- NEW: fail-fast
  |-- logging::init(config)                <-- MODIFIED: file sink + format
  |-- Storage(data_dir)
  |       |-- validate_no_unencrypted_data() (existing)
  |-- storage.run_integrity_scan()         <-- NEW
  |-- storage.compact_stale_cursors(...)   <-- NEW
  |-- storage.rebuild_quota_aggregates()   <-- NEW (existing method, newly called)
  |-- BlobEngine(storage, pool, ...)
  |-- PeerManager(...)
  |-- pm.start()
  |-- ioc.run()
```

## Patterns to Follow

### Pattern 1: Timer-Cancel for Periodic Tasks
All periodic tasks use stack-allocated steady_timer with pointer stored in member for cancellation. See existing `expiry_scan_loop()` for the canonical example. New timers must be added to `cancel_all_timers()`.

### Pattern 2: Awaitable operator|| for Concurrent Connection Coroutines
Race `message_loop()` against `heartbeat_loop()` using `asio::experimental::awaitable_operators::operator||`. Connection.h already includes the header. First to complete cancels the other (close() cancels pending async reads).

```cpp
// Connection already includes this:
#include <asio/experimental/awaitable_operators.hpp>

// In Connection::run(), after handshake:
using namespace asio::experimental::awaitable_operators;
co_await (message_loop() || heartbeat_loop());
```

### Pattern 3: Snapshot Before Iteration
Copy containers (connections_, peers_) before iterating when callbacks may modify them. Critical for disconnect handling during iteration.

### Pattern 4: Step 0 (Cheapest Check First)
Integer compare or flag check before expensive operations (crypto, I/O). All validation pipelines follow this pattern.

### Pattern 5: Config Validation as Separate Pass
`validate_config()` runs after all config sources are merged (file + CLI), before component construction. Throws with human-readable field/value/expected message. Important: NOT inside `load_config()` because CLI overrides happen after.

### Pattern 6: CMake Version Generation
Template `version.h.in` -> `configure_file()` -> generated `version.h` in build dir. Source file in .gitignore. Only changes when version string changes.

## Anti-Patterns to Avoid

### Anti-Pattern 1: Lambda Coroutines Capturing this
Coroutine frame outlives lambda stack. Use member functions or capture by value/shared_ptr. Real bug history: SIGHUP handler.

### Anti-Pattern 2: Dual Timer Cancellation Sites
Cancelling timers in both `stop()` and `on_shutdown` independently. Use single `cancel_all_timers()` method called from both.

### Anti-Pattern 3: Blocking I/O on Running Event Loop
Synchronous file I/O blocks all network operations. Acceptable at startup only (before `ioc.run()`). Not during runtime.

### Anti-Pattern 4: Reconnect Without Backoff on ACL Rejection
Resetting delay to 1s after ACL-rejected connection creates tight connect-handshake-reject cycle. Detect via `acl_rejected_` flag and enter extended backoff.

### Anti-Pattern 5: Version File in Source Control
Generated `version.h` committed to git causes merge conflicts. Keep in .gitignore.

### Anti-Pattern 6: Config Validation Inside load_config()
CLI overrides happen after `load_config()`. Validation inside load_config misses them. Validate after all sources merged.

## Suggested Build Order (Dependency-Based)

### Phase 1: Foundation (no dependencies, zero risk)
1. **Version injection (CMake)** -- Correct version in all subsequent work.
2. **Config validation fail-fast** -- Catches bad config before component construction.
3. **Timer cleanup refactoring** -- `cancel_all_timers()` prerequisite for new timers.

### Phase 2: Storage & Logging (independent of network changes)
4. **Structured logging + file sink** -- Better debugging for subsequent phases.
5. **Cursor compaction** -- Storage-only. New method + startup/periodic call.
6. **Tombstone GC investigation** -- May be documentation-only fix.
7. **Startup integrity scan** -- Cross-reference sub-databases on startup.
8. **Complete metrics** -- Add missing counters to log line.

### Phase 3: Network Resilience (highest risk, benefits from phases 1-2)
9. **Keepalive timeout** -- Connection::heartbeat_loop() with operator||.
10. **ACL-aware reconnection** -- acl_rejected_ flag + extended backoff.

### Phase 4: Verification & Documentation
11. **libmdbx crash recovery audit** -- Tests only.
12. **Delegation quota verification** -- Tests only.
13. **Documentation updates** -- README, PROTOCOL, config docs.

**Ordering rationale:**
- Phase 1 is pure infrastructure safety. No existing behavior changes.
- Phase 2 improves storage/logging tooling used to debug Phase 3.
- Phase 3 is highest-risk (network state machines). All tooling should be in place.
- Phase 4 validates everything and documents it.

## Scalability Considerations

| Concern | At 10 peers | At 100 peers | At 1000 peers |
|---------|-------------|--------------|---------------|
| Keepalive (in Connection) | 10 concurrent heartbeat coroutines, trivial | 100 coroutines, fine | 1000 -- acceptable (max_peers default 32) |
| Reconnect coroutines | Bootstrap count only, negligible | N/A | N/A |
| Cursor compaction | Microseconds | Milliseconds | Seconds -- batch or limit per scan |
| Log file I/O | Minimal | Moderate | May need level filtering per component |
| Integrity scan | Startup-only, <1s | 1-5s | 30s+ for large DBs, log progress |

**Current max_peers default: 32.** All approaches are well within budget.
